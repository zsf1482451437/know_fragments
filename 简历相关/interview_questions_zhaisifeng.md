## 二、Reolink Web 跨境商城

### 4. Reolink Web 跨境商城你主要做了什么？

我主要参与官网商城相关页面和业务模块开发，包括商品详情页、活动落地页、解决方案页等 10+ 核心页面。除了页面开发，也做了一些组件抽象，比如商品卡片、卖点模块、媒体展示、规格参数、FAQ 等 8+ 可复用组件。

另外我也参与了多端适配、SEO 和性能优化。比如针对桌面端、平板端、移动端 3 类视口做响应式布局，维护 20+ 商品/活动页面的基础 SEO 配置，并通过图片懒加载、资源拆分、语义化标签、Meta 信息维护等方式，把重点活动页 LCP 控制在 2.5s 内。

### 5. 你们为什么要做组件化？怎么判断一个模块该不该抽组件？

商城页面有很多重复结构，比如商品卡片、卖点展示、参数表、FAQ、媒体图文模块。如果每个页面都单独写，会导致样式不一致、维护成本高、活动页交付慢。

我一般从三个标准判断是否抽组件：第一，是否在多个页面重复出现；第二，是否有相对稳定的输入输出，比如商品信息、图片、文案、跳转链接；第三，后续运营或活动是否会持续复用。满足这些条件就适合抽成组件。抽的时候我会控制组件边界，不会把业务差异很大的逻辑硬塞进一个组件，而是通过 props 或组合方式保留扩展性。

### 6. 商品页和活动页的响应式适配你怎么做？

首先会根据设计稿和业务流量场景确认断点，比如桌面端、平板端、移动端。桌面端更关注内容完整展示和转化模块的位置，移动端更关注首屏信息密度、按钮可点击区域和图片加载体验。

实现上会尽量使用弹性布局、栅格、媒体查询和 Tailwind 的响应式能力。对于图片和媒体模块，会根据不同视口控制尺寸、裁剪和加载优先级。复杂模块不会只靠缩放，而是会调整 DOM 排列，比如桌面端左右结构，移动端改成上下堆叠，保证阅读和点击体验。

### 7. 你提到 LCP 控制在 2.5s 内，具体做了哪些优化？

主要从资源体积、关键渲染路径和首屏媒体三个方向做。资源体积上，减少首屏无关模块的同步加载，能拆分的组件延后加载；图片上，对首屏核心图做尺寸压缩和格式优化，非首屏图片使用懒加载；字体和样式上，避免阻塞首屏渲染。

另外，活动页通常图片较多，我会区分首屏关键资源和下方内容资源，优先保证首屏主视觉、标题、CTA 按钮快速展示。SEO 相关的语义化结构也会一起处理，避免为了动画或布局牺牲可访问性和抓取效果。

### 8. SEO 优化你具体做过哪些？

我做的主要是基础 SEO 和页面结构优化。包括维护 title、description、canonical、Open Graph 等 Meta 信息；用合理的 H1/H2 层级组织页面内容；商品页和活动页保持语义化标签；图片补充 alt 信息；避免核心内容完全依赖客户端异步渲染。

对于跨境商城，SEO 不只是写 Meta，还要考虑多语言、多地区和内容运营。我的经验是先保证每个商品/活动页面有清晰的内容结构和唯一主题，再配合运营维护关键词和页面文案。

## 三、Reolink React Native 移动端项目

### 9. Reolink 移动端项目里的核心链路是什么？

我参与的核心链路主要有 4 条：设备接入、实时预览、消息提醒、录像回放。设备接入偏流程和状态管理，实时预览偏性能和异常处理，消息提醒偏状态同步和用户触达，录像回放偏列表、时间轴和交互体验。

这些链路的特点是状态多、异常多、设备和网络环境差异大，所以不能只写页面，还要关注状态拆分、错误处理、加载反馈和回归测试。

### 10. React Native 项目里你怎么做状态管理？

我主要基于 React Hooks 做状态拆分。比如设备状态、消息状态、页面展示状态会分开管理，不会全部塞在一个组件里。对于只在页面内部使用的状态，会放在组件或自定义 Hook 里；多个页面共享的状态，会抽到更上层或统一模块里。

这样做的好处是定位问题更快，也减少无效渲染。比如设备状态更新不应该导致整个页面所有展示模块都重新计算，能局部更新就局部更新。

### 11. 移动端多机型适配你遇到过什么问题？

主要问题有三类：屏幕尺寸差异、系统行为差异、性能差异。比如小屏设备上按钮和内容容易挤压，刘海屏和安全区域需要额外处理；不同系统版本对滚动、键盘、权限弹窗表现也可能不一致；低端机上长列表和图片加载更容易卡顿。

我的处理方式是先保证布局弹性和安全区域适配，再针对高频页面做真机验证。对于列表和图片较多的场景，会控制渲染数量、拆分组件、减少不必要的状态更新。

### 12. 你提到长列表卡顿，React Native 里一般怎么优化？

首先会看是不是用了合适的列表组件，比如 FlatList，并配置 initialNumToRender、windowSize、removeClippedSubviews 等参数。其次会避免 item 组件里写过重逻辑，必要时使用 memo、useCallback、useMemo 控制重复渲染。

如果列表里有图片或复杂组件，会做懒加载和占位，避免首屏一次性渲染过多内容。最后还要检查状态更新范围，不要一个状态变化导致整个列表重新渲染。

### 13. Native Modules 你怎么理解？什么时候需要用？

Native Modules 是 React Native 和原生能力交互的桥。一般纯 UI 和普通业务逻辑用 JS 就够了，但如果涉及系统能力、设备能力、性能敏感能力，或者已有原生 SDK，就需要通过 Native Modules 接入。

我的理解是，跨端开发不能只看 JS 层。遇到相机、推送、权限、设备连接等能力时，要知道 JS 和原生边界在哪里，哪些逻辑适合放 JS，哪些应该交给原生模块处理。

### 14. 你为什么要补单元测试和集成测试？

移动端核心链路回归成本比较高，尤其是设备接入、消息提醒、录像回放这些场景，很多问题不是页面一眼能看出来的。如果只靠手工回归，容易漏掉状态流转和异常分支。

所以我围绕 3 类核心链路补了 20+ 用例，包括状态流转、页面交互、异常分支。单元测试主要验证函数、Hook、状态逻辑；React Native Testing Library 验证组件交互；Detox 更偏端到端流程验证。目标不是追求覆盖率数字，而是把高风险路径纳入稳定验证。

### 15. Jest、React Native Testing Library、Detox 的分工是什么？

Jest 更适合测纯逻辑、工具函数、状态计算和简单 Hook。React Native Testing Library 更适合从用户视角测试组件，比如某个按钮点击后是否出现对应状态。Detox 更接近真实 App 流程，可以验证跨页面、真实交互、端到端链路。

我的使用原则是：能用低成本测试覆盖的，不上重测试；核心链路和回归风险高的，再用集成或 E2E 测试兜底。

## 四、GogoalShop 与 Next.js

### 16. GogoalShop 首页性能优化你是怎么做的？

当时首页的问题主要是关键资源大、首屏加载链路长。我的优化方向有三个：第一，做代码分割和依赖治理，把不影响首屏的模块延后加载；第二，替换或裁剪过重依赖，配合 Tree Shaking 减少打包体积；第三，用 `next/image`、`next/font` 优化图片和字体加载。

结果上，关键 JavaScript 从 3MB 压到 600KB，首页 LCP 从 4s 降到 2s 以内，用户跳出率从 30% 降到 20%。这些指标能说明优化不是只看打包体积，也看用户实际体验。

### 17. 你怎么分析一个页面为什么慢？

我一般先用 Lighthouse、Performance、Network 面板看整体情况。先判断是资源下载慢、JS 执行慢、渲染慢，还是接口慢。然后看 LCP 元素是什么、关键资源有哪些、是否有阻塞渲染的 JS/CSS。

定位后再分层处理：资源大就压缩和拆分；JS 执行慢就减少首屏逻辑和依赖；图片慢就优化格式、尺寸、懒加载；接口慢就考虑缓存、骨架屏或并行请求。不会一上来就盲目改代码。

### 18. Next.js 项目里 SSR、SSG、CSR 怎么选择？

我会先看页面的内容变化频率、SEO 诉求和首屏性能要求。像商品详情页、解决方案页、活动页这类需要被搜索引擎收录、首屏内容相对明确的页面，更适合 SSR 或 SSG。内容更新不频繁的页面可以用 SSG，既能保证首屏性能，也能降低服务端压力。

CSR 更适合强交互、用户态明显、SEO 诉求不强的模块，比如后台配置页、用户中心局部模块等。我的理解是不要为了技术统一强行选择一种模式，而是按页面的业务目标取舍：SEO 和首屏优先就偏 SSR/SSG，交互和用户态优先就偏 CSR。

### 19. pnpm 相比 npm 有什么优势？

pnpm 的优势主要是依赖安装速度快、磁盘占用低、依赖结构更严格。它通过内容寻址存储复用依赖，不会每个项目都完整复制一份 node_modules。

在项目里迁移到 pnpm 后，依赖安装时间从 10s 降到 5s 以内。另一个好处是能暴露一些幽灵依赖问题，让项目依赖声明更规范。

## 五、Shopify 自研应用

### 20. 你为什么要自研 Shopify 应用？

主要是降本和满足定制化需求。第三方应用能解决通用问题，但有些业务规则不够灵活，而且长期付费成本不低。我们自研了「自定义印号 TT-Options」和「黑名单下单拦截 TTBlacklist - Order Guard」两款应用，用来满足商品定制和订单拦截场景。

自研的价值不只是省钱，还包括掌握业务规则、部署链路和后续扩展能力。上线后我也沉淀了开发、部署、上架文档，方便团队后续复用。

### 21. Shopify 应用里 GraphQL 和 Prisma 分别做什么？

GraphQL 主要用来和 Shopify Admin API 交互，比如读取商品、订单、客户或配置相关数据。Prisma 主要负责应用自身的数据模型和持久化，比如黑名单规则、配置项、用户选择记录等。

简单说，Shopify GraphQL 是对外和 Shopify 平台交互，Prisma 是应用自己的数据层。两者边界清楚，后续排查问题也更方便。

### 22. 为什么要用 Docker 部署 Shopify 应用？

主要是解决环境不一致问题。脚手架依赖、Node 版本、系统库在服务器上可能和本地不一致，直接部署容易出现“本地能跑、线上不行”。

Docker 的好处是把运行环境镜像化，Node 版本、依赖安装、启动命令都固化下来，减少宿主机影响。配合 Shell 脚本做镜像打包、版本命名和文件压缩，可以让部署过程更稳定。

## 六、工程化与协作

### 23. ESLint、Prettier、Commitlint、Husky 各自解决什么问题？

ESLint 解决代码质量和潜在问题，比如未使用变量、不规范写法。Prettier 解决格式统一问题。Commitlint 规范提交信息，方便后续追踪变更。Husky 用来在提交前挂钩这些检查，避免不符合规范的代码进入仓库。

我的理解是，这些工具不是为了“流程复杂”，而是把代码规范前置，让问题在提交阶段就暴露，减少 code review 的低价值沟通。

### 24. 你怎么做代码评审？

我会先看需求目标和实现是否一致，再看代码边界是否清楚。具体会关注几个点：有没有影响现有功能、状态管理是否过重、接口异常是否处理、组件是否过度抽象、是否有必要测试。

对于 3-5 年开发来说，我觉得 code review 不只是挑语法问题，而是帮助团队减少线上风险，同时保持代码可维护。

### 25. 你如何拆解一个需求？

我一般先确认业务目标和验收标准，再拆页面、接口、状态、异常、埋点或 SEO 等模块。然后判断哪些是核心链路，哪些是边缘场景，优先保证核心路径可用。

如果需求涉及多个端或多个角色，我会先画出流程，明确输入输出和状态变化。最后再拆开发任务和测试点，避免开发完才发现漏了异常分支。

## 七、AI 工程化与 Harness Engineering

### 26. 你说的 Harness Engineering 是什么？

我这里说的 Harness Engineering，不是单纯写 Prompt，而是围绕 AI 辅助研发建立一套可复用、可校验的工作流。比如把需求拆解、代码草拟、问题排查、测试补充、文档沉淀拆成固定步骤，并为每类任务准备结构化模板。

它的重点是“约束输入、规范输出、保留验证”。AI 生成的内容不能直接相信，需要结合项目上下文、代码规范、测试和人工 review。这样 AI 才是研发流程的一部分，而不是临时问答工具。

### 27. 你们的 AI 工作流怎么落地？

我沉淀了 1 套流程和 10+ 结构化任务模板，覆盖 5 类场景：需求拆解、代码草拟、问题排查、测试补充、文档生成。比如排查问题时，会要求 AI 先复述现象、列出可能原因、标注需要验证的证据，再给修改建议，而不是直接输出代码。

落地时我会要求每个结果都有验证动作，比如跑测试、检查构建、补充文档或人工 review。这样能降低 AI 输出偏离项目上下文的风险。

### 28. 面试官质疑 AI 这块包装太重，你怎么回答？

我会承认它不是独立业务系统，也不是替代开发的工具。我把它写进简历，是因为它确实提升了我在需求拆解、问题分析、测试补充和文档沉淀上的效率。

我的重点不是“会用 AI”，而是能把 AI 输出纳入工程流程：有模板、有上下文约束、有验证、有人工兜底。这和日常工程化的思路是一致的。

## 八、基础技术高频问题

### 29. React Hooks 使用时要注意什么？

主要注意依赖数组、闭包、状态拆分和副作用边界。比如 useEffect 的依赖不完整会导致状态不一致，依赖过多又可能导致重复执行。useCallback、useMemo 也不是越多越好，只有在子组件 memo 或计算成本较高时才有意义。

我更关注 Hooks 能不能让状态和副作用边界清楚，而不是单纯追求写法简洁。

### 30. TypeScript 给项目带来的价值是什么？

TypeScript 最大的价值是提前暴露类型问题，尤其在接口字段、组件 props、业务配置、复杂状态结构里很有用。对多人协作来说，类型也是文档的一部分。

我一般会优先给接口响应、组件 props、业务枚举、工具函数补类型。不会为了类型而类型，复杂类型如果影响可读性，也会做适当拆分。

### 31. 浏览器缓存你怎么理解？

浏览器缓存主要分强缓存和协商缓存。强缓存通过 Cache-Control、Expires 控制，命中时不发请求；协商缓存通过 ETag、Last-Modified 和服务端确认资源是否变化。

在前端工程里，一般静态资源会带 hash，设置较长强缓存；HTML 入口文件缓存时间短一些，保证发布后能拿到新资源引用。

### 32. 跨域问题怎么解决？

跨域是浏览器同源策略导致的。常见解决方式有 CORS、开发环境代理、Nginx 反向代理等。生产环境最常见还是服务端配置 CORS 或通过同域网关转发。

前端要注意的是，跨域不是单纯前端能完全解决的问题，尤其带 cookie 或鉴权头时，需要服务端配置 Access-Control-Allow-Credentials、Allowed-Headers 等。

## 九、行为面与反问

### 33. 你遇到过最大的技术挑战是什么？

可以回答 GogoalShop 性能优化。挑战在于首页资源大、依赖复杂、图片和字体影响首屏。我的做法是先用工具定位瓶颈，再分资源、JS、图片、字体几个方向处理。最终关键 JS 从 3MB 降到 600KB，LCP 从 4s 降到 2s 内。

这个经历让我形成了一个习惯：优化前先量化问题，优化中分层处理，优化后用指标验证。

### 34. 你怎么看业务开发和技术深度的关系？

我觉得前端不能只追技术，也不能只堆业务。业务开发是技术落地的场景，技术深度是解决复杂问题的能力。比如商城页面开发看起来是业务需求，但里面有性能、SEO、组件化、可维护性和工程效率问题。

我更倾向于在业务里沉淀通用能力，比如组件、工具、流程、测试和文档，而不是每次只完成当前需求。

### 35. 你有什么短板？

我目前更偏前端和跨端，对更底层的原生能力或大型前端基建系统还需要继续深入。比如 React Native 的原生模块，我能理解边界和参与联调，但如果涉及复杂原生 SDK 封装，还需要和原生同学配合。

我现在的提升方向是继续加强移动端性能分析、测试体系建设，以及复杂业务下的工程化治理能力。

### 36. 你有什么想问我们的？

我会关注三个方面：

- 团队当前主要业务方向是什么，是偏 Web、跨端，还是平台工具？
- 当前前端团队最希望解决的问题是什么，比如性能、稳定性、研发效率，还是业务交付？
- 对这个岗位 3-6 个月的预期是什么，如何判断一个人做得好？

这些问题能帮助我判断岗位和我的经验是否匹配，也能更快对齐入职后的工作重点。

## 十、重点追问与回答要点

### 37. `Harness Engineering` 具体模板长什么样？如何验证 AI 输出？

我的模板一般不是一句 Prompt，而是一个固定结构：背景、目标、输入上下文、约束条件、输出格式、验证方式。比如问题排查模板里，会要求先复述现象，再列可能原因，再说明需要补充哪些证据，最后给出修改建议和验证命令。

验证上我不会直接采用 AI 输出。代码类结果要结合项目规范人工 review，能跑测试就跑测试，涉及页面就做浏览器或真机验证，涉及文档就检查是否和实际流程一致。我的理解是 Harness Engineering 的价值在于把 AI 输出纳入可控流程，而不是把 AI 当成最终结论。

### 38. `20+ 测试用例` 分别覆盖哪些场景？哪些是单测，哪些是集成测试？

这 20+ 用例主要围绕设备接入、消息提醒、录像回放 3 类核心链路。单测更多覆盖状态流转、工具函数、异常分支，比如设备状态从未连接到连接中再到连接成功或失败。组件测试会覆盖页面交互，比如点击重试、切换筛选条件、消息状态展示。集成或 E2E 更关注跨页面流程，比如进入设备页、触发操作、看到预期反馈。

我不会为了覆盖率盲目补测试，优先覆盖回归成本高、容易出问题、人工验证麻烦的链路。这样测试用例对迭代才有实际价值。

### 39. `重点活动页 LCP 2.5s` 是怎么测的？用 Lighthouse 还是线上监控？

我会先用 Lighthouse 和 Chrome Performance 做本地分析，确认 LCP 元素是什么、关键资源有哪些、是否有阻塞渲染的 JS/CSS。这个阶段适合定位问题和验证优化方向。

如果项目有线上监控，最终还是要看真实用户环境的数据，比如不同地区、设备、网络下的 Web Vitals。简历里写的 2.5s 更适合作为重点页面优化后的阶段性指标，面试时我会说明这是对重点活动页在测试环境或重点场景下的优化结果，不会夸大成所有页面、所有用户都稳定达到。

### 40. Reolink Web 商城的 Next.js 是 SSR、SSG 还是 CSR？为什么这么选？

我的选择会按页面类型来定。商品详情页、解决方案页、活动页这种对 SEO 和首屏有要求的页面，更适合 SSR 或 SSG。内容更新频率低、结构稳定的页面可以用 SSG，性能更好、服务端压力更小；价格、库存或动态规则较多的页面可以考虑 SSR 或客户端局部请求。

CSR 更适合用户态明显、SEO 不重要的模块，比如一些后台配置、登录后个人中心、局部交互模块。我的原则是不要为了统一技术模式牺牲业务目标，商城前台一般会优先考虑 SEO 和首屏体验。

### 41. React Native 项目里 Native Modules 具体接了什么能力？

我会把 Native Modules 理解成 JS 层和原生能力之间的桥。像推送、权限、设备连接、媒体能力、原生 SDK 这类能力，如果 JS 层无法直接完成，或者性能/稳定性更适合放原生侧，就需要通过 Native Modules 暴露接口给 React Native 使用。

面试时如果被问到具体实现，我会讲边界：我主要负责 JS 侧调用、状态处理、异常兜底和联调验证；原生模块的底层封装通常会和原生同学协作。这样回答更真实，也不会把自己包装成原生专家。

### 42. 长列表卡顿如何定位，是 JS 线程问题、渲染问题还是图片问题？

我会先看卡顿发生在什么场景：首屏渲染慢、滚动掉帧、快速滑动白屏，还是图片加载导致抖动。然后用 React DevTools、日志和性能工具判断是不是重复渲染、单个 item 过重、列表窗口参数不合理，或者图片解码和加载压力过大。

处理上会分层做：列表用 FlatList 并调整 initialNumToRender、windowSize 等参数；item 组件用 memo、useCallback 控制重复渲染；图片做懒加载、占位和尺寸控制；状态更新尽量局部化，避免一个状态变化导致整列表刷新。

### 43. Shopify 两个应用分别解决了什么业务问题？审核上架遇到过什么坑？

「自定义印号 TT-Options」主要解决商品定制配置问题，让业务可以配置用户下单时需要填写或选择的定制信息。「黑名单下单拦截 TTBlacklist - Order Guard」主要解决风险订单拦截问题，比如根据邮箱、手机号、客户标识等规则阻止异常下单。

上架和部署的坑主要在权限声明、回调地址、环境变量、Webhook/接口权限、隐私政策和应用配置一致性。我的经验是把本地、测试、生产环境的配置隔离清楚，把应用权限、回调 URL、安装流程和异常兜底提前检查，不然很容易本地可用但线上安装失败。

### 44. Docker 部署具体 Dockerfile 怎么写？如何管理环境变量？

Dockerfile 通常会分几个步骤：选择 Node 基础镜像，复制 package 文件，安装依赖，复制源码，执行构建，最后暴露端口并启动服务。如果项目体积较大，可以用多阶段构建减少最终镜像体积。

环境变量不会写死在镜像里，一般通过 `.env`、服务器环境变量、CI/CD 注入或部署平台配置。像 Shopify API Key、Secret、数据库连接这类敏感信息必须放在运行环境里，镜像只保留读取逻辑。这样镜像可以复用，环境差异通过配置解决。

### 45. 你负责和团队共同完成的边界分别是什么？

我会按模块讲清楚。Reolink 项目里，整体产品和核心架构是团队共同完成的，我主要参与 Web 商城页面模块、移动端核心链路、适配优化、测试补充和部分工程流程沉淀。GogoalShop 和 Shopify 自研应用里，我负责的链路更完整，包括需求拆解、功能开发、部署和文档沉淀。

我觉得面试时讲清边界很重要。自己负责的部分可以深入讲实现细节，团队共同完成的部分可以讲参与角色和协作方式，不把团队成果全部说成个人成果。

### 46. Reolink Web 商城里哪些页面适合 SSR/SSG，哪些模块适合 CSR？为什么？

商品详情页、解决方案页、活动落地页适合 SSR/SSG，因为这些页面需要 SEO、分享预览和较好的首屏体验。内容稳定的页面适合 SSG，内容变化较频繁但仍有 SEO 诉求的页面可以用 SSR。

CSR 更适合购物车状态、用户登录状态、个性化推荐、部分筛选交互等模块。这些内容依赖用户态或实时接口，不一定适合在构建时生成。实际项目里通常是页面主体 SSR/SSG，局部交互 CSR。

### 47. 商品详情页的 SEO 信息如何维护？title、description、canonical、结构化数据分别解决什么问题？

title 主要影响搜索结果标题和页面主题，description 影响搜索结果摘要和点击吸引力，canonical 用来告诉搜索引擎哪个 URL 是主版本，避免重复页面分散权重。结构化数据可以帮助搜索引擎理解商品价格、评分、库存、面包屑等信息，有机会获得更丰富的搜索展示。

维护时我会保证每个商品页有唯一标题和描述，图片有 alt，H1/H2 结构清晰，canonical 指向规范 URL。对于多语言或多地区站点，还要注意 hreflang、地区 URL 和内容一致性。

### 48. LCP 优化时如何判断真正的 LCP 元素？如果 LCP 是主图，你会怎么优化？

可以通过 Lighthouse、Performance 面板或 Web Vitals 工具看 LCP 元素。很多商城活动页的 LCP 都是首屏主图、Banner 或大标题区域。确认元素后，不要泛泛优化，要针对这个元素的加载链路处理。

如果 LCP 是主图，我会做几件事：压缩图片体积，使用合适尺寸和格式，避免移动端加载桌面大图；首屏图不要懒加载；必要时加 preload；减少阻塞渲染的 JS/CSS；同时确保图片容器尺寸稳定，避免布局偏移。

### 49. 你抽象的 8+ 组件里，哪个最有代表性？它的 props 设计和扩展点怎么做？

比较有代表性的是商品卡片或卖点模块。商品卡片通常包含图片、标题、卖点、价格、标签、跳转链接和 CTA。props 会围绕数据和展示配置设计，比如 product、badge、showPrice、ctaText、onClick，而不是把页面逻辑写死在组件里。

扩展点上，我会保留 className、children 或 slots 类似的组合能力，让不同活动页可以在统一结构下做轻微定制。组件内部只处理通用展示和基础交互，业务差异尽量放到外层组装，避免组件越来越臃肿。

### 50. 活动页交付检查清单包含哪些内容？如何避免上线前遗漏 SEO、埋点、适配和性能问题？

检查清单一般包括：页面内容是否和设计稿一致，桌面/平板/移动端是否适配，核心链接和 CTA 是否正确，图片是否压缩，Meta 信息是否完整，H1/H2 是否合理，埋点是否触发，接口异常是否有兜底，Lighthouse 指标是否达标。

避免遗漏的关键是把这些检查标准流程化，而不是靠个人记忆。活动页上线前按清单逐项确认，复杂活动可以让设计、运营、后端一起参与验收，减少只从前端视角检查的问题。

### 51. React Native 里设备状态、消息状态、页面展示状态为什么要拆开？不拆会有什么问题？

这些状态变化频率和影响范围不一样。设备状态可能来自连接链路，消息状态可能来自推送或接口，页面展示状态更多是 UI 层交互。如果混在一起，状态更新会互相影响，容易导致组件重复渲染，也不方便定位问题。

拆开后，每类状态有明确边界，页面只订阅自己需要的状态。这样可维护性更好，也能减少复杂页面里“改一个状态影响一大片”的问题。

### 52. 20+ 测试用例里最有价值的是哪几类？有没有发现过真实回归问题？

最有价值的是核心链路的状态流转和异常分支，比如设备接入失败后的重试、消息为空时的展示、录像列表加载失败后的兜底。这些场景人工回归容易漏，但一旦线上出问题，用户感知比较明显。

如果被问有没有发现真实回归，可以稳妥回答：测试补充后主要帮助我们在迭代时更早发现状态展示和异常分支的问题，尤其是改动公共 Hook 或通用组件时，可以快速确认核心路径没有被破坏。不要夸大成完全杜绝线上问题。

### 53. Detox 用例运行成本比较高，你怎么决定哪些场景值得写 E2E？

我会优先给高价值、跨页面、人工回归成本高的场景写 E2E。比如设备接入到进入预览页、消息列表到详情页、录像回放筛选等完整流程。单个组件交互不一定需要 Detox，用 React Native Testing Library 更合适。

原则是测试金字塔：底层逻辑多写单测，中间组件写集成测试，少量关键路径用 E2E 兜底。Detox 用例少而精，避免 CI 时间太长导致团队不愿意跑。

### 54. Harness Engineering 的 10+ 任务模板具体有哪些？哪一个对实际研发帮助最大？

模板可以包括需求拆解模板、接口联调模板、Bug 排查模板、代码 review 模板、测试用例补充模板、文档生成模板、上线检查模板、性能分析模板、组件抽象模板、风险评估模板等。

对我帮助最大的是 Bug 排查模板和测试补充模板。Bug 排查模板能强制先收集现象和证据，而不是直接改代码；测试补充模板能帮助把核心路径、异常分支、边界输入列出来，提高回归质量。

### 55. Shopify 黑名单下单拦截应用的拦截时机在哪里？失败兜底怎么处理？

黑名单拦截通常要结合 Shopify 提供的订单、客户或结账相关能力来实现，具体方案取决于店铺权限和应用接入方式。核心思路是在订单生成前后识别风险信息，比如邮箱、手机号、客户 ID、地址等，匹配黑名单规则后阻止或标记风险订单。

失败兜底上，要考虑接口失败、规则读取失败、网络异常等情况。我的处理思路是：关键规则要有日志和告警，不能静默失败；对于无法实时拦截的情况，可以标记订单风险并通知运营处理；后台配置要保留操作记录，方便追踪误拦或漏拦原因。
