# 网站、App、桌面程序从 Idea 到稳定维护的生命周期

一个产品从 idea 到稳定维护，本质上不是“写完上线”这么简单，而是经历：

```text
想法验证 -> 需求定义 -> 技术方案 -> 原型设计 -> 架构搭建 -> 功能开发
  -> 测试验证 -> 灰度/发布 -> 监控反馈 -> 迭代优化 -> 稳定维护
```

网站、App、桌面程序的整体生命周期类似，但在发布方式、运行环境、兼容性、监控和维护策略上有明显差异。

## 一、通用生命周期

### 1. Idea 与业务验证

目标是确认这个产品为什么要做、解决谁的问题、核心价值是什么。

需要明确：

- 目标用户是谁
- 核心场景是什么
- 解决什么痛点
- 成功指标是什么，比如转化率、留存、访问量、任务完成率、崩溃率
- 是 MVP 验证，还是正式产品建设

这一阶段最重要的是避免一开始就陷入技术细节，而是先确认业务方向是否成立。

### 2. 需求定义与范围拆解

把 idea 拆成可交付的需求。

常见产出：

- PRD / 需求说明
- 用户流程图
- 页面结构
- 核心功能列表
- 非功能需求，比如性能、安全、兼容性、可用性
- MVP 范围和后续迭代范围

技术侧需要参与评估：

- 哪些功能必须首期完成
- 哪些可以后置
- 哪些存在技术风险
- 哪些依赖后端、客户端、第三方服务

### 3. 技术方案与架构设计

这一阶段决定产品的技术骨架。

需要考虑：

- 技术栈选择
- 前后端边界
- 数据流设计
- 路由与模块划分
- 状态管理方案
- API 协议设计
- 权限与鉴权
- 日志、监控、错误上报
- 构建、发布、回滚方案
- 后续扩展与维护成本

如果是中大型项目，还需要提前设计工程规范：

- 目录结构
- 命名规范
- ESLint / Prettier
- Git 分支策略
- Code Review 规则
- 测试策略
- CI/CD 流程

### 4. 原型、UI 与交互设计

产品和设计会输出原型和视觉稿，技术侧需要评估实现成本。

重点关注：

- 页面层级是否清晰
- 交互是否符合平台习惯
- 多端适配是否合理
- 异常、空状态、加载态是否完整
- 是否有复杂动效、拖拽、图表、编辑器等高风险模块

对于前端来说，这一阶段要提前识别“看起来简单但实现复杂”的地方，比如：

- 权限组合
- 表单联动
- 大数据量表格
- 可视化画布
- 富文本编辑器
- 多端响应式适配

### 5. 工程搭建与基础能力建设

正式开发前，先把基础工程搭好。

常见内容：

- 项目脚手架
- 路由体系
- 请求封装
- 状态管理
- 组件库接入或二次封装
- 权限控制
- 错误处理
- 环境变量
- Mock 数据
- 测试框架
- 构建与部署脚本

这一阶段的目标是让后续业务开发有统一标准，而不是每个模块各写各的。

### 6. 功能开发与联调

进入业务功能实现阶段。

开发流程通常是：

```text
需求拆解 -> 接口约定 -> 页面开发 -> 状态流转 -> 接口联调
  -> 异常处理 -> 自测 -> 提测
```

需要重点保证：

- 核心链路可用
- 数据状态清晰
- 异常分支完整
- 交互反馈及时
- 代码结构可维护
- 关键模块有测试保障

### 7. 测试验证

测试不只是 QA 的事情，开发也需要完成基础验证。

常见验证范围：

- 功能测试
- 接口联调测试
- 兼容性测试
- 性能测试
- 安全测试
- 回归测试
- 自动化测试
- 灰度验证

前端常用验证方式：

- 单元测试
- 组件测试
- E2E 测试
- Lighthouse / Web Vitals
- 真机测试
- 日志和错误上报检查

### 8. 发布上线

发布阶段关注的是可控、可回滚、可观测。

需要确认：

- 构建产物是否正确
- 环境变量是否正确
- 静态资源缓存策略是否正确
- 接口地址是否正确
- 权限配置是否正确
- 是否支持灰度发布
- 是否有回滚方案
- 监控告警是否可用

### 9. 监控与线上反馈

上线后需要持续观察真实用户环境。

常见监控指标：

- 页面访问量
- 错误率
- 接口失败率
- 白屏率
- 首屏加载时间
- LCP / FCP / CLS / INP
- 崩溃率
- 用户转化率
- 关键链路完成率

这一阶段的重点是从“开发环境可用”转向“真实用户环境稳定”。

### 10. 迭代优化与长期维护

稳定维护阶段主要处理：

- Bug 修复
- 性能优化
- 依赖升级
- 安全漏洞修复
- 架构重构
- 业务迭代
- 技术债治理
- 文档维护
- 新人接手成本降低

长期维护的核心不是代码永远不变，而是让系统在不断变化中仍然可理解、可扩展、可回滚。

## 二、网站的生命周期

网站通常包括官网、商城、后台系统、SaaS 控制台、活动页等。

### 1. 网站从 idea 到上线

```text
业务目标 -> 页面结构 -> 技术选型 -> SEO/性能方案 -> 前后端接口
  -> 页面开发 -> 联调测试 -> 部署上线 -> 数据监控 -> 持续优化
```

网站最核心的特点是：

- 通过浏览器访问
- 发布和回滚相对快
- 受网络、浏览器、设备影响大
- SEO、首屏性能、可访问性通常更重要

### 2. 网站技术关注点

需要重点考虑：

- SSR / SSG / CSR 怎么选
- 路由结构
- SEO Meta 信息
- 静态资源缓存
- CDN 加速
- 图片和字体优化
- 首屏性能
- 浏览器兼容性
- 前端安全，比如 XSS、CSRF、CORS、CSP
- 埋点和转化分析

### 3. 网站功能开发阶段技术选型

| 层级 | 常见技术选型 | 适合场景 | 主要取舍 |
| :--- | :--- | :--- | :--- |
| 前端框架 | React / Vue / Svelte | 中后台、官网、商城、SaaS 控制台 | React 生态和复杂组件能力强；Vue 上手快、模板表达直观；Svelte 编译时优化好但团队生态要评估 |
| 渲染方案 | CSR / SSR / SSG / ISR / RSC | 官网、内容站、商城、后台系统 | CSR 迭代快但首屏和 SEO 弱；SSR/SSG 更利于 SEO 和首屏；RSC 能减少客户端 JS，但架构复杂度更高 |
| 构建工具 | Vite / Webpack / Rspack / Turbopack | 新项目、老项目迁移、大型工程 | Vite 开发体验好；Webpack 生态成熟；Rspack/Turbopack 更关注大型项目构建性能 |
| UI 与样式 | Ant Design / MUI / TailwindCSS / CSS Modules / SCSS | 中后台、设计系统、营销页 | 组件库提效快但风格约束强；Tailwind 适合统一样式约束；CSS Modules/SCSS 更接近传统样式组织 |
| 状态管理 | Redux / Zustand / Jotai / Pinia / React Query | 复杂表单、权限、接口缓存、跨页面状态 | Redux 规范强但样板代码多；Zustand 简洁；React Query 更适合服务端状态缓存 |
| 后端接口 | REST / GraphQL / tRPC / BFF | 前后端分离、聚合接口、复杂数据查询 | REST 简单通用；GraphQL 查询灵活但治理成本高；BFF 能适配前端页面模型但增加服务维护成本 |
| 服务端技术 | Node.js / Go / Java / Python | BFF、中间层、业务服务、数据聚合 | Node 对前端友好；Go 性能和部署简单；Java 适合企业级复杂业务；Python 适合 AI/数据相关服务 |
| 数据存储 | MySQL / PostgreSQL / MongoDB / Redis / Elasticsearch | 交易、内容、配置、缓存、搜索 | 关系型数据库适合强一致业务；MongoDB 适合文档配置；Redis 适合缓存；ES 适合全文搜索 |
| 数据分析 | GA4 / GTM / 神策 / 自研埋点 / BI 看板 | 增长、转化、运营分析 | 第三方接入快；自研灵活但建设成本高；BI 适合经营分析和长期数据沉淀 |

网站选型的核心是先看业务类型：官网和商城优先考虑 SEO、首屏和缓存；中后台优先考虑权限、表单、表格、组件复用和接口稳定性；SaaS 控制台则要更重视模块边界、状态管理和长期可维护性。

### 4. 网站稳定维护

网站上线后，重点维护：

- 页面性能指标
- SEO 收录情况
- 接口稳定性
- 页面错误率
- 静态资源缓存
- 兼容性问题
- 活动页快速交付
- 组件复用和设计规范

面试可以这样说：

> 网站的生命周期更强调快速迭代、性能和可观测性。尤其是商城、官网这类业务，需要从页面结构、SEO、首屏加载、资源缓存、埋点转化和发布回滚几个维度保证稳定。

## 三、App 的生命周期

App 包括原生 App、React Native / Flutter 跨端 App、Hybrid App。

### 1. App 从 idea 到上线

```text
业务场景 -> 端能力评估 -> 技术方案 -> 原型与交互 -> 双端开发
  -> 原生能力联调 -> 真机测试 -> 灰度发布 -> 应用商店审核 -> 线上监控
```

App 和网站最大的区别是：

- 运行在 iOS / Android 系统上
- 需要处理设备权限、系统版本、机型差异
- 发版周期更长
- 应用商店审核和用户升级会影响迭代速度
- 崩溃、卡顿、内存、弱网体验更关键

### 2. App 技术关注点

需要重点考虑：

- iOS / Android 平台差异
- 屏幕适配
- 权限管理
- 网络状态和弱网重试
- 本地缓存
- Push / 消息提醒
- 原生模块通信
- 应用启动速度
- 页面卡顿
- 内存泄漏
- 崩溃收集
- App 发布和灰度

React Native 场景还要关注：

- JS 与 Native 通信边界
- Native Modules 接入
- 页面导航
- 状态管理
- 长列表性能
- 多机型兼容
- 热更新或版本兼容策略

### 3. App 功能开发阶段技术选型

| 层级 | 常见技术选型 | 适合场景 | 主要取舍 |
| :--- | :--- | :--- | :--- |
| 客户端框架 | 原生 iOS / 原生 Android / React Native / Flutter | 高性能原生能力、跨端业务、快速迭代 | 原生体验和系统能力最好但双端成本高；RN 复用 React 生态、适合业务型 App；Flutter 渲染一致性强但生态和包体要评估 |
| UI 与交互 | Native UI / RN 组件 / Flutter Widget / Hybrid WebView | 标准页面、复杂动效、活动页、内嵌页面 | Native UI 性能稳定；RN/Flutter 跨端效率高；WebView 灵活但性能和端能力受限 |
| 状态管理 | Redux / MobX / Zustand / Context / Riverpod | 设备状态、登录态、消息状态、复杂页面流转 | Redux 适合规范化复杂状态；MobX 响应式强；Zustand 简洁；Context 适合轻量全局状态 |
| 端能力接入 | Native Modules / TurboModules / Plugin / JSBridge | 相机、蓝牙、推送、定位、文件、音视频 | 原生桥接能力强但需要端侧协作；JSBridge 适合 Hybrid；新架构性能更好但接入成本更高 |
| 后端接口 | REST / GraphQL / WebSocket / MQTT / gRPC | 普通业务、实时消息、IoT、音视频控制 | REST 通用；WebSocket 适合实时双向通信；MQTT 适合 IoT 设备消息；GraphQL 适合复杂聚合查询 |
| 服务端技术 | Java / Go / Node.js / Python | 账号、设备、消息、配置、推送服务 | Java 适合企业级业务；Go 适合高并发和网关；Node 适合 BFF 和轻服务；Python 常用于 AI/数据处理 |
| 本地数据 | AsyncStorage / MMKV / SQLite / Realm / Keychain | 缓存、离线数据、敏感信息、本地队列 | MMKV 性能好；SQLite 适合结构化数据；Keychain/Keystore 存敏感信息；AsyncStorage 适合轻量配置 |
| 数据与监控 | Crashlytics / Sentry / Firebase / 自研日志 / 埋点平台 | 崩溃、性能、用户行为、设备诊断 | 第三方接入快；自研可控性强；移动端要特别关注崩溃率、卡顿率、启动耗时和弱网成功率 |

App 选型的核心是看端能力深度和迭代效率。如果大量依赖相机、蓝牙、音视频、设备通信，原生协作和桥接设计要提前规划；如果主要是业务页面和多端一致体验，React Native 或 Flutter 更容易提升交付效率。

### 4. App 稳定维护

App 稳定维护重点是：

- 崩溃率
- 卡顿率
- 启动耗时
- 弱网成功率
- 设备兼容性
- 权限异常
- 线上日志
- 版本覆盖率
- 应用商店反馈

面试可以这样说：

> App 的生命周期更强调端侧稳定性和多环境兼容。相比网站，App 发版成本更高，所以前期要更重视真机测试、灰度发布、崩溃监控、弱网处理和版本兼容。

## 四、桌面程序的生命周期

桌面程序包括 Electron 应用、原生桌面客户端、IDE 插件、工具类软件等。

### 1. 桌面程序从 idea 到上线

```text
使用场景 -> 平台选择 -> 架构设计 -> 本地能力接入 -> UI 开发
  -> 安装包构建 -> 自动更新 -> 崩溃监控 -> 用户反馈 -> 长期维护
```

桌面程序的特点是：

- 运行在 Windows / macOS / Linux
- 更依赖本地文件系统、系统权限、进程通信
- 安装包体积和自动更新很重要
- 用户环境差异大
- 长时间运行的稳定性很重要

### 2. 桌面程序技术关注点

需要重点考虑：

- 多平台兼容
- 安装包构建
- 自动更新
- 本地文件读写
- 系统权限
- 主进程与渲染进程通信
- 离线能力
- 数据本地存储
- 崩溃恢复
- 内存占用
- 长时间运行稳定性

如果是 Electron，还要关注：

- 主进程和渲染进程职责边界
- IPC 通信安全
- Node 能力暴露控制
- preload 脚本安全
- 打包体积优化
- 自动更新机制

### 3. 桌面程序功能开发阶段技术选型

| 层级 | 常见技术选型 | 适合场景 | 主要取舍 |
| :--- | :--- | :--- | :--- |
| 桌面框架 | Electron / Tauri / Qt / 原生 Windows/macOS | 跨平台工具、IDE、客户端、重型桌面软件 | Electron Web 生态成熟但包体和内存较大；Tauri 更轻但生态相对新；Qt/原生性能强但开发成本高 |
| 前端界面 | React / Vue / Svelte / Canvas / WebGL | 工具面板、编辑器、可视化、复杂交互 | React/Vue 适合常规 UI；Canvas/WebGL 适合图形和高频绘制；复杂编辑器要关注渲染性能 |
| 本地能力 | Node.js API / Rust Commands / Native Addon / 系统 SDK | 文件、进程、系统托盘、剪贴板、设备调用 | Node 接入快；Rust/Native 性能和安全更好；系统 SDK 能力强但跨平台适配成本高 |
| 进程通信 | Electron IPC / preload / MessageChannel / WebSocket | 主进程与渲染进程通信、插件通信、后台任务 | IPC 要控制权限边界；preload 适合安全暴露能力；WebSocket 适合本地服务或多进程通信 |
| 后端/本地服务 | 内嵌 Node 服务 / Go 本地服务 / Rust 后端 / 云端 API | 本地任务、同步服务、重计算、账号体系 | 内嵌服务方便集成；Go/Rust 更适合高性能本地任务；云端 API 便于统一数据和账号 |
| 数据存储 | SQLite / IndexedDB / LevelDB / 本地文件 / 云数据库 | 离线数据、配置、历史记录、用户文件 | SQLite 适合结构化本地数据；IndexedDB 适合 Web 侧缓存；本地文件适合用户可管理数据；云端适合同步 |
| 更新与监控 | autoUpdater / Sentry / 自研更新器 / 日志上传 | 自动更新、崩溃收集、版本治理 | 自动更新要考虑失败恢复；桌面端日志更依赖本地落盘和用户环境诊断 |

桌面程序选型的核心是平衡 Web 开发效率和本地能力。如果是工具型产品，Electron 能快速复用前端生态；如果对包体、内存、安全要求很高，可以考虑 Tauri、Qt 或原生方案。

### 4. 桌面程序稳定维护

桌面程序维护重点是：

- 崩溃日志
- 自动更新成功率
- 安装失败率
- 系统兼容性
- 内存泄漏
- 本地数据损坏恢复
- 权限异常
- 版本回滚

面试可以这样说：

> 桌面程序的生命周期更强调本地环境、系统能力和长期运行稳定性。相比网站，它不只是页面交互，还要考虑安装、更新、本地文件、权限、进程通信和崩溃恢复。

## 五、三类产品生命周期对比

| 类型 | 发布特点 | 核心关注点 | 稳定维护重点 |
| :--- | :--- | :--- | :--- |
| 网站 | 发布快，回滚快 | SEO、性能、兼容性、缓存、接口稳定 | 页面错误、Web Vitals、接口失败率、转化数据 |
| App | 发版慢，审核和用户升级影响大 | 端能力、权限、机型适配、弱网、崩溃 | 崩溃率、卡顿率、启动耗时、版本兼容 |
| 桌面程序 | 依赖安装包和自动更新 | 本地能力、进程通信、系统兼容、离线能力 | 自动更新、崩溃恢复、内存、权限、本地数据 |

## 六、最佳实践框架

做网站、App 或桌面程序时，不建议只从“用什么框架”开始想，而是先按一套稳定的工程框架做判断：

```text
业务目标 -> 用户场景 -> 平台约束 -> 技术选型 -> 架构边界
  -> 工程规范 -> 质量保障 -> 发布策略 -> 监控反馈 -> 持续演进
```

### 1. 业务先行

先判断产品的核心目标：

- 是拉新、转化、留存，还是内部提效
- 是内容展示、交易闭环，还是复杂生产工具
- 是短期活动，还是长期平台
- 是面向 C 端用户，还是 B 端运营和内部员工

业务目标不同，技术选型也不同。比如官网更重 SEO 和首屏，后台更重权限和可维护性，App 更重端侧稳定性，桌面程序更重本地能力。

### 2. 平台约束

不同平台要先看运行环境：

- Web：浏览器、网络、SEO、缓存、兼容性
- App：iOS/Android、权限、机型、弱网、应用商店
- 桌面：Windows/macOS/Linux、安装包、自动更新、本地文件、系统权限

选型时不要只看技术先进性，还要看平台约束是否匹配。

### 3. 架构边界

提前划清边界，后期维护成本会低很多：

- 前端负责什么
- 后端负责什么
- BFF 是否需要
- 哪些状态在客户端
- 哪些数据必须服务端权威
- 哪些能力需要原生端或系统层支持
- 哪些逻辑可以配置化

常见原则是：展示和交互尽量靠前端，核心业务规则和数据一致性放后端，端能力和系统能力交给原生或桌面主进程。

### 4. 工程规范

项目一开始就要建立最低限度的工程规范：

- TypeScript 类型约束
- ESLint / Prettier
- Git 分支策略
- Commit 规范
- 目录结构约定
- API 类型定义
- 组件边界
- 环境变量管理
- CI 校验
- Code Review

规范不是为了形式，而是为了多人协作、持续迭代和降低交接成本。

### 5. 质量保障

质量保障要覆盖开发前、开发中、上线后：

- 开发前：需求评审、技术方案、风险识别
- 开发中：类型检查、单元测试、组件测试、接口 Mock、Code Review
- 上线前：E2E、兼容性、性能、安全、灰度验证
- 上线后：监控、告警、日志、用户反馈、回滚机制

越是核心链路，越不能只靠人工点点页面。

### 6. 可观测性

上线后要能回答三个问题：

- 用户有没有成功完成任务
- 系统哪里出错了
- 性能瓶颈在哪里

常见监控包括：

- 错误监控：Sentry、Crashlytics
- 性能监控：Web Vitals、启动耗时、卡顿率
- 接口监控：成功率、耗时、错误码
- 用户行为：PV、UV、转化漏斗、关键点击
- 业务指标：订单数、支付成功率、任务完成率

### 7. 演进策略

架构不是一次设计完就结束，而是跟业务一起演进。

常见演进方向：

- 从单页面到模块化
- 从复制页面到组件体系
- 从手工发布到 CI/CD
- 从人工排查到监控告警
- 从单端实现到跨端复用
- 从临时脚本到平台化工具
- 从经验驱动到数据驱动

## 七、不同应用场景的选型建议

### 1. 官网 / 内容站 / 营销页

| 维度 | 推荐选型 | 原因 |
| :--- | :--- | :--- |
| 前端框架 | Next.js / Nuxt / Astro | 重视 SEO、首屏、内容组织 |
| 渲染方式 | SSG / SSR / ISR | 内容可预生成，动态内容可服务端渲染 |
| 样式方案 | TailwindCSS / SCSS / CSS Modules | 页面样式变化多，需要快速还原设计 |
| 数据来源 | CMS / Headless CMS / Markdown / API | 内容运营需要可配置 |
| 部署 | CDN / Vercel / Cloudflare / 静态托管 | 静态资源分发快，回滚简单 |
| 重点指标 | LCP、CLS、SEO 收录、转化率 | 直接影响自然流量和投放转化 |

适合策略：

- 首屏资源尽量轻
- 图片和字体要重点优化
- HTML 不做长缓存，静态资源做 hash 长缓存
- 页面结构语义化，Meta、canonical、OG 信息完整

### 2. 电商网站 / 交易系统

| 维度 | 推荐选型 | 原因 |
| :--- | :--- | :--- |
| 前端框架 | Next.js / React / Vue | 商品、购物车、结算链路复杂 |
| 状态管理 | Zustand / Redux / React Query | 区分客户端状态和服务端状态 |
| 后端接口 | REST / GraphQL / BFF | 聚合商品、库存、价格、优惠、订单数据 |
| 数据存储 | MySQL / PostgreSQL + Redis | 订单和库存需要一致性，热点数据需要缓存 |
| 支付链路 | 服务端创建订单与支付会话 | 金额、库存、订单状态必须以后端为准 |
| 重点指标 | LCP、加购率、结算转化率、支付成功率 | 性能和链路稳定性直接影响收入 |

适合策略：

- 商品价格、库存、订单金额不要只信前端
- 购物车可以前端缓存，但结算必须服务端校验
- 支付、库存扣减、订单状态变更要有事务或一致性设计
- 大促场景要考虑缓存、限流、降级和库存保护

### 3. B 端后台 / SaaS 控制台

| 维度 | 推荐选型 | 原因 |
| :--- | :--- | :--- |
| 前端框架 | React / Vue | 表单、表格、弹窗、权限场景多 |
| UI 体系 | Ant Design / Arco / MUI / 自研组件库 | 快速搭建一致的后台体验 |
| 状态管理 | Redux / Zustand / Pinia / React Query | 权限、筛选、表格、详情页状态复杂 |
| 后端接口 | REST / BFF / GraphQL | 页面经常需要多接口聚合 |
| 数据存储 | MySQL / PostgreSQL / Redis / Elasticsearch | 业务数据、缓存、检索各有职责 |
| 重点指标 | 操作效率、错误率、接口稳定性、权限正确性 | 后台更重稳定和可维护性 |

适合策略：

- 权限模型要提前设计，避免后期散落在页面里
- 表单和表格要抽象公共能力，但不要过早做大而全低代码
- 查询条件、分页、排序、导出要和后端约定统一规范
- 关键操作要有二次确认、审计日志和失败回滚提示

### 4. 低代码 / 工作流 / 可视化编辑器

| 维度 | 推荐选型 | 原因 |
| :--- | :--- | :--- |
| 前端框架 | React + TypeScript | 复杂状态和组件组合更可控 |
| 图形工具 | React Flow / X6 / G6 / D3 / Canvas | 流程编排、节点画布、图可视化 |
| 编辑器 | Monaco Editor / TipTap / ProseMirror | 代码编辑、富文本、模板编辑 |
| 状态管理 | Zustand / Redux / Jotai | 画布状态、选中状态、历史记录复杂 |
| 数据模型 | JSON Schema / DSL / AST | 配置化和可持久化需要稳定模型 |
| 后端接口 | BFF / REST / WebSocket | 保存、预览、协作、执行状态 |

适合策略：

- 先设计 DSL 或配置模型，再设计 UI
- 画布操作要支持撤销、重做、复制、粘贴、导入、导出
- 节点和边要有稳定 ID，避免状态错乱
- 大画布要考虑虚拟化、局部渲染和拖拽性能

### 5. 移动 App / React Native App

| 维度 | 推荐选型 | 原因 |
| :--- | :--- | :--- |
| 跨端框架 | React Native / Flutter | 复用业务逻辑，提高双端交付效率 |
| 原生能力 | Native Modules / TurboModules | 相机、蓝牙、音视频、推送等能力需要端侧支持 |
| 导航 | React Navigation / 原生导航 | 页面栈、Tab、深链等移动端能力 |
| 本地存储 | MMKV / SQLite / Keychain | 缓存、离线、敏感信息分级存储 |
| 通信方式 | REST / WebSocket / MQTT | 普通业务、实时消息、IoT 设备链路 |
| 重点指标 | 启动耗时、崩溃率、卡顿率、弱网成功率 | App 用户体验更依赖端侧稳定性 |

适合策略：

- 复杂端能力不要全塞 RN，需要明确 Native 和 JS 边界
- 长列表、图片、动画要做真机验证
- 弱网、断网、重试、超时、取消请求要提前设计
- 发版要考虑灰度、回滚、版本兼容和应用商店审核

### 6. 桌面工具 / Electron 应用

| 维度 | 推荐选型 | 原因 |
| :--- | :--- | :--- |
| 桌面框架 | Electron / Tauri / Qt | 根据 Web 复用、本地能力、包体要求选择 |
| 前端界面 | React / Vue / Canvas / WebGL | 工具面板、编辑器、图形渲染 |
| 本地能力 | Node API / Rust / Native Addon | 文件、系统、进程、设备调用 |
| 本地数据 | SQLite / IndexedDB / 文件系统 | 离线数据、配置、历史记录 |
| 通信 | IPC / preload / 本地 WebSocket | 主进程、渲染进程、本地服务通信 |
| 重点指标 | 安装成功率、自动更新成功率、崩溃率、内存占用 | 桌面端更重长期运行和本地环境 |

适合策略：

- 渲染进程不要直接暴露完整 Node 能力
- preload 只暴露白名单 API
- 自动更新要支持失败恢复
- 本地数据要考虑损坏恢复和迁移
- 长时间运行要重点排查内存泄漏

### 7. AI 应用 / Agent 控制台

| 维度 | 推荐选型 | 原因 |
| :--- | :--- | :--- |
| 前端框架 | React / Next.js | 对话、任务状态、日志、工具调用 UI 复杂 |
| 通信方式 | SSE / WebSocket / HTTP Streaming | 模型输出需要流式展示 |
| 状态管理 | Zustand / Redux / React Query | 会话、任务、工具调用、执行日志状态多 |
| 后端服务 | Node.js / Python / Go | 接入模型 API、任务编排、工具调用 |
| 数据存储 | PostgreSQL / Redis / Vector DB | 会话、任务状态、缓存、向量检索 |
| 重点指标 | 首 token 时间、任务成功率、工具调用成功率、成本 | AI 应用要同时关注体验、稳定性和成本 |

适合策略：

- 前端要清晰展示模型响应、任务状态、工具调用和错误日志
- 后端要管理上下文、重试、超时、取消、审计
- Prompt、工具 schema、模型参数要版本化
- 成本、延迟、失败率要进入监控

## 八、选型决策清单

做技术选型时，可以按这 10 个问题判断：

1. 这个产品最核心的业务指标是什么？
2. 用户主要在哪个平台使用？
3. 首屏、交互、离线、实时性哪个更重要？
4. 是否需要 SEO？
5. 是否需要端能力或本地系统能力？
6. 数据一致性要求高不高？
7. 团队最熟悉的技术栈是什么？
8. 未来半年业务变化会不会很快？
9. 发布、回滚、监控有没有成熟方案？
10. 这个选型 3 个月后还能不能维护？

面试里可以这样说：

> 我做技术选型不会只看框架热度，而是先看业务目标、运行平台、团队能力和长期维护成本。比如官网和商城更重 SEO、首屏和缓存，后台系统更重权限、表单、数据流和可维护性，App 更重端能力、弱网、崩溃和版本兼容，桌面程序更重本地能力、自动更新和长期运行稳定性。最后再结合团队技术栈、发布体系和监控能力做取舍。

## 九、面试总结版

如果面试官问“一个产品从 idea 到稳定维护的生命周期是什么”，可以这样回答：

> 我会把它拆成几个阶段：首先是 idea 验证，确认目标用户、核心场景和成功指标；然后进入需求拆解和技术方案设计，明确技术栈、架构边界、数据流、权限、发布和监控方案；接着是工程搭建、功能开发、接口联调和测试验证；上线阶段要保证灰度、回滚和监控；上线后再根据错误率、性能指标、用户反馈和业务数据持续迭代。  
> 网站更关注 SEO、首屏性能和浏览器兼容；App 更关注端侧能力、机型适配、弱网、崩溃和发版审核；桌面程序更关注安装更新、本地能力、进程通信和长期运行稳定性。三者流程类似，但稳定维护的重点不一样。
