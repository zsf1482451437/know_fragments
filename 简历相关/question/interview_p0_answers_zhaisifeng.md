## Q3：你简历里的 AI skills / AI Agent 辅助研发具体是什么？

我这里说的 AI skills / AI Agent，不是模型训练，也不是简单让 AI 帮忙写代码，而是把一些重复性、上下文依赖强的研发动作沉淀成结构化流程。

在 Reolink 移动端项目里，设备接入、消息提醒、录像回放这些链路上下文比较长，新人或者跨模块协作时容易出现需求理解不一致、异常排查依赖经验、文档沉淀分散的问题。所以我把流程拆成 5 个环节：需求拆解、接口联调、异常排查、测试补充、文档沉淀。

具体做法是沉淀 10+ 结构化任务模板，比如需求拆解模板会要求说明业务背景、入口页面、接口依赖、异常分支、验收标准；异常排查模板会要求先列日志、复现路径、影响范围，再让 AI 辅助给排查方向。AI 输出后不会直接采纳，会经过代码 review、测试用例、lint/build 或人工验证。

所以它的价值不是替代工程判断，而是把上下文组织、重复分析和文档沉淀流程化。

### 追问回答要点

- 不做模型训练，不说 TensorFlow/PyTorch。
- AI 不是最终决策者，最终靠人工 review 和测试验证。
- 具体产出：`1 套流程 + 10+ 模板 + 5 类场景`。

### 追问逐题回答

#### 不是概念包装的话，具体落地在哪些环节？

具体落地在 5 个环节：需求拆解、接口联调、异常排查、测试补充、文档沉淀。比如需求拆解会要求明确业务背景、入口页面、接口依赖、异常分支和验收标准；异常排查会先输入复现路径、日志和影响范围，再让 AI 辅助给排查方向；测试补充会根据核心链路生成用例清单，再由人工筛选和实现。

我不会直接采纳 AI 输出。代码类结果要经过 diff review、lint、test、build 和人工验证；问题排查类结果要回到日志、复现路径和实际代码验证；文档类结果要和真实业务流程、接口字段、异常分支对齐。AI 只负责辅助组织和初稿，最终判断仍然由人和验证结果决定。

## Q5：LCP 是怎么测的？

### 推荐回答

LCP 我一般分两类看：实验室数据和实际表现。

实验室侧主要用 Lighthouse 和 Chrome DevTools Performance。Lighthouse 看整体 Web Vitals，Performance 看主线程、Network 瀑布流和 LCP element，先确定最大内容元素到底是首屏 Banner、商品主图、活动主视觉图，还是文本块。

实际表现侧，如果项目已有 Web Vitals 上报，就结合线上采样数据看趋势；如果没有固定上报系统，我不会说成接了内部监控，而是以 Lighthouse、Performance 和上线前后的对比数据作为主要口径。

Reolink Web 重点活动页里，LCP 主要受首屏大图、字体加载、首屏 JS 执行和接口返回影响。优化时我主要做了图片压缩与合理尺寸裁剪、非首屏图片懒加载、首屏图避免误 lazy、路由级代码分割、减少首屏无关 JS、字体加载优化。最终重点活动页 LCP 稳定控制在 2.5s 内。

### 追问回答要点

- 工具：`Lighthouse`、`Chrome DevTools Performance`、可选 Web Vitals 上报。
- 图片型 LCP：尺寸、格式、CDN、priority/preload、避免首屏 lazy。
- 文本型 LCP：字体加载、CSS 阻塞、SSR/SSG 输出。
- 数据边界：不默认说接入内部监控平台。

### 追问逐题回答

#### 你用的是 Lighthouse、Chrome Performance 还是线上 Web Vitals 上报？

主要用 Lighthouse 和 Chrome DevTools Performance 做实验室分析。Lighthouse 看整体 Web Vitals，Performance 看主线程、网络瀑布流和 LCP element。如果项目已有 Web Vitals 上报，我会结合线上采样趋势；如果没有固定监控系统，我不会说接了内部平台，只按本地测量和上线前后对比来讲。

#### LCP element 通常是什么？

商城页面里常见 LCP element 是首屏 Banner、商品主图、活动主视觉图，或者首屏最大标题块。具体不能凭经验猜，要在 Lighthouse 或 Performance 里看 LCP marker，先确认它是图片还是文本，再决定优化方向。

#### 如果 LCP 元素是首屏 Banner，你会怎么优化？

我会先检查图片尺寸、格式、压缩质量和加载优先级。首屏 Banner 不应该被 lazy loading 误伤，可以考虑 priority 或 preload 思路；同时使用合适尺寸和响应式图片，避免移动端加载桌面端大图。还要检查首屏 JS 是否阻塞渲染，以及 CSS 和字体是否影响首屏展示。

#### 如果 LCP 元素是文本块，你会怎么优化？

文本型 LCP 重点看字体加载、CSS 阻塞和服务端输出时机。比如使用 next/font 或字体预加载策略减少字体阻塞，控制关键 CSS，确保首屏文本能在 SSR/SSG 阶段尽快输出，而不是等客户端接口回来才渲染。

#### 2.5s 内这个数据是实验室数据还是线上采样趋势？

更稳的说法是：重点活动页通过 Lighthouse 和 Performance 做上线前后对比，LCP 控制在 2.5s 内；如果项目有 Web Vitals 上报，再结合线上采样趋势观察。不要把它说成所有用户、所有网络环境下的绝对值。

#### 没有固定监控系统时，你怎么避免数据说不清？

我会明确说明数据来源：本地 Lighthouse、Chrome Performance、Network 瀑布流、构建体积和上线前后对比。没有固定监控系统就不说“线上监控证明”，而是讲实验室测量方法、测试页面、测试环境和优化前后对比。

## Q6：SEO 具体做了哪些？

### 推荐回答

SEO 我主要做基础 SEO 和页面结构优化，目标是让商品页、活动页、解决方案页更容易被搜索引擎理解和收录。

第一是 Meta 信息维护，比如 title、description、canonical、open graph，确保不同商品和活动页面有明确主题。

第二是语义化结构，页面保留清晰的 h1、h2、section、article 层级，避免核心内容全部由 div 堆叠。

第三是内容可索引，商品标题、卖点、规格参数、FAQ、活动规则尽量在 SSR/SSG 阶段输出，不把核心内容完全依赖客户端接口渲染。

第四是图片和媒体优化，包括图片 alt、尺寸控制、非首屏懒加载，避免图片资源拖慢页面。

当时覆盖了 20+ 商品/活动页面的基础 SEO 配置，和性能优化一起支撑自然流量获取。

### 追问回答要点

- SEO 不是“堆关键词”，而是 `元信息 + 结构 + 可索引内容 + 性能`。
- SSR/SSG 对 SEO 友好，纯 CSR 对搜索引擎理解有不确定性。
- 验收方式：看 HTML 源码、Meta、移动端访问、核心内容是否可见。

### 追问逐题回答

#### title、description、canonical、OG 分别解决什么问题？

title 是页面标题，影响搜索结果和页面主题判断；description 是摘要，帮助搜索引擎和用户理解页面内容；canonical 用来声明规范 URL，避免重复内容分散权重；OG 信息主要服务社交分享展示，比如分享标题、描述和图片。

#### h1/h2/section/article 这些语义化标签怎么使用？

页面通常只保留一个主 h1，表示页面核心主题，比如商品名或活动主题。h2/h3 用于商品卖点、规格参数、FAQ 等层级。section/article 用来表达内容区块，不把所有内容都堆在 div 里。这样搜索引擎和辅助工具更容易理解页面结构。

#### 商品卖点、规格参数、FAQ 怎么保证可索引？

核心内容尽量在 SSR/SSG 阶段输出到 HTML，而不是完全依赖客户端接口渲染。商品卖点、规格参数、FAQ 这些内容要有稳定的文本结构和语义标签。上线前可以检查页面源码或禁用 JS 的可见内容，确认搜索引擎能拿到主要信息。

#### SSR/SSG/CSR 对 SEO 的影响是什么？

SSR 和 SSG 更利于 SEO，因为核心 HTML 内容在首屏请求时就能返回。CSR 完全依赖浏览器执行 JS 后再渲染，搜索引擎虽然能处理部分 JS，但稳定性和时效性不如 SSR/SSG。商品页、活动页、解决方案页这类需要自然流量的页面，优先考虑 SSR/SSG。

#### 图片 alt 怎么写才不是形式主义？

alt 应该描述图片和业务内容的关系，而不是堆关键词。比如商品主图可以写产品型号和核心场景，卖点图可以描述功能点。装饰性图片可以留空或弱化，避免所有图片都写重复关键词。

#### 20+ 商品/活动页面 SEO 配置怎么验收？

验收时看每个页面的 title、description、canonical、h1、核心内容、图片 alt 是否完整；检查 HTML 源码里是否有核心内容；移动端访问是否正常；必要时用 Lighthouse SEO 检查基础问题。这个验收偏基础 SEO，不会说成完整搜索排名提升。

## Q12：React Native 状态怎么拆？

### 推荐回答

我会按业务职责拆状态，不把所有状态塞进一个大对象。Reolink 移动端主要拆成三类：设备状态、消息状态、页面展示状态。

设备状态包括在线/离线、连接中、预览中、回放中等和设备生命周期相关的数据。消息状态包括告警消息、未读状态、推送入口状态。页面展示状态包括 loading、弹窗、筛选条件、当前 tab、分页等 UI 状态。

技术实现上，简单 UI 状态用 useState；复杂状态流转，比如设备连接、预览、回放这类状态机式逻辑，用 useReducer；跨组件复用逻辑抽成自定义 Hook，比如 useDeviceStatus、useMessageList。

这样拆的好处是状态边界清楚，一个状态变化不会带动整个页面无意义重渲染，也更方便写 Jest 或 RNTL 测试。

### 追问回答要点

- useState：简单局部 UI。
- useReducer：复杂状态流转。
- 自定义 Hook：复用业务逻辑。
- 性能：减少无关渲染，配合 useMemo/useCallback/React.memo。

### 追问逐题回答

#### 设备状态、消息状态、页面展示状态分别包含什么？

设备状态包括在线/离线、连接中、预览中、回放中、连接失败等；消息状态包括告警列表、未读数、已读状态、推送入口状态；页面展示状态包括 loading、弹窗、tab、筛选条件、分页、空状态和错误提示。

#### 为什么不用一个全局大状态对象？

全局大状态容易导致状态边界不清楚，一个字段变化触发大量无关组件更新，也会让测试和排查变困难。按业务职责拆分后，状态变化范围更小，组件依赖更清楚，也更容易定位问题。

#### useState、useReducer、自定义 Hook 怎么分工？

useState 用于简单 UI 状态，比如弹窗、loading、tab；useReducer 用于复杂状态流转，比如设备连接、预览、回放这类有多个状态和事件的场景；自定义 Hook 用于复用业务逻辑，比如 useDeviceStatus、useMessageList，把接口请求、状态流转和异常处理封装起来。

#### 状态拆分如何减少无意义渲染？

状态拆分后，组件只订阅自己真正需要的状态。比如消息列表更新不应该影响设备预览区域，弹窗状态变化也不应该让整个设备列表重渲染。再配合 React.memo、useMemo、useCallback，可以进一步减少列表项和子组件重复渲染。

#### 状态拆分和测试用例有什么关系？

状态拆得清楚后，测试可以按状态流转写得更稳定。比如设备连接从 loading 到 success/fail，可以用 Jest 测 reducer 或 hook；页面交互可以用 RNTL 测按钮点击后的状态变化；核心端到端链路再用 Detox 覆盖。

## Q13：长列表卡顿怎么定位？

### 推荐回答

我会先判断卡顿来源：是数据量太大、列表渲染太重，还是图片资源过大。

React Native 长列表一般优先检查 FlatList 或 SectionList，而不是直接用 ScrollView 渲染大量数据。重点看 keyExtractor 是否稳定，initialNumToRender、windowSize、maxToRenderPerBatch、removeClippedSubviews 等参数是否合理，以及 item 组件是否因为 props 或 inline 函数反复重渲染。

定位工具上，我会先用 RN 性能面板、React DevTools Profiler、Flipper 做初步判断；必要时结合 Android Studio Profiler 或 Xcode Instruments 看 JS 线程和 UI 线程是否阻塞。

优化上主要做列表虚拟化、item 组件 React.memo、useCallback 固定事件函数、useMemo 缓存计算结果、图片缩略图和缓存、避免在列表项里做复杂计算。

### 追问回答要点

- 不用 ScrollView 承载大量数据。
- JS 线程阻塞：复杂计算、频繁 setState。
- UI 线程掉帧：图片、布局、原生渲染压力。
- 55fps 是重点场景验证口径，不说成全量线上绝对值。

### 追问逐题回答

#### 你怎么判断是数据量问题、渲染问题还是图片问题？

先看卡顿发生时机。如果数据一多就卡，优先看列表虚拟化和分页；如果每次状态变化都卡，优先看重复渲染和 item 组件；如果滚动到图片区域明显卡，优先看图片尺寸、缓存和加载策略。工具上用 RN 性能面板、React DevTools Profiler、Flipper 或原生 profiler 辅助判断。

#### FlatList 哪些参数会影响性能？

常见参数包括 initialNumToRender、windowSize、maxToRenderPerBatch、removeClippedSubviews、keyExtractor。它们会影响首屏渲染数量、窗口外组件保留数量、每批渲染数量和列表项复用稳定性。参数要结合页面复杂度和设备性能调，不是越大越好。

#### 为什么不直接用 ScrollView 渲染大量数据？

ScrollView 会一次性渲染所有子项，数据量大时会造成首屏慢、内存高和滚动卡顿。FlatList/SectionList 有虚拟化机制，只渲染可见区域附近的内容，更适合大量列表数据。

#### React.memo、useCallback、useMemo 分别解决什么？

React.memo 用于避免 props 没变时子组件重复渲染；useCallback 用于稳定函数引用，避免传给子组件的事件函数每次都变；useMemo 用于缓存计算结果，避免每次渲染都重新计算。但它们不是越多越好，只在渲染成本或引用稳定确实有收益时使用。

#### Flipper、RN Performance Monitor、Android Studio Profiler、Xcode Instruments 怎么配合使用？

RN Performance Monitor 可以快速看 JS/UI 帧率趋势；Flipper 适合看 RN 调试、网络、日志和部分性能信息；Android Studio Profiler / Xcode Instruments 更适合深入看原生线程、内存和 CPU。一般先用轻量工具定位方向，再用原生 profiler 深挖。

## Q20：JS 体积从 3MB 到 600KB 具体怎么做？

### 推荐回答

我会先用打包分析工具看体积来源，比如哪些依赖最大、是否存在重复依赖、哪些非首屏代码被打进首页 bundle。

具体优化分几类：

第一是代码分割。页面级和非首屏模块使用动态导入，比如 next/dynamic 或 import()，避免首页一次性加载不需要的模块。

第二是依赖治理。检查工具库、日期库、图表库这类大包，如果只使用少量能力，就换成轻量库或按需引入，避免整包导入。

第三是 Tree Shaking。优先选择支持 ESM 的依赖，检查 sideEffects 配置，避免 CommonJS 或错误构建配置导致无效代码无法摇掉。

第四是重复依赖排查。通过 bundle analyzer 和 pnpm why 观察是否存在多个版本依赖被重复打包。

最后通过构建产物体积、Network 下载体积和 Lighthouse 结果对比，验证优化确实减少了首屏负担。

### 追问回答要点

- 工具：`@next/bundle-analyzer`、构建产物分析、`pnpm why`。
- 手段：动态导入、依赖替换、按需引入、ESM、sideEffects。
- 风险：不能只把体积转移到后续关键路径。

### 追问逐题回答

#### 你用什么工具分析 bundle？

主要用 @next/bundle-analyzer 看构建产物里哪些模块体积大、是否有重复依赖；也会结合 Network 下载体积和 Lighthouse 看首屏资源影响。如果是 pnpm 项目，可以用 pnpm why 查某个包为什么被安装和引用。

#### 哪些依赖最大，怎么判断是否替换？

通常关注工具库、日期库、图表库、富文本或多媒体相关库。判断是否替换时看三个点：是否只用了少量能力、是否支持按需引入、是否有更轻量替代。替换前要确认 API 差异和业务风险，不能为了体积盲目替换。

#### next/dynamic / 动态 import 用在哪里？

用在非首屏组件、重型组件、低频弹窗、图表模块、复杂交互模块这些不需要首屏立即加载的地方。目标是减少首页初始 bundle，让用户先看到核心内容。

#### 重复依赖怎么排查？

通过 bundle analyzer 看是否打进多个版本的同类依赖，再用 pnpm why 或 lockfile 查依赖来源。处理方式可能是统一版本、升级依赖、替换包，或者调整依赖声明。

#### 优化后怎么验证不是只转移了体积？

除了看首页 bundle 下降，还要看用户关键路径是否变好，比如 LCP、Network 下载体积、交互延迟、懒加载模块加载时机。如果只是把首屏体积挪到用户马上点击的路径上，体验不一定变好，所以要结合业务路径验证。

## Q21：LCP 元素是什么？

### 推荐回答

LCP 元素不是固定的，要看具体页面。商城首页或活动页里，常见 LCP 元素一般是首屏 Banner、商品主图、活动主视觉图，或者首屏最大标题块。

我会用 Lighthouse 明细或 Chrome Performance 里的 LCP marker 先确认它是图片还是文本。

如果是图片型 LCP，优化重点是图片格式、尺寸、CDN、加载优先级、避免首屏图片被 lazy loading 误伤。Next.js 项目里可以结合 next/image、priority 或 preload 思路提升首屏图片加载优先级。

如果是文本型 LCP，重点看字体加载、CSS 阻塞和 SSR/SSG 输出时机，比如使用 next/font 减少字体阻塞和布局抖动。

所以 LCP 优化的第一步不是直接改代码，而是先确认最大内容元素和加载链路。

### 追问回答要点

- 图片型 LCP：格式、尺寸、CDN、priority/preload。
- 文本型 LCP：字体、CSS、SSR/SSG。
- 验证：优化前后比较 LCP 时间和 Network 瀑布流。

### 追问逐题回答

#### 首页 LCP 是图片还是文本？

这个不能固定说，要以具体页面测量结果为准。商城首页常见是首屏 Banner 或主视觉图，也可能是首屏最大标题块。面试时我会说先通过 Lighthouse 或 Performance 确认 LCP element，再针对图片或文本采取不同策略。

#### 图片型 LCP 怎么优化？

优化图片格式、尺寸、压缩质量和 CDN；首屏图不要 lazy loading；必要时用 priority 或 preload 提升加载优先级；移动端和桌面端使用合适尺寸的响应式图片，避免加载过大的桌面图。

#### 文本型 LCP 怎么优化？

重点优化字体加载、CSS 阻塞和服务端输出。可以用 next/font 或字体预加载减少字体阻塞，确保关键文本在 SSR/SSG 阶段输出，减少客户端等待接口后才渲染的情况。

#### next/image、next/font 分别解决什么问题？

next/image 主要解决图片尺寸、格式、懒加载、响应式图片和加载优先级问题；next/font 主要解决字体加载、字体文件托管和布局抖动问题。两者都服务于首屏体验和 Web Vitals。

#### LCP 从 4s 到 2s 以内的验证方式是什么？

通过优化前后 Lighthouse、Performance、Network 瀑布流和构建产物对比验证。口径上我会说这是重点页面在测试环境和上线前后对比中的结果，不说成所有用户环境的绝对值。

## Q32：你搭建 ESLint + Prettier、Commitlint + Husky 的目的是什么？

### 推荐回答

这套规范的目的不是为了“形式统一”，而是减少团队协作里的低级问题，把代码风格和提交规范前置到本地开发阶段。

ESLint 主要解决代码质量和潜在问题，比如未使用变量、Hooks 规则、可能的错误写法。Prettier 主要解决格式统一，避免团队在缩进、换行、分号这些问题上浪费 review 时间。

Commitlint + Husky 主要是把提交信息和提交前检查自动化。Husky 可以在 pre-commit 或 commit-msg 阶段触发 lint、format 或 commit message 校验；Commitlint 保证提交信息符合约定，后续排查版本、生成 changelog 或回溯问题更方便。

我理解规范不能一刀切。历史项目接入时要先处理增量代码，必要时对旧代码分批修复，避免一次性改动过大影响业务迭代。

### 追问回答要点

- ESLint：质量问题。
- Prettier：格式统一。
- Husky：本地钩子。
- Commitlint：提交语义。
- 取舍：先控增量，再治理存量。

### 追问逐题回答

#### ESLint 和 Prettier 分别解决什么问题？

ESLint 解决代码质量和潜在错误，比如未使用变量、Hooks 规则、可能的问题写法；Prettier 解决格式统一，比如缩进、换行、分号、引号。ESLint 更偏质量，Prettier 更偏格式。

#### Commitlint 和 Husky 如何接入提交流程？

Husky 负责 Git hooks，比如 pre-commit 或 commit-msg；Commitlint 在 commit-msg 阶段校验提交信息是否符合规范。pre-commit 阶段也可以接 lint-staged，只检查本次改动文件，减少全量检查成本。

#### 规范太严格影响效率怎么办？

先区分规则价值。影响质量的规则保留，比如 Hooks、潜在错误；纯风格问题交给 Prettier 自动修复。历史项目接入时先控制增量，不一次性要求全项目修完，避免影响业务交付。

#### 如何处理历史代码不符合规范的问题？

先做增量治理，对新增和修改文件执行规范；存量代码分批处理，或者在业务迭代触达时顺手修复。不要一次性格式化大量历史文件，否则 diff 太大，review 成本和风险都高。

#### 代码规范如何和团队 code review 配合？

自动化工具处理格式和基础规则，code review 更关注业务逻辑、边界条件、性能、可维护性和安全问题。这样 review 不会浪费在缩进和命名这类可自动处理的问题上。

## Q35：React Hooks 的核心使用原则是什么？

### 推荐回答

我理解 React Hooks 的核心是让函数组件具备状态和副作用管理能力，但关键不是“会用 API”，而是把状态、副作用和复用逻辑拆清楚。

useState 适合简单局部状态；useReducer 适合复杂状态流转，比如设备连接、预览、回放这类状态机式逻辑；useEffect 处理副作用，但依赖项要基于实际使用的变量来写，避免漏依赖导致闭包问题，也避免不必要依赖导致重复请求。

useMemo 和 useCallback 不是越多越好，只有在计算成本较高、或者为了稳定传给子组件的引用时才使用。React.memo 也只有在 props 稳定、组件渲染成本较高时才有意义。

自定义 Hook 的边界是复用业务逻辑，而不是把页面代码简单搬进去。比如 useDeviceStatus、useMessageList 这类 Hook 可以封装接口请求、状态流转和异常处理，让页面组件更关注组合和展示。

### 追问回答要点

- useEffect：依赖项按使用变量确定。
- useMemo/useCallback：解决计算或引用稳定，不滥用。
- React.memo：props 稳定才有效。
- 自定义 Hook：复用逻辑，不承载页面所有职责。

### 追问逐题回答

#### useEffect 依赖项怎么判断？

依赖项应该包含 effect 内部使用到、且来自组件作用域的变量和函数。不能为了少执行就漏依赖，否则容易出现闭包旧值；如果依赖导致频繁执行，要考虑拆分 effect、稳定函数引用或调整状态结构，而不是强行忽略规则。

#### useMemo 和 useCallback 什么时候不该用？

计算很轻、组件不频繁渲染、没有传给 memo 子组件的引用，不需要用。滥用 useMemo/useCallback 会增加代码复杂度，也有维护成本。只有在计算成本高、引用稳定能减少子组件渲染时才有价值。

#### 自定义 Hook 的边界是什么？

自定义 Hook 应该封装可复用的状态和副作用逻辑，比如请求、状态流转、异常处理。它不应该变成页面组件垃圾桶，把所有页面逻辑都塞进去。页面负责组合和展示，Hook 负责可复用逻辑。

#### React.memo 为什么有时没效果？

常见原因是 props 每次都变，比如传入 inline 函数、对象、数组；或者组件本身渲染成本很低，memo 的比较成本反而不划算。React.memo 需要和稳定 props、useCallback、useMemo 配合才更有效。

#### Hooks 状态拆分如何影响可测试性？

状态拆分后，每个 Hook 或 reducer 的输入输出更清楚，可以单独测状态流转和异常分支。比如设备连接状态可以独立测试 loading、success、fail，而不用启动整个页面。

## Q36：TypeScript 在项目里解决了什么问题？

### 推荐回答

TypeScript 在项目里主要解决三个问题：接口契约、组件契约和重构安全。

接口契约上，后端或 GraphQL 返回的数据结构可以定义类型，减少字段写错、空值未处理的问题。组件契约上，props 类型能明确组件需要什么数据，比如商品卡片需要 title、price、image、href、tags，调用方传错字段时能提前发现。

在 React Native 里，TypeScript 也能约束 Native Modules 的 JS 调用层，比如入参、返回值、错误码，减少 JS 和原生协作时的低级错误。

在 Shopify 项目里，Prisma schema 生成的类型也能帮助服务端处理店铺信息、黑名单规则、配置数据时更稳定。

我不会为了类型而类型，业务里尽量避免 any 泛滥，但也会在第三方库类型不完整时做局部类型补充或封装，保证主要业务链路类型清晰。

### 追问回答要点

- 组件 props 类型：约束调用方。
- 接口类型：减少字段错误和空值问题。
- Native Modules：约束 JS/原生边界。
- Prisma：数据模型类型化。
- any：可临时兜底，但不能扩散到核心链路。

### 追问逐题回答

#### 组件 props 类型怎么设计？

先根据组件职责定义必要字段和可选字段。比如商品卡片需要 title、image、href，price、tags 可以根据业务可选。复杂配置要拆成子类型，避免一个巨大 props。事件回调也要明确参数类型。

#### 接口返回类型怎么维护？

如果是 GraphQL，可以基于查询结构维护类型；普通接口则定义 response 类型，并在数据转换层做兜底处理。核心是不要让接口原始结构直接散落到各个组件里，而是通过 service 或 adapter 收口。

#### any 过多会带来什么问题？

any 会绕过类型检查，让 TypeScript 失去价值。字段写错、空值未处理、调用参数错误都会延后到运行时暴露。可以局部使用 unknown 或临时 any，但核心业务链路要尽量补齐类型。

#### 泛型在业务组件中有哪些实际场景？

常见场景是列表组件、表单组件、接口响应结构、选择器组件。比如通用 Select 可以用泛型约束 option 的 value 类型，接口响应可以用 ApiResponse<T> 表达不同业务数据。

#### TypeScript 如何减少跨团队联调成本？

类型能把接口字段、组件入参、Native Module 调用约定提前暴露出来。联调时如果字段变化或入参不匹配，编译阶段就能发现一部分问题，减少运行时才发现的低级错误。

## Q40：AI skills 辅助研发流程具体怎么设计？

### 推荐回答

我设计 AI skills 辅助研发流程时，核心原则是让 AI 参与“结构化辅助”，而不是替代工程判断。

流程主要覆盖 5 个环节：需求拆解、接口联调、异常排查、测试补充、文档沉淀。

需求拆解阶段，模板会要求明确业务背景、入口页面、涉及接口、异常分支、验收标准。接口联调阶段，会整理请求参数、返回结构、错误码和兜底状态。异常排查阶段，会要求先提供复现路径、日志、影响范围，再让 AI 辅助列排查方向，避免空猜。测试补充阶段，会根据核心链路生成用例清单，再由人工筛选哪些适合 Jest、RNTL 或 Detox。文档沉淀阶段，会把最终结论整理成新人可复用的排查手册或任务模板。

这套流程沉淀了 10+ 结构化任务模板，覆盖 5 类高频研发场景。它的价值是提高需求理解、异常排查和文档输出的一致性，但最终仍然要靠人工 review、测试验证和业务验收。

### 追问回答要点

- AI 不替代判断，只辅助上下文组织和重复分析。
- 校验方式：diff review、lint、test、build、人工验收。
- 不适合 AI 的任务：业务决策、权限安全策略、无上下文的复杂重构。

### 追问逐题回答

#### 需求拆解模板包含哪些字段？

包含业务背景、目标用户、入口页面、核心流程、涉及接口、状态流转、异常分支、验收标准、影响范围和待确认问题。模板的价值是把需求从一句话拆成可开发、可验证的任务。

#### 代码草拟后如何进行人工校验？

先看 diff 是否符合需求，再检查是否符合现有目录结构、组件风格、类型约束和边界处理。然后跑 lint、test、build，必要时本地验证核心流程。AI 草稿只能作为初稿，不能绕过 review 和验证。

#### 异常排查如何避免 AI 胡乱猜？

先给足事实：复现步骤、日志、错误堆栈、影响范围、最近改动、相关代码。要求 AI 基于证据列假设和验证步骤，而不是直接给结论。没有日志或复现路径时，不让 AI 编原因。

#### 测试补充如何确保覆盖真实核心链路？

先从业务核心链路反推测试，而不是从代码覆盖率反推。比如设备接入、消息提醒、录像回放是高频链路，就覆盖成功、失败、空状态、异常提示和关键交互。低价值 UI 不用过度写 E2E。

#### 文档沉淀如何复用到新人 onboarding？

文档不只写结论，还要写背景、入口、接口、常见异常、排查步骤和验收方式。新人接手类似问题时，可以按文档找到上下文和处理路径，减少反复问人的成本。

## Q44：你如何处理需求不清晰或频繁变更？

### 推荐回答

我一般先把需求从“描述”拆成可确认的边界：业务目标、入口页面、用户路径、接口依赖、异常分支、验收标准。

需求变更时，我会判断变更影响范围：是文案和配置调整，还是组件结构、接口字段、业务流程变化。小变更直接处理；影响核心链路的变更要重新确认排期和风险。

AI skills 在这里可以辅助生成需求拆解模板和验收清单，但最终需求边界一定要和业务、设计、后端确认。

### 追问回答要点

- 拆解维度：目标、入口、路径、接口、异常、验收。
- 活动页减少返工：检查清单 + 联调问题手册。
- 变更评估：配置变更 vs 结构/流程变更。

### 追问逐题回答

#### 你如何拆需求和确认边界？
