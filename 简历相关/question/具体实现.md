# React Native 实时预览链路具体实现

用途：用于回答面试中“实时预览是怎么实现的”“Native Modules 怎么封装”“为什么要封 previewService”“状态怎么流转”“怎么测试”这类追问。

注意：下面代码是面试理解稿和抽象示例，不是项目真实源码。真实项目中模块名、事件名、错误码会以团队实际封装为准。

## 1. 一句话回答

实时预览不是 RN 直接实现底层播放，而是 RN 侧对接项目已有的原生预览桥接层。我的职责主要是把桥接层暴露的播放能力封装成 `previewService` 和 `usePreview`，再把原生事件映射成页面状态，处理加载、播放、失败、重连、退出清理和测试验证。

调用链路可以这样理解：

```text
PreviewScreen 页面
  -> usePreview Hook
  -> previewService
  -> NativeModules.PreviewModule 原生预览桥接层
  -> Android/iOS 原生播放模块或设备 SDK
```

## 1.1 智能安防场景适合的双端音视频技术

先明确边界：下面是智能安防 App 常见的音视频技术方案，用于理解实时预览链路。面试时如果没有项目事实支撑，不要说“项目一定用了某个具体播放器或协议”，更稳的说法是“底层由原生播放模块或设备 SDK 承接，RN 侧负责调用封装和状态接入”。

智能安防实时预览通常关注 5 个目标：

- 低延迟：用户打开设备后要尽快看到画面。
- 稳定性：弱网、设备离线、网络切换时要有重连和兜底。
- 双端一致：Android / iOS 对外暴露给 RN 的接口和事件要尽量统一。
- 硬解能力：视频解码尽量走系统硬件能力，降低 CPU 和发热。
- 安全性：设备鉴权、传输加密、权限控制不能只靠前端页面判断。

常见技术层级可以这样理解：

```text
业务层：设备 ID、码流类型、清晰度、静音、重试
  -> RN 调用层：previewService / usePreview
  -> 原生桥接层：统一 Android/iOS 调用接口和事件
  -> 播放能力层：ExoPlayer / AVPlayer / 厂商设备 SDK / 自研播放 SDK
  -> 媒体能力层：H.264/H.265、AAC/Opus、硬解码、缓冲、重连
  -> 传输能力层：WebRTC / RTSP / 私有 P2P / Relay / HTTPS 拉流
```

不同方案的适用场景：

| 技术 | 适合场景 | 面试表达边界 |
|---|---|---|
| `WebRTC` | 低延迟实时预览、双向音频、弱网重连 | 可以说适合实时互动场景，但不要说项目一定使用，除非确认 |
| `RTSP` | 摄像头视频流、局域网或设备侧标准流 | 常见于摄像头设备，但移动端直接播放通常需要原生 SDK 或播放器承接 |
| `HLS` | 直播/回放、CDN 分发、延迟要求不极致的场景 | 不适合强调“极低延迟实时预览”，更适合回放或内容分发 |
| `ExoPlayer` | Android 原生播放承载 | 只有确认 Android 用它时才具体说 |
| `AVPlayer` | iOS 原生播放承载 | 只有确认 iOS 用它时才具体说 |
| 厂商设备 SDK / 自研 SDK | 设备连接、P2P、加密、私有协议、播放能力整合 | 最适合用“底层设备 SDK 或原生播放能力”概括 |

如果面试官问“智能安防 App 为什么不能只用普通视频播放器”，可以这样答：

```text
普通播放器主要解决 URL 播放问题，但智能安防实时预览还涉及设备鉴权、P2P 或中继连接、码流切换、设备在线状态、弱网重连、权限控制、错误码映射和双端事件回传。所以项目里通常会由原生播放模块或设备 SDK 先承接这些底层能力，再通过 Native Module 暴露给 RN。
```

如果面试官问“你负责哪一层”，可以这样答：

```text
底层协议、编解码和原生播放器不是我主导的部分。我负责 RN 侧对原生预览桥接层的调用封装，比如 startPreview、stopPreview、retryPreview、switchQuality，以及 loading、playing、error、reconnecting 这些状态接入和 UI 兜底。
```

## 1.2 本项目更稳的技术方案选择

结合智能安防 App 的业务特征，更稳的方案不是让 RN 直接处理视频播放，也不是简单理解成“播放一个 URL”，而是：

```text
RN 页面
  -> usePreview / previewService
  -> PreviewModule 原生预览桥接层
  -> 厂商/自研设备 SDK
  -> P2P / Relay / 局域网连接
  -> 原生播放能力与硬解码
```

选择这个方案的原因：

- 设备接入不是普通 URL 播放，需要设备鉴权、在线状态、P2P 或中继连接。
- 实时预览要求低延迟和弱网重连，单纯 HLS 更偏回放或内容分发，不适合作为主实时预览口径。
- Android / iOS 底层播放能力和 SDK 接口不同，需要原生桥接层先抹平差异。
- RN 更适合处理业务状态、页面 UI、错误兜底和资源释放，不适合直接处理编解码和私有协议。

面试时可以这样说：

```text
我会把它理解成“设备 SDK 承接底层连接和播放能力，原生预览桥接层统一双端接口，RN 侧封装业务调用和状态流转”。这样既符合智能安防业务，也能把我的职责边界讲清楚。
```

## 3. Native Module 接口抽象

RN 侧只认桥接层暴露出来的接口，不直接认 Android/iOS SDK。

```ts

  startPreview(params: {
    deviceId: string;
    streamType: string;
    quality: string;
    muted: boolean;
  }): Promise<void>;

  stopPreview(params: { deviceId: string }): Promise<void>;

  switchQuality(params: {
    deviceId: string;
    quality: string;
  }): Promise<void>;

  setMuted(params: {
    deviceId: string;
    muted: boolean;
  }): Promise<void>;
}

const { PreviewModule } = NativeModules as {
  PreviewModule: NativePreviewModule;
};
```

面试解释：

```text
这里的 PreviewModule 可以理解为项目里的原生预览桥接层。它不是播放器内核，而是把 startPreview、stopPreview、清晰度切换、静音这些能力暴露给 RN。
```

## 3.1 原生预览桥接层实现思路

原生预览桥接层的核心职责是：**把 Android/iOS 设备 SDK 或播放能力封装成 RN 能调用的一组方法和事件**。

它通常做 5 件事：

- 参数转换：把 RN 的 `deviceId / streamType / quality / muted` 转成设备 SDK 需要的参数。
- 启停控制：封装 `startPreview / stopPreview / switchQuality / setMuted`。
- 状态回传：把底层播放状态转成 `onPlayerLoading / onPlayerStarted / onPlayerError / onNetworkReconnect`。
- 错误码统一：把 Android/iOS 或设备 SDK 的错误码映射成 RN 可理解的错误类型。
- 资源释放：页面退出时停止播放、移除监听、释放播放器或连接资源。

Android 侧可以抽象成这样：

```kotlin
class PreviewModule(
  reactContext: ReactApplicationContext,
  private val deviceSdk: DevicePreviewSdk
) : ReactContextBaseJavaModule(reactContext) {

  override fun getName(): String = "PreviewModule"

  @ReactMethod
  fun startPreview(params: ReadableMap, promise: Promise) {
    val deviceId = params.getString("deviceId") ?: ""
    val streamType = params.getString("streamType") ?: "main"
    val quality = params.getString("quality") ?: "hd"
    val muted = params.getBoolean("muted")

    sendEvent("onPlayerLoading", mapOf("deviceId" to deviceId))

    deviceSdk.startPreview(
      deviceId = deviceId,
      streamType = streamType,
      quality = quality,
      muted = muted,
      onStarted = {
        sendEvent("onPlayerStarted", mapOf("deviceId" to deviceId))
      },
      onError = { errorCode ->
        sendEvent(
          "onPlayerError",
          mapOf("deviceId" to deviceId, "code" to errorCode)
        )
      },
      onReconnect = {
        sendEvent("onNetworkReconnect", mapOf("deviceId" to deviceId))
      }
    )

    promise.resolve(null)
  }

  @ReactMethod
  fun stopPreview(params: ReadableMap, promise: Promise) {
    val deviceId = params.getString("deviceId") ?: ""
    deviceSdk.stopPreview(deviceId)
    promise.resolve(null)
  }

  private fun sendEvent(eventName: String, payload: Map<String, Any>) {
    val params = Arguments.createMap()
    payload.forEach { (key, value) ->
      when (value) {
        is String -> params.putString(key, value)
        is Boolean -> params.putBoolean(key, value)
      }
    }

    reactApplicationContext
      .getJSModule(DeviceEventManagerModule.RCTDeviceEventEmitter::class.java)
      .emit(eventName, params)
  }
}
```

## 4. previewService 封装

`previewService` 的职责是把桥接层接口转换成业务侧稳定接口，并统一处理参数转换和错误映射。

```ts
const mapNativeError = (error: any): PreviewError => {
  const code = error?.code;

  if (code === 'PERMISSION_DENIED') {
    return {
      type: 'permission_denied',
      message: '缺少预览权限，请检查授权设置',
      rawCode: code,
    };
  }

  if (code === 'NETWORK_UNAVAILABLE') {
    return {
      type: 'network_error',
      message: '网络异常，请稍后重试',
      rawCode: code,
    };
  }

  if (code === 'DEVICE_OFFLINE') {
    return {
      type: 'device_offline',
      message: '设备离线，请检查设备状态',
      rawCode: code,
    };
  }

  return {
    type: 'unknown',
    message: '预览失败，请稍后重试',
    rawCode: code,
  };
};

export const previewService = {
  async startPreview(params: StartPreviewParams): Promise<void> {
    try {
      await PreviewModule.startPreview({
        deviceId: params.deviceId,
        streamType: params.streamType,
        quality: params.quality,
        muted: params.muted,
      });
    } catch (error) {
      throw mapNativeError(error);
    }
  },

  async stopPreview(deviceId: string): Promise<void> {
    await PreviewModule.stopPreview({ deviceId });
  },

  async switchQuality(
    deviceId: string,
    quality: StartPreviewParams['quality'],
  ): Promise<void> {
    await PreviewModule.switchQuality({ deviceId, quality });
  },

  async setMuted(deviceId: string, muted: boolean): Promise<void> {
    await PreviewModule.setMuted({ deviceId, muted });
  },
};
```

面试解释：

```text
previewService 不直接处理 UI，它只负责把 Native Module 的能力变成业务侧稳定方法，并把原生错误码转成前端能理解的错误类型。这样页面不需要知道底层错误码。
```

## 5. 事件订阅和状态流转

原生预览桥接层会通过事件回传播放状态。RN 侧在 Hook 里统一订阅。

```ts
import { NativeEventEmitter, NativeModules } from 'react-native';

const previewEmitter = new NativeEventEmitter(NativeModules.PreviewModule);

type PreviewAction =
  | { type: 'START_LOADING' }
  | { type: 'PLAYING' }
  | { type: 'RECONNECTING' }
  | { type: 'ERROR'; error: PreviewError }
  | { type: 'RESET' };

interface PreviewState {
  status: PreviewStatus;
  error: PreviewError | null;
}

const initialState: PreviewState = {
  status: 'idle',
  error: null,
};

const previewReducer = (
  state: PreviewState,
  action: PreviewAction,
): PreviewState => {
  switch (action.type) {
    case 'START_LOADING':
      return { status: 'loading', error: null };
    case 'PLAYING':
      return { status: 'playing', error: null };
    case 'RECONNECTING':
      return { ...state, status: 'reconnecting' };
    case 'ERROR':
      return { status: 'error', error: action.error };
    case 'RESET':
      return initialState;
    default:
      return state;
  }
};
```

面试解释：

```text
我会把播放状态当成一个小状态机处理，而不是在页面里堆多个 useState。这样 loading、playing、error、reconnecting 的状态切换更清楚，也更容易写单测。
```

## 6. usePreview Hook

`usePreview` 负责生命周期：开始预览、订阅事件、失败重试、退出清理。

```ts
import { useCallback, useEffect, useReducer } from 'react';

export const usePreview = (params: StartPreviewParams) => {
  const [state, dispatch] = useReducer(previewReducer, initialState);

  const start = useCallback(async () => {
    dispatch({ type: 'START_LOADING' });

    try {
      await previewService.startPreview(params);
    } catch (error) {
      dispatch({ type: 'ERROR', error: error as PreviewError });
    }
  }, [params.deviceId, params.streamType, params.quality, params.muted]);

  const retry = useCallback(() => {
    return start();
  }, [start]);

  const stop = useCallback(async () => {
    await previewService.stopPreview(params.deviceId);
    dispatch({ type: 'RESET' });
  }, [params.deviceId]);

  useEffect(() => {
    const subscriptions = [
      previewEmitter.addListener('onPlayerLoading', event => {
        if (event.deviceId === params.deviceId) {
          dispatch({ type: 'START_LOADING' });
        }
      }),

      previewEmitter.addListener('onPlayerStarted', event => {
        if (event.deviceId === params.deviceId) {
          dispatch({ type: 'PLAYING' });
        }
      }),

      previewEmitter.addListener('onNetworkReconnect', event => {
        if (event.deviceId === params.deviceId) {
          dispatch({ type: 'RECONNECTING' });
        }
      }),

      previewEmitter.addListener('onPlayerError', event => {
        if (event.deviceId === params.deviceId) {
          dispatch({
            type: 'ERROR',
            error: mapNativeError(event),
          });
        }
      }),
    ];

    start();

    return () => {
      subscriptions.forEach(subscription => subscription.remove());
      previewService.stopPreview(params.deviceId).catch(() => {
        // 页面退出时的兜底清理，不在这里打断卸载流程。
      });
    };
  }, [params.deviceId, start]);

  return {
    status: state.status,
    error: state.error,
    retry,
    stop,
  };
};
```

面试解释：

```text
Hook 这一层是完整链路的核心。它统一管理启动、事件订阅、状态映射、重试和卸载清理。尤其是 return 里的 remove listener 和 stopPreview，能避免页面退出后重复监听或后台继续播放。
```

## 7. 页面如何消费

页面只消费状态，不关心 Native Module 细节。

```tsx
export const PreviewScreen = ({ route }) => {
  const { deviceId } = route.params;

  const preview = usePreview({
    deviceId,
    streamType: 'main',
    quality: 'hd',
    muted: true,
  });

  if (preview.status === 'loading') {
    return <PreviewLoading />;
  }

  if (preview.status === 'error') {
    return (
      <PreviewError
        message={preview.error?.message ?? '预览失败'}
        onRetry={preview.retry}
      />
    );
  }

  return (
    <PreviewLayout>
      {preview.status === 'reconnecting' && <ReconnectBanner />}
      <NativePreviewView deviceId={deviceId} />
      <PreviewToolbar muted />
    </PreviewLayout>
  );
};
```

面试解释：

```text
页面层只根据 status 渲染 loading、error、reconnecting 和 playing。这样 UI 和底层播放能力解耦，页面测试也更直接。
```

## 8. 多机型适配具体实现

### 8.1 适配目标

多机型适配不是简单判断机型，而是把影响 UI 稳定性的因素拆开处理：

```text
屏幕尺寸：小屏、大屏、横竖屏比例不同
安全区：刘海屏、底部 Home Indicator、Android 状态栏
系统差异：Android/iOS 阴影、层级、返回手势、底部区域差异
字体缩放：系统字体放大后，按钮和卡片容易溢出
核心页面：设备管理、列表浏览、实时预览、底部操作区优先验证
```

### 8.2 安全区和尺寸收敛

```tsx
const useAdaptiveLayout = () => {
  const insets = useSafeAreaInsets();
  const window = useWindowDimensions();

  const isSmallScreen = window.width < 360;
  const previewRatio = window.width / window.height > 0.7 ? 0.48 : 0.56;

  return {
    pagePadding: isSmallScreen ? 12 : 16,
    previewHeight: Math.round(window.width * previewRatio),
    bottomSafePadding: Math.max(insets.bottom, 12),
    headerPaddingTop: Platform.OS === 'ios' ? insets.top : 0,
  };
};

const PreviewLayout = ({ children }: PropsWithChildren) => {
  const layout = useAdaptiveLayout();

  return (
    <View style={{ flex: 1, paddingHorizontal: layout.pagePadding }}>
      <View
        style={{
          height: layout.previewHeight,
          borderRadius: 12,
          overflow: 'hidden',
        }}
      >
        {children}
      </View>
      <View style={{ paddingBottom: layout.bottomSafePadding }}>
        <PreviewToolbar />
      </View>
    </View>
  );
};
```

面试解释：

```text
我会把适配逻辑收敛到 useAdaptiveLayout，而不是每个页面散落判断。这样页面只关心布局结果，后续发现某类机型问题，也能集中调整。
```

### 8.3 平台差异和字体缩放

```ts
const platformShadow = Platform.select({
  ios: {
    shadowColor: '#000',
    shadowOpacity: 0.08,
    shadowRadius: 8,
    shadowOffset: { width: 0, height: 4 },
  },
  android: {
    elevation: 4,
  },
});

const getSafeFontSize = (size: number) => {
  const fontScale = PixelRatio.getFontScale();

  // 面试表达：不是禁止用户字体缩放，而是对关键按钮、底部操作区做上限兜底。
  if (fontScale > 1.2) {
    return Math.round(size / fontScale);
  }

  return size;
};
```

面试解释：

```text
Android 和 iOS 的阴影、层级、安全区表现不完全一样，我会用 Platform.select 做小范围隔离。字体缩放上不建议全局粗暴禁用，而是对关键操作区做兜底，避免按钮被撑爆。
```

## 9. 长列表优化具体实现

### 9.1 优化目标

可以用录像回放列表作为具体例子：

```text
场景：用户进入某个摄像头的录像回放页，按日期查看移动侦测、告警、手动录像等片段。
数据量：普通设备单日几十到几百条；如果设备安装在门口、店铺等高频触发场景，单日可能有 300-500 条录像片段。
加载策略：首屏只请求 20 条，触底后按 cursor 继续加载下一页；不把 300-500 条一次性塞进 ScrollView。
展示内容：每条包含缩略图、开始时间、时长、录像类型、播放入口，图片和文本都会带来渲染成本。
```

长列表重点解决三类问题：

```text
首屏渲染慢：一次渲染过多 item
滑动卡顿：item 重复渲染、图片过大、计算过重
加载不稳定：分页重复请求、触底频繁触发、空状态和错误态缺失
```

### 9.2 FlatList 参数和稳定引用

```tsx
const ITEM_HEIGHT = 88;
const PAGE_SIZE = 20;

const DeviceItem = React.memo(({ item, onPress }: DeviceItemProps) => {
  const handlePress = useCallback(() => {
    onPress(item);
  }, [item, onPress]);

  return (
    <Pressable onPress={handlePress} style={{ height: ITEM_HEIGHT }}>
      <DeviceCover uri={item.coverThumbUrl} />
      <DeviceInfo name={item.name} status={item.status} />
    </Pressable>
  );
});

const DeviceList = ({ devices }: { devices: Device[] }) => {
  const openDevice = useCallback((device: Device) => {
    navigation.navigate('DeviceDetail', { deviceId: device.id });
  }, []);

  const renderItem = useCallback(
    ({ item }: { item: Device }) => (
      <DeviceItem item={item} onPress={openDevice} />
    ),
    [openDevice],
  );

  return (
    <FlatList
      data={devices}
      keyExtractor={item => item.id}
      renderItem={renderItem}
      initialNumToRender={8}
      windowSize={5}
      maxToRenderPerBatch={8}
      updateCellsBatchingPeriod={50}
      removeClippedSubviews={Platform.OS === 'android'}
      getItemLayout={(_, index) => ({
        length: ITEM_HEIGHT,
        offset: ITEM_HEIGHT * index,
        index,
      })}
    />
  );
};
```

面试解释：

```text
FlatList 本身提供虚拟化，但如果 renderItem、onPress、item 组件每次都变化，还是会触发不必要渲染。所以我会稳定 key、renderItem 和事件函数，并把 item 用 React.memo 包起来。
```

### 9.3 分页、图片和性能定位

```ts
const loadMoreRecords = async () => {
  if (loadingRef.current || !nextCursorRef.current) return;

  loadingRef.current = true;

  try {
    const result = await playbackService.getRecords({
      deviceId,
      cursor: nextCursorRef.current,
      pageSize: PAGE_SIZE,
    });

    setRecords(prev => [...prev, ...result.list]);
    nextCursorRef.current = result.nextCursor;
  } finally {
    loadingRef.current = false;
  }
};
```

```text
图片处理：列表中优先展示缩略图，不直接加载原图；能复用缓存就复用缓存。
定位工具：先用 RN Performance Monitor 看 JS/UI FPS，再用 Flipper 或 React DevTools Profiler 看重复渲染；如果怀疑原生侧图片解码、内存或掉帧，再看 Android Studio Profiler / Xcode Instruments。
```

面试解释：

```text
我会先从 RN 层排查是否重复渲染、是否一次加载过多、是否图片过大；只有怀疑原生层解码、内存或主线程阻塞时，才进一步看 Android Studio Profiler 或 Xcode Instruments。
```

## 11. 面试回答模板

如果面试官问“实时预览怎么实现”，可以这样答：

```text
我负责的是 RN 侧实时预览链路，不是底层播放器内核。整体链路是页面进入后拿到 deviceId、码流类型、清晰度等参数，然后调用 RN 侧封装的 previewService。previewService 再调用项目里的原生预览桥接层，桥接层负责和 Android/iOS 播放能力或设备 SDK 对接。

播放过程中，原生桥接层通过事件回传 loading、started、error、reconnect 等状态。RN 侧在 usePreview 里统一订阅这些事件，并映射成 loading、playing、error、reconnecting 这些页面状态。页面只根据状态渲染加载态、播放态、错误兜底和重连提示。

退出页面时，会 stopPreview 并移除 event listener，避免重复订阅和后台继续播放。测试上，Jest 测状态流转，RNTL 测 UI 兜底，Detox 只覆盖进入预览和异常兜底这类核心路径。
```
