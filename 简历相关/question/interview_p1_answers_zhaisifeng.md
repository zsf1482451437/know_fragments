## 一、Reolink Web 跨境商城

## Q7：组件抽象的边界怎么判断？

### 推荐回答

我判断组件是否要抽象，不只看代码是否重复，还会看复用频率、业务稳定性、配置复杂度和维护成本。

在 Reolink Web 商城里，商品卡片、卖点模块、媒体展示、规格参数、FAQ 这些模块会在商品页、活动页、解决方案页里反复出现，而且结构相对稳定，所以适合抽成业务组件。组件通过 `React + TypeScript` 定义 props，比如标题、图片、跳转链接、标签、按钮文案等变化点，样式上用 TailwindCSS 或 CSS Module 保持一致。

但如果某个模块只是一次性活动视觉，或者运营样式变化很频繁，我不会一开始就抽成复杂配置组件。否则配置项会膨胀，后期维护成本比复制局部代码还高。

一句话总结：稳定、高频、边界清晰的抽象；变化大、强定制的保留页面内实现。

### 追问逐题回答

#### 为什么商品卡片、卖点模块、规格参数、FAQ 适合抽象？

因为这些模块跨商品页、活动页、解决方案页都会重复出现，业务结构比较稳定。商品卡片通常都有图片、标题、价格/标签、跳转；FAQ 通常是问答结构；规格参数通常是表格或分组列表。这类模块抽象后能统一样式和交互，减少重复开发。

#### 一次性活动模块为什么不一定适合抽象？

一次性活动模块往往视觉强定制、生命周期短、变更频繁。如果强行抽象，需要加很多配置项来兼容不同视觉，最后组件会变成难维护的“大而全配置器”。这种场景更适合页面内定制，等出现稳定复用形态后再抽象。

#### 组件 props 怎么设计，怎么避免配置项膨胀？

props 只暴露高频变化点，比如文案、图片、链接、按钮、标签；布局结构和基础样式尽量收在组件内部。如果一个组件开始出现大量布尔开关和嵌套配置，说明抽象过度，需要拆成更小组件，或者把强定制逻辑放回页面。

#### TypeScript 在组件抽象里起什么作用？

TypeScript 用来约束组件入参，减少调用错误。比如商品卡片的 `title`、`image`、`href` 是必填，`tags`、`price` 可以是可选。这样页面接入组件时，字段缺失或类型错误能提前暴露。

#### TailwindCSS / CSS Module 怎么支撑组件复用？

TailwindCSS 可以统一间距、字体、断点和状态样式，适合快速组合；CSS Module 更适合封装复杂组件样式，避免样式污染。无论哪种方式，核心是把响应式规则和视觉层级沉淀下来，不让每个页面重复写一套。

## Q8：活动页上线前检查清单有哪些？

### 推荐回答

我会按 8 类检查：内容、交互、接口、性能、SEO、兼容、埋点、回滚。

内容看商品信息、价格、活动时间、优惠文案、图片和跳转链接是否正确；交互看 CTA、弹窗、轮播、表单、购买跳转是否可用；接口看价格、库存、活动状态、异常兜底；性能看 LCP、图片体积、首屏 JS 和非首屏懒加载；SEO 看 title、description、canonical、h1、alt 和核心内容可索引；兼容看桌面、平板、移动端以及 Chrome/Safari 差异；埋点看曝光、点击、购买入口是否正常；回滚看上线异常时是否能快速下线或降级。

这个清单的价值是减少活动页协作中的遗漏，降低上线后返工。

### 追问逐题回答

#### 内容、交互、接口、性能、SEO、兼容、埋点、回滚分别检查什么？

内容检查商品、价格、文案、图片、链接；交互检查按钮、弹窗、表单、轮播、购买路径；接口检查价格、库存、活动状态和异常兜底；性能检查首屏资源、LCP、图片体积；SEO 检查 Meta、语义化和可索引内容；兼容检查多视口和浏览器；埋点检查曝光和点击；回滚检查异常时能否快速下线或恢复旧版本。

#### 商品价格、活动时间、库存状态怎么校验？

价格和库存要和接口返回、运营配置或后台数据对齐，活动时间要检查时区、开始结束状态和过期文案。上线前用真实或准生产数据走一遍购买链路，接口异常时也要验证页面兜底，不让页面空白。

#### 接口失败时页面怎么兜底？

要区分核心接口和非核心接口。核心接口失败时给错误提示、重试入口或降级展示；非核心模块失败可以隐藏模块或展示默认内容。至少要避免白屏、按钮不可解释地失效、价格/库存展示错误。

#### Safari 和 Chrome 常见兼容差异有哪些？

常见差异包括 CSS 支持、视频自动播放策略、日期解析、position sticky、输入框样式、滚动行为等。活动页重点看移动端 Safari，因为跨境商城移动端访问占比通常不低。

#### 活动上线后发现价格或跳转错误怎么处理？

先判断影响范围，能配置修复的优先改配置；不能配置修复的快速回滚或下线入口。价格和购买跳转属于高风险问题，要同步运营和后端确认数据源，同时补充复盘，避免下次检查清单遗漏。

## Q9：新活动页开发效率提升是怎么来的？

### 推荐回答

效率提升主要来自三点：组件复用、交付清单、问题沉淀。

组件复用让新活动页不需要从零写商品卡片、卖点模块、规格参数、FAQ 等基础模块；交付清单统一 SEO、埋点、响应式、跳转、活动时间、接口兜底等验收标准；常见联调问题手册沉淀价格、库存、活动状态、跳转链接、接口异常这些高频问题。

所以效率提升不是因为“写得熟”，而是减少重复开发、减少遗漏、减少联调返工。

### 追问逐题回答

#### 是编码更快，还是返工更少？

两者都有，但更核心是返工更少。组件复用减少从零开发，检查清单减少上线遗漏，联调手册减少重复排查。面试里我会强调这是流程和组件沉淀带来的效率，而不是单纯个人手速。

#### 8+ 业务组件如何减少从零开发？

新活动页可以通过组合商品卡片、卖点模块、媒体展示、规格参数、FAQ 等组件完成主体结构，只需要替换文案、图片、跳转和部分样式配置，不必每次重新写页面结构和响应式细节。

#### 检查清单如何减少遗漏？

清单把 SEO、埋点、接口兜底、响应式、活动时间、购买跳转这些容易遗漏的点前置到开发和验收阶段。设计、运营、后端和前端可以按同一套标准检查，减少最后上线前才发现问题。

#### 常见联调问题处理手册沉淀了哪些问题？

主要是商品价格、库存状态、活动时间、跳转链接、接口异常、移动端样式、埋点缺失这些问题。后续遇到类似问题时，可以直接按排查路径定位，而不是每次从头问人。

#### 如果面试官要求量化，你能提供什么保守口径？

我会说简历里没有写具体百分比，所以不临场编数字。可以给保守口径：通过 8+ 组件、1 套检查清单和常见联调手册，减少重复开发和验收遗漏。真正量化要看团队任务记录或活动页交付周期统计。

## 二、Reolink React Native 移动端项目

## Q14：Native Modules 用在什么场景？

### 推荐回答

Native Modules 主要用于 JS 层无法直接完成，或者 JS 做不合适的能力，比如原生 SDK、系统权限、推送、音视频、文件缓存、设备能力等。

在智能安防 App 里，设备接入、推送、音视频预览、权限申请这类场景经常需要和原生能力协作。我的边界是：普通业务逻辑尽量留在 JS 层，只有性能、系统能力或已有原生 SDK 必须介入时，才通过 Native Modules 下沉。

JS 侧会封装统一调用层，用 TypeScript 定义入参和返回值；原生侧通过事件回传连接状态、播放状态、权限结果；失败场景要有错误码、超时、重试或降级 UI。

### 追问逐题回答

#### 哪些能力适合下沉到原生？

系统权限、推送、音视频 SDK、文件缓存、设备能力、蓝牙/网络状态等适合下沉。普通业务判断、页面展示、表单逻辑不适合下沉，否则会增加双端维护成本。

#### JS 与原生怎么通信？

React Native 里 JS 可以通过 NativeModules 调用原生暴露的方法，也可以通过事件机制接收原生回调。JS 侧最好封装一层 service，不让页面直接散落调用 NativeModules。

#### 原生事件怎么回传到 JS？

原生侧通过 event emitter 发送事件，JS 侧注册监听，比如设备连接状态变化、播放状态变化、权限结果。JS 收到事件后再更新状态或触发 UI 变化。

#### TypeScript 怎么约束 Native Module 调用？

在 JS 调用层定义接口类型，比如方法入参、返回 Promise 类型、错误码结构。这样页面调用时能提前发现参数错误，减少运行时和原生联调的低级问题。

#### 原生调用失败怎么兜底？

要有错误码、超时、重试和降级 UI。比如权限失败提示用户授权，设备连接失败提供重试，音视频初始化失败展示错误状态，不能让页面一直 loading。

#### 哪些普通业务逻辑不应该下沉到原生？

商品展示、列表筛选、页面状态、普通表单、业务规则判断一般不该下沉。这些放在 JS 层更易迭代和测试，下沉到原生会增加 Android/iOS 双端维护成本。

## Q15：Detox 和 Jest 怎么分工？

### 推荐回答

我按测试金字塔分工。Jest 测纯函数、reducer、Hook 和数据转换；React Native Testing Library 测组件渲染和交互；Detox 测端到端核心链路。

不是所有用例都写 Detox，因为 E2E 成本高、运行慢、环境依赖强。我的做法是用 Jest/RNTL 覆盖大多数逻辑和组件，Detox 只覆盖设备接入、消息提醒、录像回放这类关键路径。

### 追问逐题回答

#### Jest、React Native Testing Library、Detox 分别适合测什么？

Jest 适合测 reducer、Hook、工具函数、数据转换；RNTL 适合测组件展示、按钮点击、状态变化；Detox 适合测跨页面端到端流程，比如从进入页面到完成设备接入或查看录像。

#### 为什么不把所有场景都写成 Detox？

Detox 执行慢、环境依赖强、维护成本高。如果所有细节都用 E2E 测，反馈链路会很长，也容易因为环境波动失败。Detox 应该只覆盖高价值核心链路。

#### E2E 用例不稳定时怎么处理？

先减少对真实网络、真实设备状态和时间等待的依赖。能 mock 的接口和权限尽量 mock；等待页面状态时用明确元素和状态判断，少用固定 sleep；失败时保留截图、日志和重试机制。

#### 接口、权限、设备状态怎么 mock？

接口可以用 mock server 或请求拦截；权限可以封装权限服务，在测试环境返回固定状态；设备状态可以用固定 mock 数据模拟在线、离线、连接失败等分支。

#### 哪些核心链路必须做端到端测试？

设备接入、消息提醒、录像回放这些影响用户核心体验、且容易跨页面回归的链路适合做 Detox。普通静态展示和低风险 UI 不一定需要 E2E。

## Q16：多机型适配怎么做？

### 推荐回答

我会从屏幕尺寸、安全区、系统版本、字体缩放、平台差异几个维度处理。

React Native 里不能按一个设计稿写死尺寸。安全区用 `react-native-safe-area-context`；平台差异用 `Platform.select`；响应式布局根据屏幕宽度、比例和设计系统处理；重点验证设备管理、列表浏览、预览操作、底部操作区这些高频场景。

### 追问逐题回答

#### 安全区、刘海屏、底部操作区怎么处理？

用 safe area 相关能力处理顶部刘海和底部 Home Indicator 区域，避免按钮贴边或被遮挡。底部操作区要考虑 iOS 和 Android 导航栏差异，不能只按固定高度写死。

#### Android 和 iOS 默认行为有哪些差异？

常见差异包括返回行为、权限弹窗、输入框、状态栏、阴影、字体渲染、滚动惯性、键盘遮挡等。跨端项目里要明确哪些差异用统一组件封装，哪些用平台差异处理。

#### Platform.select 用在什么场景？

用于少量平台差异，比如状态栏高度、阴影样式、权限提示、某些原生能力调用参数。不能把大量业务逻辑都写进 Platform.select，否则代码会变得难维护。

#### 字体缩放和不同分辨率怎么适配？

字体要考虑系统字体缩放和最小可读性，布局要避免固定宽高导致小屏溢出。图片和视频区域按比例或容器约束处理，关键按钮保证可点击区域足够。

#### 20+ 主流机型怎么选？

按用户设备分布和常见屏幕类型选，包括小屏、大屏、高分辨率、不同 Android/iOS 版本、刘海屏和非刘海屏。面试时我会把它说成兼容验证范围，不说所有机型都覆盖。

## Q17：多端兼容和重点列表场景优化是怎么验证的？

### 推荐回答

这类优化我会按“兼容验证”和“性能验证”两条线讲，不把它说成全量线上绝对指标。

多端兼容主要围绕 20+ 主流机型、不同屏幕尺寸、Android/iOS 系统版本、安全区、底部操作区、字体缩放等场景做验证，重点看设备管理、列表浏览、实时预览这些高频页面是否存在遮挡、错位、点击区域异常或平台差异问题。

重点列表场景优化主要通过真机验证和调试工具观察卡顿改善，不说成全量线上帧率指标。技术上会用 RN Performance Monitor、Flipper、React DevTools Profiler，必要时结合 Android Studio Profiler 或 Xcode Instruments，看 JS/UI 线程是否阻塞，再针对 FlatList 参数、item memo、图片加载和状态更新粒度做优化。

### 追问逐题回答

#### 多端兼容问题主要覆盖哪些场景？

主要覆盖安全区、刘海屏、底部操作区、不同屏幕尺寸、字体缩放、Android/iOS 默认行为差异，以及设备管理、列表浏览、实时预览这些高频页面。验证重点是页面不遮挡、布局不溢出、按钮可点击、异常提示可见。

#### 重点列表场景真机验证怎么做？

会选择数据量较大、滑动频繁的列表场景，在典型 Android/iOS 机型上对比优化前后的滑动表现。配合 RN Performance Monitor、Flipper 或 Profiler 看是否存在 JS 线程阻塞、item 重复渲染、图片加载过重等问题。

#### 如果没有线上性能平台，你怎么回答？

如果没有线上性能平台，就明确说这是本地真机和调试工具验证，不包装成线上全量数据。重点讲验证页面、验证机型、使用工具和具体优化动作。

#### JS 线程阻塞和 UI 线程掉帧怎么区分？

JS 线程阻塞通常和复杂计算、频繁 setState、大量 JS 执行有关；UI 线程掉帧可能和布局、图片、原生渲染、动画有关。可以通过 RN 性能面板、Flipper 和原生 profiler 看线程表现。

#### 这类优化的边界怎么讲才不夸大？

用“重点列表场景”“20+ 主流机型验证范围”“真机验证”“参与优化动作”这些边界词，不说所有页面、所有机型、所有用户都达到同一表现。

## Q18：20+ 测试用例主要覆盖哪些场景？

### 推荐回答

主要覆盖设备接入、消息提醒、录像回放 3 类核心链路。

设备接入覆盖成功、失败、超时、重试、权限异常；消息提醒覆盖未读、已读、空列表、点击跳转、异常提示；录像回放覆盖时间筛选、列表加载、播放入口、无录像兜底。Jest 测状态流转和工具函数，RNTL 测组件交互，Detox 测端到端关键路径。

### 追问逐题回答

#### 设备接入链路有哪些测试点？

连接成功、连接失败、超时、权限缺失、loading、重试、错误提示。重点是状态流转和异常兜底。

#### 消息提醒链路有哪些测试点？

消息列表展示、未读状态、点击跳转、空列表、接口失败、已读状态更新。重点是状态一致性和用户入口可用。

#### 录像回放链路有哪些测试点？

时间选择、录像列表加载、播放入口、无录像兜底、加载失败提示、返回路径。重点是列表和播放入口状态。

#### 状态流转怎么测？

复杂状态用 reducer 或 Hook 抽出来，用 Jest 输入事件，断言状态变化。比如设备连接从 idle 到 loading，再到 success 或 error。

#### 端到端链路怎么避免过度测试？

只选高价值路径，不把所有 UI 细节都放进 Detox。比如设备接入成功、消息查看、录像回放入口这些必须测；普通文案和静态展示交给组件测试或人工验收。

## 三、GogoalShop 跨境商城

## Q22：Tree Shaking 为什么有时不生效？

### 推荐回答

Tree Shaking 不生效通常有几个原因：依赖是 CommonJS 而不是 ESM；代码存在副作用，打包器不敢删；导入方式是整包导入；构建配置把 ESM 转成了 CommonJS。

所以排查时要看包格式、导入方式、sideEffects 配置和构建产物，而不是只说“用了 Tree Shaking”。优化方式包括优先使用 ESM 包、按需导入、替换不支持摇树的大包，并用 bundle analyzer 验证。

### 追问逐题回答

#### ESM 和 CommonJS 对 Tree Shaking 有什么影响？

ESM 是静态模块结构，打包器更容易判断哪些导出没被使用；CommonJS 是运行时 require，静态分析困难，Tree Shaking 效果通常更差。

#### sideEffects 配置是什么？

`sideEffects` 用来告诉打包器某些文件是否有副作用。如果包声明没有副作用，未使用代码更容易被删除；如果配置不准确，可能误删或不敢删。

#### 整包导入为什么会影响体积？

整包导入可能把不需要的模块也打进 bundle，尤其是 CommonJS 或不支持按需的库。比如只用几个工具函数，却 import 整个工具库，会造成体积浪费。

#### Babel 或构建配置为什么可能破坏 Tree Shaking？

如果 Babel 提前把 ESM 转成 CommonJS，打包器就失去静态分析优势，Tree Shaking 效果会下降。所以构建链路里要注意模块格式保留。

#### 怎么通过 bundle analyzer 验证？

看优化前后目标依赖在 bundle 里的体积是否下降，是否仍然整包打入，是否存在重复版本。不能只看配置，要看构建产物。

## Q23：pnpm 为什么更快？

### 推荐回答

pnpm 快主要是因为内容寻址存储和硬链接。相同版本的依赖不会在每个项目重复完整安装，而是存到全局 store，再通过硬链接引用。这样安装速度和磁盘占用都更好。

另外 pnpm 的依赖结构更严格，能减少幽灵依赖问题。`pnpm-lock.yaml` 也能保证团队和 CI 环境依赖版本一致。

### 追问逐题回答

#### 内容寻址存储是什么？

依赖包按内容 hash 存储在全局 store，相同内容只保存一份。不同项目使用同一版本依赖时，不需要重复下载和复制。

#### 硬链接如何减少重复安装？

项目 node_modules 里的文件通过硬链接指向全局 store，避免每个项目都复制一份完整依赖。这样磁盘占用和安装时间都会下降。

#### pnpm 为什么能减少幽灵依赖？

pnpm 的 node_modules 结构更严格，项目只能访问自己 package.json 声明的依赖，不容易误用间接依赖。npm/yarn 传统扁平结构更容易出现“没声明但能用”的幽灵依赖。

#### pnpm-lock.yaml 对团队协作有什么价值？

锁定依赖版本和解析结果，保证不同开发机和 CI 安装出一致依赖，减少“我本地能跑、别人不能跑”的问题。

#### npm 到 pnpm 迁移有哪些风险？

风险包括幽灵依赖暴露、脚本兼容问题、lockfile 变化、CI 配置调整。迁移时要先跑 install、lint、build、test，逐个补齐未声明依赖。

## Q24：多语言工具怎么保证数据准确？

### 推荐回答

多语言工具最容易出问题的是 key 对不上、覆盖已有翻译、写入格式错误。所以我会把流程拆成抓取、校验、写入、回滚。

Chrome 插件负责抓取翻译数据，Node 服务负责结构化写入。写入前检查 key 是否存在、语言字段是否完整、是否覆盖人工翻译；写入时保持 JSON 结构稳定；失败时保留日志和备份，方便回滚。

### 追问逐题回答

#### Chrome 插件抓取了什么数据？

主要抓取翻译后的文案、对应语言、页面或 namespace、key 信息。具体结构要和项目多语言文件结构对齐。

#### Node 自动写入服务怎么避免覆盖人工翻译？

写入前检查目标 key 是否已有人工翻译，如果已有内容，需要跳过、提示冲突或要求确认。不能静默覆盖。

#### key 缺失、重复、空值怎么校验？

写入前做 schema 校验，检查 key 是否存在、是否重复、value 是否为空、语言字段是否完整。失败时输出错误日志，不继续写入。

#### 写入失败怎么回滚？

写入前备份目标文件，或者通过 Git diff 保留可回退状态。失败时输出错误位置和原因，恢复备份或撤销本次写入。

#### 多语言页面如何验证展示正确？

抽样切换语言查看页面，检查 fallback 文案、特殊字符、换行、长度溢出和关键按钮文案。自动写入后不能只看文件成功。

## 四、Shopify 自研应用

## Q26：Shopify 应用上架有哪些坑？

### 推荐回答

Shopify 应用上架不只是功能开发，还涉及 OAuth、权限、回调地址、Webhook、隐私合规、测试账号和审核材料。

常见坑是权限申请过大、生产回调 URL 配错、卸载后数据处理不符合要求、隐私协议不完整、Webhook 不稳定。我的做法是权限最小化，生产域名和 HTTPS 配置正确，安装/卸载流程稳定，并准备应用说明、隐私协议和测试账号。

### 追问逐题回答

#### OAuth 安装流程怎么处理？

用户安装应用时跳转 Shopify 授权页，授权后回调到应用服务端，服务端用 code 换 access token，并保存 shop、token、安装状态。回调地址必须和 Shopify 后台配置一致。

#### Admin API scope 怎么控制最小权限？

只申请业务需要的权限，比如商品配置、订单读取/标记相关权限。权限过大会增加审核风险，也不利于用户信任。

#### Webhook 需要处理哪些事件？

常见是 app uninstall、订单相关事件、数据删除相关事件。卸载时要清理安装状态或 token，订单事件用于拦截或标记风险订单。

#### 隐私协议、数据删除、测试账号为什么重要？

Shopify 审核会关注数据如何使用、如何删除、应用如何测试。隐私协议和测试账号不完整，会影响审核通过。

#### 回调 URL 和生产域名不一致会出什么问题？

OAuth 可能失败，应用安装后无法回调，Webhook 也可能无法触达。生产域名、HTTPS、环境变量和 Shopify 后台配置必须一致。

## Q27：GraphQL 查询怎么设计？

### 推荐回答

GraphQL 查询原则是按页面需要取字段，不一次性取完整对象。配置页需要什么字段就取什么字段，列表类数据用 cursor pagination，重复字段用 fragment，错误处理要区分 GraphQL errors、userErrors 和网络异常。

这样能减少接口体积，也让前端数据结构更稳定。

### 追问逐题回答

#### 为什么不一次性取完整商品数据？

完整商品数据体积大、字段多，很多字段页面用不上，会增加接口耗时和前端处理成本。GraphQL 的优势就是精确取数。

#### variables、cursor pagination、fragment 分别解决什么问题？

variables 用来传商品 ID、筛选条件等动态参数；cursor pagination 解决大列表分页；fragment 复用重复字段结构，减少查询重复。

#### GraphQL errors 和 userErrors 怎么处理？

GraphQL errors 通常是查询或服务异常；userErrors 是业务层错误，比如参数不合法、权限不足。前端要分别提示和记录，不能只判断 HTTP 状态码。

#### 查询字段变化如何避免影响前端页面？

通过类型定义、数据适配层和默认值兜底。接口字段变化不要直接散落影响组件，最好在 service 层做转换。

#### GraphQL 相比 REST 的优势和成本是什么？

优势是按需取字段、减少多接口拼接、类型结构清晰；成本是查询设计、缓存、错误处理和学习成本更高。

## Q28：黑名单拦截失败怎么兜底？

### 推荐回答

黑名单拦截不能只依赖单点实时判断。兜底要从规则校验、日志记录、订单标记、人工复核几个方向做。

如果实时拦截失败，至少要在后续订单流程里标记风险订单，比如加 tag、记录命中规则、通知运营复核。这样即使没有第一时间阻断，也不会完全漏掉风险订单。

### 追问逐题回答

#### 黑名单规则包含哪些字段？

可以包含邮箱、手机号、地址、用户标识、IP、订单特征等。具体字段要根据业务风险和 Shopify 可获取数据决定。

#### 实时拦截失败后怎么标记风险订单？

通过 Shopify Admin API 给订单加 tag、备注或风险标识，并记录命中规则和错误日志，方便运营后续复核。

#### Shopify Admin API 如何给订单打 tag 或备注？

通过 Admin API mutation 更新订单 tags 或 note，具体字段和权限要看 Shopify API 版本。前端不直接操作，通常通过服务端接口执行。

#### 高风险订单如何进入人工复核？

命中规则或服务异常时，将订单标记为风险订单，通知运营或客服检查。人工复核比完全静默失败更安全。

#### 规则服务异常时应该放行还是保守阻断？

要看业务策略。一般不能随意阻断所有订单，否则影响转化；可以对高风险命中走保守策略，对不确定情况标记复核。这个需要和业务方共同确定。

## Q29：Docker 解决了什么问题？

### 推荐回答

Docker 主要解决环境一致性和部署可复现问题。

Shopify 应用脚手架和服务器环境可能有 Node 版本、系统依赖、包管理工具差异，直接部署容易出现本地能跑、服务器不能跑。Docker 把 Node 版本、依赖安装、构建命令、启动方式固化到镜像里，服务器只要运行镜像即可。

### 追问逐题回答

#### Node 版本、系统依赖、包管理工具差异分别会造成什么问题？

Node 版本不同可能导致构建失败或运行 API 不兼容；系统依赖缺失可能导致安装或编译失败；包管理工具差异可能导致 lockfile 和依赖解析不一致。

#### Dockerfile 里通常要固定哪些内容？

固定基础镜像、Node 版本、包管理工具、依赖安装命令、构建命令、启动命令、端口暴露。必要时用多阶段构建减少镜像体积。

#### 环境变量如何注入，为什么不能写死？

通过 `.env`、服务器环境变量或 CI/CD 配置注入 Shopify key、secret、database url。不能写死在代码里，因为有安全风险，也不方便区分环境。

#### 镜像版本如何支持回滚？

每次发布打 tag，比如按版本号或时间戳。出现问题时重新运行旧 tag 镜像，就能回滚到旧版本。

#### Docker 和普通 PM2 / Node 部署相比取舍是什么？

Docker 环境一致性更好、回滚更清晰，但构建和镜像管理有额外成本。PM2 部署更轻，但更依赖宿主机环境。对环境兼容问题明显的项目，Docker 更稳。

## Q30：Prisma 用来存什么？

### 推荐回答

Prisma 主要存店铺安装信息、access token、商品定制配置、黑名单规则、订单拦截记录和异常日志。它的价值是通过 schema 管理数据模型，并生成类型，减少字段写错和结构不一致。

### 追问逐题回答

#### 店铺信息、access token、商品配置、黑名单规则分别怎么建模？

店铺表存 shop domain、token、安装状态；商品配置表存商品 ID、配置项、开关状态；黑名单表存规则类型、匹配值、启用状态；拦截记录表存订单 ID、命中规则、处理结果。

#### Prisma schema 的类型安全有什么价值？

schema 生成 TypeScript 类型，查询和写入时字段错误能提前暴露。服务端处理配置和规则时更稳定。

#### migration 怎么管理表结构变化？

通过 Prisma migration 记录 schema 变化，开发、测试、生产按迁移执行，避免手动改表导致环境不一致。

#### access token 这类敏感信息怎么保护？

不能打印到日志，不能暴露到前端。存储时要控制数据库权限，必要时加密或使用安全存储策略，服务端使用环境变量保护密钥。

#### Prisma 和直接写 SQL 的取舍是什么？

Prisma 类型安全和开发效率更好，适合业务配置类应用；复杂 SQL 和极限性能场景直接 SQL 更灵活。Shopify 应用这类配置型场景用 Prisma 比较合适。

## Q31：部署成功率 100% 和发布效率 70% 怎么统计？

### 推荐回答

这两个数据要讲边界。

部署成功率 100% 指我参与发布周期内，Docker 化后环境类部署失败被消除，不是所有历史发布永远不会失败。发布效率 70% 是按人工步骤和单次发布耗时粗略估算，比如从手动安装依赖、构建、命名、压缩、上传等 5 步，脚本化成 1 步。

### 追问逐题回答

#### 100% 是所有历史发布，还是你参与周期内的发布？

是我参与周期内的发布口径，主要指环境类部署失败被消除。不能说所有历史发布或所有异常都 100% 成功。

#### 哪些失败属于环境类部署失败？

Node 版本不一致、系统依赖缺失、包管理工具差异、服务器缺少构建依赖，这些属于环境类问题。

#### 70% 是按步骤数、耗时还是人工操作成本估算？

主要按操作步骤和单次发布耗时估算。原来多步手工操作，脚本化后变成一条命令，人工操作成本明显下降。

#### Shell 脚本自动化了哪些步骤？

自动化镜像打包、版本命名、文件压缩、产物校验、上传或部署触发等步骤。脚本里要保留错误退出和日志。

#### 脚本失败时如何输出日志和中断流程？

Shell 脚本使用明确的错误处理，比如 `set -e`、关键步骤输出日志、失败时停止后续操作。不能失败后继续发布。

## 五、工程化与团队协作

## Q33：你怎么做代码评审？

### 推荐回答

我做 code review 不会只看格式，格式交给 ESLint/Prettier。人工 review 更关注业务逻辑、边界条件、异常兜底、性能影响、类型设计和可维护性。

比如前端页面会看接口失败有没有兜底、loading/empty/error 状态是否完整；React/RN 会看是否有无意义重渲染、列表性能问题；跨端会看 Android/iOS 差异；工程上会看是否引入大依赖、是否破坏组件边界。

### 追问逐题回答

#### 你会优先看业务逻辑、边界条件还是代码风格？

优先看业务逻辑和边界条件。代码风格尽量交给工具。人工 review 的价值在发现业务理解、异常兜底、数据流和可维护性问题。

#### 如何发现性能问题和重复渲染问题？

看状态是否过大、props 是否稳定、列表项是否 memo、是否有 inline 对象/函数导致子组件重复渲染，是否引入大依赖或在渲染中做复杂计算。

#### 对跨端兼容问题如何 review？

看是否处理安全区、平台差异、权限状态、不同分辨率和系统版本；涉及 Native Modules 时看错误码、超时和事件回传是否完整。

#### review 意见和业务进度冲突时怎么取舍？

影响线上稳定性、数据正确性、安全和核心链路的问题必须处理；纯风格和低风险优化可以后置。取舍标准是风险和影响范围。

#### 如何避免 review 变成纯格式检查？

把格式检查自动化，PR 中重点描述业务背景、改动范围、测试结果和风险点。review 时围绕这些信息看问题。

## Q34：CI/CD 在你的项目里主要做什么？

### 推荐回答

CI/CD 的价值是把重复验证和发布流程标准化。前端项目里常见环节包括 lint、test、build、产物检查、部署触发。

我不会把自己包装成完整 DevOps 架构师，更多是参与和推动前端侧质量门禁，比如 ESLint、测试用例、构建检查、发布脚本和上线检查清单。

### 追问逐题回答

#### 你是否真正搭建过完整 CI/CD，还是参与使用？

我更准确的表述是参与和推动前端侧 CI/CD 质量流程，不说自己完整主导大型 CI/CD 平台。具体包括 lint、test、build、脚本发布和检查清单。

#### lint、test、build、deploy 哪些环节接入了自动化？

lint 和 build 是最基础的自动化环节；核心测试用例可以接入迭代验证；deploy 侧通过 Shell 脚本或 CI 配置减少手工步骤。不同项目接入程度不同。

#### 前端项目发布前最关键的质量门禁是什么？

lint、type check、build、核心测试、关键页面手动验收。活动页还要检查 SEO、埋点、跳转、响应式和回滚。

#### CI 失败如何定位？

先看失败阶段，是安装、lint、test、build 还是部署。再看日志、依赖变化、环境变量、Node 版本和最近改动。不要直接重跑掩盖问题。

#### 自动化不足时如何用检查清单补位？

用发布检查清单覆盖自动化没覆盖的风险，比如活动时间、价格、跳转、埋点、SEO、移动端适配。清单不是替代自动化，但可以减少遗漏。

## 六、前端基础

## Q37：浏览器缓存策略你怎么理解？

### 推荐回答

浏览器缓存主要分强缓存和协商缓存。强缓存命中时不请求服务器，常见字段是 Cache-Control；协商缓存会请求服务器确认资源是否变化，常见字段是 ETag、Last-Modified。

前端静态资源通常用 hash 文件名配合长缓存，内容变了文件名也变；HTML 文件一般不做太长强缓存，避免用户拿到旧入口文件。

### 追问逐题回答

#### 强缓存和协商缓存区别是什么？

强缓存命中时浏览器直接用本地缓存，不发请求；协商缓存会向服务器确认资源是否变化，没变化返回 304。

#### Cache-Control、ETag、Last-Modified 怎么用？

Cache-Control 控制强缓存策略；ETag 是资源内容标识；Last-Modified 是最后修改时间。ETag 精度更高，Last-Modified 实现简单。

#### 静态资源 hash 文件名和缓存策略怎么配合？

JS/CSS/图片这类带 hash 的资源可以设置长缓存，因为内容变化会生成新文件名。这样能提升二次访问速度。

#### HTML 文件和 JS/CSS 资源缓存策略为什么不同？

HTML 是入口文件，如果强缓存太久，用户可能拿不到最新资源引用。JS/CSS 带 hash，可以长缓存。HTML 通常需要更保守缓存策略。

#### 缓存导致线上问题怎么排查？

检查 Network 里资源是否 from disk cache、是否 304、响应头是否符合预期，确认 HTML 是否引用旧资源。必要时调整缓存头或刷新 CDN。

## Q38：跨域问题怎么解决？

### 推荐回答

跨域本质是浏览器同源策略限制，不是服务端不能访问。常见解决方式是服务端配置 CORS，开发环境可以用代理转发，特殊场景也可能用 JSONP 或 iframe，但现代项目主要是 CORS 和代理。

### 追问逐题回答

#### CORS 的核心响应头有哪些？

主要是 `Access-Control-Allow-Origin`、`Access-Control-Allow-Methods`、`Access-Control-Allow-Headers`，带 cookie 时还需要 `Access-Control-Allow-Credentials`。

#### 简单请求和预检请求区别是什么？

简单请求满足特定方法和 header 条件，会直接发请求；复杂请求会先发 OPTIONS 预检，确认服务端允许后再发正式请求。

#### credentials 场景要注意什么？

如果携带 cookie，前端要设置 credentials，服务端要允许 credentials，并且 `Access-Control-Allow-Origin` 不能是 `*`，必须是明确域名。

#### 代理转发为什么能解决开发环境跨域？

因为浏览器请求同源开发服务器，开发服务器再转发到目标接口。跨域限制发生在浏览器层，服务端转发不受同源策略限制。

#### 跨域和前端安全有什么关系？

CORS 配置过宽可能导致敏感接口被不可信站点调用。跨域策略要按域名和接口权限收敛，不能随意 `*`。

## Q39：前端安全你关注哪些点？

### 推荐回答

我主要关注 XSS、CSRF、敏感信息泄露、第三方脚本风险和用户输入处理。

XSS 要避免不可信 HTML 直接渲染，对用户输入做转义或白名单过滤；CSRF 可以通过 SameSite Cookie、CSRF Token、Referer/Origin 校验防护；token 存储要结合安全和业务取舍，避免暴露敏感信息。

### 追问逐题回答

#### XSS 怎么产生，如何防御？

XSS 是不可信脚本被注入并执行。防御方式包括输出转义、避免直接使用 dangerouslySetInnerHTML、富文本白名单过滤、CSP、对用户输入做校验。

#### CSRF 怎么产生，如何防御？

CSRF 是用户登录态被恶意站点利用发起请求。防御方式包括 SameSite Cookie、CSRF Token、校验 Origin/Referer、关键操作二次确认。

#### token 存 localStorage 和 cookie 的取舍是什么？

localStorage 易被 XSS 读取；cookie 可用 HttpOnly 降低 JS 读取风险，但要处理 CSRF。没有绝对最优，要结合业务和安全策略。

#### 第三方脚本和 Shopify 应用有什么安全风险？

第三方脚本可能窃取信息、影响性能或污染全局环境。Shopify 应用要注意权限最小化、token 保护、Webhook 验签、敏感数据不暴露到前端。

#### 用户输入和富文本内容怎么处理？

用户输入要校验和转义；富文本要用白名单 sanitizer，限制标签和属性，禁止 script、危险 URL 和事件属性。

## 七、AI skills / AI Agent 辅助研发

## Q41：AI 生成代码怎么保证质量？

### 推荐回答

AI 生成代码不能直接合入。我的流程是先提供足够上下文，比如需求、目录结构、组件规范、接口类型、异常分支；生成后做 diff review，检查是否符合现有风格和边界；再跑 lint、test、build，必要时本地验证核心流程。

不适合交给 AI 的任务包括业务决策、安全策略、无上下文复杂重构和高风险权限逻辑。

### 追问逐题回答

#### 你会给 AI 哪些上下文？

需求背景、现有代码片段、目录结构、接口类型、组件规范、异常分支、验收标准和限制条件。上下文越清楚，输出越可控。

#### AI 输出后如何做 diff review？

看改动范围是否超出需求，是否破坏现有结构，是否有边界处理，是否引入不必要依赖，是否符合类型和代码风格。

#### 是否接入 lint、test、build 验证？

是。AI 代码至少要经过 lint、type check、test 或 build 验证，核心流程还要人工跑一遍。不能只看代码“像对的”。

#### AI 生成结果和现有代码风格不一致怎么办？

优先按项目既有风格调整，而不是为了 AI 输出引入新风格。必要时把组件结构、命名规则、工具函数使用方式作为上下文重新约束。

#### 什么任务不适合交给 AI？

业务最终决策、安全权限策略、没有上下文的大规模重构、涉及资金和隐私的高风险逻辑不适合直接交给 AI。AI 可以辅助分析，但不能替代责任人判断。

## Q42：你怎么解释“AI 工程化”不是概念包装？

### 推荐回答

我会用具体产出来解释，而不是讲概念。我的 AI 工程化主要是 1 套流程、10+ 任务模板、5 类高频场景。它落在需求拆解、接口联调、异常排查、测试补充、文档沉淀这些真实研发动作里。

如果没有模板、流程、验证和复用，只是问 AI 几个问题，那确实是概念包装。我的口径是把 AI 当作结构化辅助工具，最终仍然靠人工 review 和测试验证。

### 追问逐题回答

#### 有没有具体模板、具体流程、具体产出？

有。模板包括需求拆解、接口联调、异常排查、测试补充、上线检查、文档沉淀等；流程覆盖输入上下文、生成初稿、人工校验、测试验证和沉淀复用。

#### 有没有被团队成员复用？

我会谨慎表达：这套流程用于团队协作和新人理解复杂链路，能降低上下文沟通成本。如果没有明确统计，不说大规模推广，只说在项目协作中沉淀和复用。

#### 有没有减少重复文档或排查成本？

有减少重复整理的效果，尤其是异常排查、接口联调和文档沉淀。但如果没有精确统计，我不会说具体百分比，只讲减少重复问答和上下文整理成本。

#### 你是否把 AI 当成最终决策者？

不是。AI 只做辅助，最终决策靠工程师 review、测试结果、业务验收和线上反馈。

#### AI skills 和普通 ChatGPT 问答有什么区别？

区别是结构化和可复用。普通问答是临时提问，AI skills 是固定输入格式、约束、输出结构和验证流程，能在类似任务中复用。
