# 岗位库字段设计

目标：长期维护 Boss 岗位库，支持岗位录入、筛选、分档、应聘进度管理、JSON 导入导出。

技术栈：

```text
React + TypeScript + TailwindCSS
持久化：JSON
```

## 1. 核心管理对象

一个岗位对应一条 `Job` 数据。

字段设计原则：

```text
原始 JD 要保留，便于回看。
结构化字段要足够筛选，便于批量管理。
分档规则和个人意向要分开，避免主观标记污染系统判断。
应聘进度要独立维护，方便长期跟踪。
```

| `updatedAt` | `string` | 是 | 更新时间 |

## 3. 枚举设计

### 3.1 融资/上市阶段

```ts

  series_c: 'C 轮',
  series_d: 'D 轮',

  unknown: '未知',

```

### 3.2 岗位分档

```ts
export type JobTier = 'practice' | 'fit' | 'target';
```

中文展示：

```ts
export const jobTierLabel: Record<JobTier, string> = {
  practice: '适合练手',
  fit: '适合去',
  target: '想去',
};
```

融资阶段默认分档规则：

```ts
export const getStageTier = (stage: CompanyStage): JobTier => {
  if (['no_financing', 'angel', 'series_a'].includes(stage)) {
    return 'practice';
  }

  if (['series_b', 'series_c'].includes(stage)) {
    return 'fit';
  }

  if (['series_d', 'listed'].includes(stage)) {
    return 'target';
  }

  return 'practice';
};
```

说明：

```text
适合练手：不需要融资、天使轮、A 轮
适合去：B 轮、C 轮
想去：D 轮、上市
```

### 3.3 个人意向

`userIntent` 和 `fitTier` 分开。原因是：有些公司融资阶段一般，但你主观想去；也有些上市公司岗位不匹配，不应该强行进入最终想去池。

```ts
export type UserIntent = 'unset' | 'not_interested' | 'normal' | 'want';
```

中文展示：

```ts
export const userIntentLabel: Record<UserIntent, string> = {
  unset: '未标记',
  not_interested: '不感兴趣',
  normal: '一般',
  want: '个人想去',
};
```

## 6. JSON 持久化结构

建议使用一个 JSON 文件作为完整岗位库。

```json
{
  "version": 1,
  "currentSalaryK": 18,
  "updatedAt": "2026-07-04T00:00:00.000Z",
  "jobs": []
}
```

对应类型：

```ts
export interface JobDatabase {
  version: number;
  currentSalaryK: number;
  updatedAt: string;
  jobs: Job[];
}
```

纯前端实现建议：

```text
浏览器内：localStorage 保存当前岗位库。
文件级备份：支持导入/导出 jobs.json。
后续如果要真正写入本地文件，可以再加 Node/Koa 或 Electron/Tauri。
```

## 8. 推荐页面功能

### 8.1 岗位列表页

需要支持：


## 9. 首版实现建议

首版不要做太复杂，按这个顺序实现：

```text
1. 手动维护 JSON 岗位库。
2. React 页面读取 JSON 并展示列表。
3. 支持按档位、进度、融资阶段筛选。
4. 支持修改应聘进度和个人意向。
5. 支持导出 JSON。
6. 再考虑从 1.txt 自动解析岗位。
```

## 10. 当前规则落地注意点

### 融资阶段不是唯一标准

你的新规则是：

```text
适合练手：不需要融资、天使轮、A 轮
适合去：B 轮、C 轮
想去：D 轮、上市
```

但实际推荐时仍建议保留 `fitTier`：

```text
stageTier：只看融资阶段。
fitTier：综合技术匹配、薪资、业务方向、风险后的最终推荐。
userIntent：你个人手动标记。
```

这样做的好处是：

```text
上市公司但岗位不匹配，可以 stageTier=target，fitTier=practice。
不需要融资但薪资高、AI 匹配强，可以 stageTier=practice，fitTier=fit。
你主观想去的岗位，可以 userIntent=want，不影响系统规则。
```
