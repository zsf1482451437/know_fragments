# 翟思丰 P2 面试题答案稿

来源：`interview_question_bank_zhaisifeng.md` 中所有 `P2` 问题。

P2 的定位是补充技术深度，不是面试开场高频题。回答时优先控制在 1-2 分钟内：先讲判断标准，再讲项目里的使用场景，最后补一句边界。

## Q10：Next.js SSR、SSG、CSR 怎么选？

### 速记链路

```text
看 SEO
  -> 看数据实时性
  -> 看是否用户态
  -> 公共内容优先 SSR/SSG
  -> 用户私有数据 CSR 补充
```

### 推荐回答

我一般不会固定说某类页面一定用 SSR、SSG 或 CSR，而是按四个维度判断：是否需要 SEO、数据是否实时变化、内容是否和用户身份强相关、以及页面性能和维护成本。

如果是商品详情页、解决方案页、活动落地页这类公共内容页面，核心内容需要被搜索引擎稳定抓取，我会优先考虑 SSG 或 SSR。内容更新频率低、上线前就能确定的页面适合 SSG，比如部分活动页、品牌内容页、解决方案页；如果页面内容必须请求服务端实时数据，又要保证首屏 HTML 可见，可以考虑 SSR。

CSR 更适合用户中心、订单页、后台配置页、登录后页面这类用户态强、SEO 要求低的场景。这类页面数据和用户身份绑定，不适合做公共缓存，也没有必要把所有内容都放到服务端渲染。

在跨境商城项目里，我的取舍会是：商品卖点、规格、FAQ、活动文案这类 SEO 内容尽量服务端或静态输出；价格、库存、购物车、用户优惠这类动态或私有数据，可以在客户端补充。

### 追问逐题回答

#### 商品详情页适合 SSR 还是 SSG？

看商品数据变化频率。商品基础信息、卖点、规格、FAQ 相对稳定时，SSG 更合适，能利用 CDN 缓存，首屏也更稳定。如果价格、库存、促销状态实时变化，可以拆成“基础内容 SSG + 动态数据 CSR”。只有当核心内容也强依赖实时服务端数据，并且必须首屏输出时，才考虑 SSR。

#### 活动页适合 SSG 吗？

多数活动页适合 SSG，因为活动主视觉、文案、规则、SEO 信息通常上线前就确定。对于倒计时、库存、用户专属优惠这类动态内容，可以客户端请求补充。这样能保证静态内容快，动态内容也不影响公共缓存。

#### 用户态数据怎么处理？

用户态数据一般放到客户端处理。比如登录状态、购物车、订单、用户优惠，都和用户身份绑定，不适合静态缓存到公共页面。更稳的方式是公共内容 SSR/SSG，用户私有模块在 hydration 后 CSR 请求。

#### SSR 的成本是什么？

SSR 会让每次请求都依赖服务端渲染，带来服务器计算成本、接口超时风险、缓存复杂度和部署稳定性要求。如果接口慢，SSR 首屏也会慢。所以 SSR 不是越多越好，只有页面确实需要实时服务端首屏内容时才值得用。

### 面试收口

```text
我的判断原则是：SEO 和公共首屏内容优先 SSR/SSG，用户态和强交互内容用 CSR 补充。不是追求某一种渲染方式，而是在 SEO、性能、实时性和实现成本之间做平衡。
```

### 风险边界

- 不要说“商品页一定 SSR”，很多商品基础内容适合 SSG。
- 不要说“CSR 都不利于 SEO 所以不能用”，用户态页面用 CSR 很正常。
- 不要主动讲 Next.js 版本升级，当前简历已经去掉升级口径。

## Q11：事件循环怎么理解？

### 速记链路

```text
同步代码先执行
  -> 宏任务开始
  -> 清空微任务
  -> 浏览器尝试渲染
  -> 执行下一个宏任务
```

### 推荐回答

事件循环是 JavaScript 调度同步任务、异步回调和页面渲染的一套机制。因为 JS 主线程一次只能执行一段代码，所以异步任务不会立即打断当前执行栈，而是进入对应队列，等当前同步代码执行完再调度。

我会把它理解成三个层次：同步代码、微任务、宏任务。同步代码先执行；一个宏任务执行结束后，会先清空当前产生的微任务队列；微任务清空后，浏览器才有机会进行渲染，然后进入下一轮宏任务。

常见宏任务有 `setTimeout`、`setInterval`、I/O、UI 事件；常见微任务有 `Promise.then`、`queueMicrotask`、`MutationObserver`。实际开发里，理解这个机制主要是为了判断异步执行顺序、避免微任务过多阻塞渲染，以及理解 React 状态更新为什么不是立刻读到最终 DOM。

### 追问逐题回答

#### Promise 和 setTimeout 谁先执行？

同一轮同步代码里，`Promise.then` 是微任务，`setTimeout` 是宏任务。同步代码执行完后，会先清空微任务，所以通常 Promise 回调先执行，setTimeout 在下一轮宏任务里执行。

#### async/await 和事件循环有什么关系？

`await` 前面的代码是同步执行的。遇到 `await` 后，后面的代码会等 Promise 状态确定后，再以微任务的形式继续执行。所以 `await` 不是阻塞线程，而是把后续逻辑拆到异步阶段。

#### 微任务太多会有什么问题？

浏览器需要等微任务队列清空后才有机会渲染。如果代码持续创建大量微任务，会导致渲染迟迟不能发生，表现为页面卡顿、点击响应慢、动画不流畅。

#### React setState 和事件循环有什么关系？

React 的状态更新会被调度和批处理，不应该简单理解为 `setState` 后马上同步更新 DOM。实际开发里，如果需要基于最新状态处理逻辑，要注意状态更新的时机、闭包问题，以及 React 批处理带来的渲染合并。

### 面试收口

```text
我理解事件循环的重点不是背宏任务列表，而是知道同步代码、微任务、渲染、下一轮宏任务的顺序。这个顺序对判断异步执行、渲染卡顿和状态更新时机很关键。
```

### 风险边界

- 不要说微任务永远比渲染早，要结合浏览器事件循环阶段理解。
- 不要把 React 状态更新简单等同于原生事件循环，它还有自己的调度和批处理。

## Q12：浏览器渲染流程是怎样的？

### 速记链路

```text
HTML -> DOM
CSS -> CSSOM
DOM + CSSOM -> Render Tree
Layout -> Paint -> Composite
```

### 推荐回答

浏览器渲染可以理解为从资源到像素的过程。首先解析 HTML 生成 DOM，解析 CSS 生成 CSSOM，然后把 DOM 和 CSSOM 合并成 Render Tree。接着进行 Layout，计算元素的位置和尺寸；然后 Paint，把颜色、文字、边框、阴影等绘制出来；最后 Composite，把不同图层合成并显示到屏幕上。

前端性能优化里常说的重排、重绘、合成，对应的就是这些阶段。修改宽高、位置、字体大小这类会影响几何信息的属性，通常会触发 Layout，也就是重排；修改颜色、背景这类不影响布局的样式，通常触发 Paint；`transform`、`opacity` 这类属性很多情况下可以走 Composite，成本相对更低。

在商城项目里做性能优化时，我会关注首屏关键资源加载、图片体积、CSS/JS 阻塞、长任务、LCP 元素加载时机。浏览器渲染流程不是只用于背概念，而是帮助定位为什么首屏慢、为什么动画卡、为什么交互响应慢。

### 追问逐题回答

#### DOMContentLoaded 和 load 区别是什么？

`DOMContentLoaded` 表示 HTML 已经解析完成，DOM 可以操作，但图片、字体等资源不一定加载完。`load` 表示页面依赖的资源基本加载完成。做性能分析时，不能把这两个时间点混为一谈。

#### 重排和重绘有什么区别？

重排是重新计算元素尺寸和位置，通常成本更高，因为可能影响后续节点布局。重绘是重新绘制视觉样式，比如颜色、背景、阴影。重排通常会伴随重绘，但重绘不一定触发重排。

#### 哪些操作容易触发重排？

修改 `width`、`height`、`margin`、`padding`、`display`、`position`、字体大小，或者频繁读取 `offsetWidth`、`clientHeight`、`getBoundingClientRect` 并穿插写样式，都可能触发重排。

#### 为什么 transform/opacity 动画更推荐？

因为它们很多情况下可以在合成阶段处理，不需要重新布局和绘制，能减少主线程压力。但也不能滥用，图层过多会增加内存占用和合成成本。

### 面试收口

```text
浏览器渲染流程对性能优化的价值在于定位成本来源：是资源阻塞、布局计算、绘制成本，还是合成和长任务问题。实际优化会围绕减少阻塞、减少重排重绘、优化图片和避免 JS 长任务展开。
```

### 风险边界

- 不要说 `transform/opacity` 一定不会触发任何成本，它们也可能带来合成和内存压力。
- 不要把 LCP 只归因到前端代码，图片、网络、服务端响应、CDN 都会影响。

## Q13：React diff 怎么理解？

### 速记链路

```text
不是通用树 diff
  -> 同层比较
  -> 类型不同直接替换
  -> 类型相同比较 props 和 children
  -> 列表依赖稳定 key
```

### 推荐回答

React diff 的目标是在状态变化后，尽量找出需要更新的最小 UI 范围。但它不是做理论上最完整的树比较，因为完整树 diff 成本太高，不适合高频 UI 更新。React 采用的是基于 UI 场景的启发式 diff。

核心规则有三点：第一，不同类型的节点会直接替换，比如 `div` 变成 `span`，或者组件类型变了，React 会认为是不同结构；第二，同类型节点会比较 props，并继续比较子节点；第三，列表通过 key 识别元素的稳定身份，判断是复用、移动、新增还是删除。

实际开发里，diff 相关问题经常不是 React 算法本身，而是我们写法导致的：列表 key 不稳定、用随机 key、组件 props 每次都生成新引用、状态粒度过大，都会带来重复渲染或状态错乱。

### 追问逐题回答

#### 为什么 React 不做完整树比较？

完整树比较复杂度高，不适合前端高频更新。React 基于常见 UI 变化规律做同层比较和类型判断，用稳定 key 辅助列表复用，在性能和正确性之间做平衡。

#### key 为什么不能随便用 index？

如果列表只是静态展示，index 问题不大。但如果列表会插入、删除、排序，index 会导致元素身份错乱，React 可能错误复用组件状态。比如输入框列表用 index 作为 key，删除一项后可能出现输入内容错位。

#### 随机 key 为什么不好？

随机 key 每次渲染都变化，React 会认为所有节点都是新的，导致无法复用组件和状态，反而增加渲染成本，还可能导致输入状态丢失。

#### diff 和性能优化有什么关系？

diff 只是决定哪些节点需要更新。真正的性能优化还包括状态拆分、组件 memo、稳定 props、避免无意义状态提升、长列表虚拟化等。不能只靠 diff 自动解决所有性能问题。

### 面试收口

```text
我理解 React diff 的重点是稳定身份和同层比较。实际写代码时，会用稳定业务 ID 做 key，控制状态粒度，避免无意义 props 变化，让 React 更容易复用已有组件。
```

### 风险边界

- 不要说 key 能让列表不重新渲染，它只是帮助 React 判断身份。
- 不要说虚拟 DOM diff 一定比手写 DOM 快，框架抽象也有成本。

## Q14：虚拟 DOM 的价值是什么？

### 速记链路

```text
JS 对象描述 UI
  -> 状态驱动视图
  -> diff 找变化
  -> 调度合并更新
  -> 支持跨平台宿主
```

### 推荐回答

虚拟 DOM 本质上是用 JavaScript 对象描述 UI 结构。它的价值不是简单地说“比真实 DOM 快”，而是给 React 提供了一层 UI 抽象，让开发者用声明式方式描述状态和视图的关系。

有了虚拟 DOM，状态变化后，React 可以先在内存里生成新的 UI 描述，再通过 diff 找出变化范围，最后把必要更新提交到真实宿主环境。这个宿主环境在 Web 里是 DOM，在 React Native 里是原生视图。

所以虚拟 DOM 的价值主要有三点：第一，提升开发模型，让我们从手动操作 DOM 变成状态驱动 UI；第二，给框架调度和批量更新提供空间；第三，支持跨平台抽象，让 React 的组件模型可以应用到 Web、Native 等不同环境。

### 追问逐题回答

#### 虚拟 DOM 一定比真实 DOM 快吗？

不一定。简单场景下手动操作 DOM 可能更直接。虚拟 DOM 的核心价值是可维护性、声明式抽象、跨平台和调度能力，不是绝对性能更快。

#### 虚拟 DOM 和 React Native 有什么关系？

React Native 不渲染浏览器 DOM，但仍然使用 React 的组件模型和 reconciliation 思路。React 会把组件树映射到原生视图，底层宿主环境不同，但状态驱动 UI 的模型是相通的。

#### 虚拟 DOM 有哪些成本？

成本包括创建虚拟节点、diff 计算、内存占用和调度开销。大型列表、频繁更新、复杂组件树仍然需要做状态拆分、memo、虚拟列表等优化。

#### 为什么虚拟 DOM 能提升开发效率？

因为开发者不用手动维护复杂的 DOM 增删改和状态同步，只需要描述“当前状态应该是什么 UI”。这减少了命令式 DOM 操作带来的状态不一致问题。

### 面试收口

```text
虚拟 DOM 的价值是抽象和工程化。它让 React 用声明式方式管理 UI，并给 diff、调度、跨平台渲染提供基础，但不代表所有场景都天然更快。
```

### 风险边界

- 不要把虚拟 DOM 神化成性能银弹。
- 不要说 React Native 有浏览器 DOM，它是映射到原生视图。

## Q15：Webpack 和 Vite 构建差异是什么？

### 速记链路

```text
Webpack：先打包再启动
Vite：开发期 ESM 按需加载
依赖预构建：esbuild
生产构建：Rollup
选型看生态和项目约束
```

### 推荐回答

Webpack 和 Vite 最大的差异在开发阶段。Webpack 通常会先对项目进行打包，再启动 dev server。项目规模变大后，冷启动和热更新都可能变慢。Vite 开发阶段利用浏览器原生 ES Module，源码按需请求，依赖用 esbuild 预构建，所以冷启动和 HMR 通常更快。

生产构建上，Vite 默认使用 Rollup 做打包优化，而 Webpack 使用自己的打包体系。Webpack 的优势是生态成熟、loader 和 plugin 丰富，适合复杂历史项目和深度定制构建；Vite 的优势是开发体验好、配置更轻，适合现代 ESM 项目和中小型应用。

结合我的项目经历，Next.js 商城更多遵循框架自身构建体系；Shopify 自研应用使用 Vite，更看重开发启动速度、配置简洁和前端应用的交付效率。实际选型不是谁更先进，而是看框架约束、插件生态、团队熟悉度、CI 部署和迁移成本。

### 追问逐题回答

#### Vite 为什么冷启动快？

因为它开发阶段不需要完整打包业务代码，而是让浏览器按 ESM 模块按需请求源码。依赖部分通过 esbuild 预构建，速度很快。

#### Webpack 的优势是什么？

Webpack 生态成熟，loader/plugin 丰富，对历史项目、复杂资源处理、构建链路定制支持更强。大型老项目如果强依赖 Webpack 插件，迁移 Vite 要谨慎评估。

#### Vite 生产环境为什么还要打包？

开发阶段可以按需加载，但生产环境需要考虑兼容性、压缩、tree shaking、chunk 分割、缓存策略和加载性能，所以仍然需要打包优化。

#### esbuild 和 Rollup 在 Vite 里分别做什么？

开发阶段 esbuild 主要用于依赖预构建和快速转译；生产构建主要由 Rollup 负责打包、tree shaking 和产物组织。

#### 什么时候不适合迁移 Vite？

如果项目强依赖 Webpack 插件、loader、历史构建定制，或者框架本身不支持，迁移成本可能高于收益。迁移前要评估插件替代、构建产物、CI、部署和兼容性。

### 面试收口

```text
Webpack 和 Vite 的核心差异是开发期构建模型不同。Vite 借助 ESM 和 esbuild 提升开发体验，Webpack 胜在生态成熟和复杂定制。选型要看项目约束，不是简单追新。
```

### 风险边界

- 不要说 Vite 全场景一定优于 Webpack。
- 不要说 Vite 生产环境不需要打包。
- 不要主动说做过复杂 Webpack 插件，除非简历和项目能支撑。

## Q16：React Native 新架构怎么理解？

### 速记链路

```text
旧架构：Bridge 异步序列化通信
新架构：JSI + TurboModules + Fabric
目标：减少通信成本，提升渲染和模块调用效率
边界：了解原理，不夸大迁移经验
```

### 推荐回答

React Native 新架构主要围绕 JSI、TurboModules、Fabric 这几块展开，目标是改善旧 Bridge 架构下 JS 和 Native 通信成本高、模块加载不够高效、渲染链路不够现代化的问题。

旧架构里，JS 和 Native 通信主要通过 Bridge，数据需要序列化、异步传递。如果频繁调用原生能力，或者传递大量数据，会有性能成本。新架构中，JSI 让 JS 引擎可以更直接地和 C++/原生对象交互；TurboModules 让原生模块可以按需加载，并提升类型和调用效率；Fabric 是新的渲染系统，更好地和 React 的并发渲染方向配合。

结合我的项目，我会把它理解成 RN 和原生协作的底层演进。像实时预览这种场景，RN 侧仍然要关注 Native Module 的边界、事件回传、状态映射和资源释放。但我不会说自己主导了 RN 新架构迁移，除非项目真实做过。

### 追问逐题回答

#### JSI 解决什么问题？

JSI 让 JS 可以更直接地访问 C++/原生能力，减少传统 Bridge 中 JSON 序列化和异步队列通信的成本。它是 TurboModules 和新架构的重要基础。

#### TurboModules 和旧 Native Modules 有什么区别？

TurboModules 更强调按需加载、类型约束和更高效的调用链路。旧 Native Modules 主要通过 Bridge 异步通信，模块加载和调用成本相对更高。

#### Fabric 是什么？

Fabric 是 RN 新的渲染系统，用来替代旧 UIManager 渲染链路。它更贴近 React 新的渲染模型，目标是提升渲染一致性、响应能力和跨端表现。

#### 新架构是不是一定要马上迁移？

不一定。迁移要看 RN 版本、三方库兼容、原生模块复杂度、业务稳定性和团队经验。成熟业务不应该为了新技术盲目迁移。

#### 你该怎么讲自己的边界？

可以说“了解 RN 新架构方向，理解 Bridge 通信成本和 Native Modules 边界，在项目中主要负责 RN 侧业务接入、状态管理和原生事件消费”。不要说自己主导新架构或底层改造。

### 面试收口

```text
我对 RN 新架构的理解是，它围绕 JSI、TurboModules、Fabric 改善旧 Bridge 通信和渲染链路的问题。我的项目里更关注 RN 侧如何和原生能力稳定协作，不夸大成底层迁移经验。
```

### 风险边界

- 不要说自己主导 RN 新架构迁移。
- 不要把 JSI、TurboModules、Fabric 混成一个概念。
- 不要说新架构一定能解决所有性能问题，业务代码和三方库也会影响性能。

## Q17：JSBridge 原理是什么？

### 速记链路

```text
H5 JS 发消息
  -> Native 拦截并解析协议
  -> 执行原生能力
  -> callbackId 回调 JS
  -> 做权限、安全、版本兜底
```

### 推荐回答

JSBridge 是 WebView 里 H5 和 Native 通信的一套机制，本质是让网页里的 JavaScript 能调用 App 原生能力，同时 Native 也能把结果回调给 JS。常见场景包括登录、分享、支付、定位、相册、设备信息、扫码等。

常见实现方式有几类：第一，Native 向 WebView 注入 JS 对象，H5 直接调用对象方法；第二，H5 通过自定义 URL scheme 发起调用，Native 拦截 URL 后解析协议；第三，使用 `postMessage` 或平台 message handler 传递消息。Native 执行完原生逻辑后，再通过 `evaluateJavaScript` 或统一事件回调把结果返回给 H5。

设计 JSBridge 时，真正重要的不只是能调通，而是协议、回调、权限、安全和兼容性。协议里通常会包含 `method`、`params`、`callbackId`、`version`、`timestamp` 等字段；Native 侧要做方法白名单、域名白名单、参数校验、超时和错误码兜底。

### 追问逐题回答

#### JS 调 Native 常见方式有哪些？

常见方式包括注入对象调用、URL scheme 拦截、`postMessage` 或 message handler。不同平台实现细节不同，但本质都是 JS 发送结构化消息，Native 接收后执行能力。

#### Native 怎么回调 JS？

Native 可以通过 `evaluateJavaScript` 执行 JS 回调，也可以通过统一事件机制把结果回传。一般会用 `callbackId` 标识一次调用，避免多个异步请求回调错乱。

#### JSBridge 协议怎么设计？

协议通常包含 `method`、`params`、`callbackId`、`version`、`timestamp`。`method` 表示调用能力，`params` 是参数，`callbackId` 用于异步回调，`version` 用于兼容，`timestamp` 可用于超时或排查。

#### JSBridge 有哪些安全风险？

风险包括不可信 H5 调用敏感原生能力、参数注入、域名伪造、方法滥用、回调被伪造。需要做域名白名单、方法白名单、参数校验、登录态和权限校验。

#### JSBridge 和 RN Native Modules 有什么区别？

JSBridge 通常指 WebView 内 H5 和 App Native 的通信；RN Native Modules 是 React Native JS 运行时和原生模块的通信。两者都解决 JS 与 Native 通信，但宿主环境不同，调用链路和安全边界也不同。

### 面试收口

```text
JSBridge 的核心不是把 JS 和 Native 调通，而是把协议、回调、权限、安全、版本兼容和异常兜底设计清楚。能调通是基础，能稳定、安全地运行才是关键。
```

### 风险边界

- 不要把 JSBridge 和 RN Native Modules 完全等同。
- 不要只讲调用方式，要补安全和协议设计。
- 如果项目没有深度做 WebView 容器，不要说自己做过完整 JSBridge 底层。
