## Q10：Next.js SSR、SSG、CSR 怎么选？

### 推荐回答

我一般会按页面内容是否依赖实时数据、是否需要 SEO、更新频率和个性化程度来选。

如果页面强依赖 SEO，内容相对稳定，比如商品详情页、解决方案页、活动介绍页，优先考虑 SSG 或 SSR。SSG 适合内容更新频率不高、可以构建时生成的页面，比如部分商品页、品牌内容页、解决方案页；SSR 适合需要服务端实时取数、但又要保证首屏内容和 SEO 的页面。

CSR 更适合用户态强、个性化强、SEO 要求低的页面，比如用户中心、订单页、后台配置页、登录后才能看到的数据页。这类页面没必要把所有内容放到服务端渲染。

在商城类项目里，我的取舍是：SEO 页面尽量不要完全依赖 CSR；活动页和商品页要优先保证首屏内容可索引和 LCP；用户态数据可以客户端再补充。

### 追问逐题回答

#### 商品详情页适合 SSR 还是 SSG？

要看商品数据更新频率。

如果商品信息、卖点、规格、FAQ 相对稳定，适合 SSG，配合增量构建或定时更新，可以兼顾 SEO 和性能。用户看到的核心商品内容在 HTML 里就有，搜索引擎也更容易收录。

如果商品价格、库存、促销状态变化很频繁，或者页面必须实时读取服务端数据，可以考虑 SSR。但也可以做拆分：商品基础信息用 SSG，价格、库存、用户态优惠用 CSR 请求补充。这样既保留 SEO，又避免每次请求都走 SSR。

#### 活动页适合 SSG 吗？

很多活动页适合 SSG，因为活动页通常有明确的活动内容、主视觉、文案、商品模块和 SEO 信息，活动上线前就能生成静态页面。SSG 的好处是首屏快、CDN 缓存友好、服务端压力小。

但如果活动价格、库存、倒计时、用户专属优惠实时变化，就不能全部静态化。可以采用“静态内容 SSG + 动态数据 CSR”的方式：活动规则、主视觉、SEO 内容静态输出；价格、库存、用户状态客户端请求。

#### 需要用户态数据的页面怎么处理？

用户态数据通常放到客户端处理。比如登录状态、购物车、用户优惠、订单信息，这些内容和用户身份绑定，不适合直接做公共静态页面。

常见做法是页面主体用 SSR/SSG 输出通用内容，用户态模块在客户端 hydration 后再请求接口。这样既保证公共内容的首屏和 SEO，又避免缓存用户私有数据导致安全问题。

#### SSR 会带来哪些性能和运维成本？

SSR 每次请求都需要服务端参与渲染，相比 SSG 会增加服务器计算成本、接口依赖成本和缓存复杂度。如果接口慢，SSR 页面首屏也会被拖慢。SSR 还需要考虑服务端稳定性、错误兜底、超时、缓存策略和部署资源。

所以 SSR 不是越多越好。只有页面确实需要实时数据，同时又需要服务端输出首屏内容时，SSR 才更合适。能静态化的页面优先静态化，能客户端补充的用户态数据就客户端补充。

#### SEO 页面为什么不能完全依赖 CSR？

CSR 页面核心内容需要浏览器执行 JS 后才出现。虽然现代搜索引擎能处理一部分 JS，但抓取和渲染存在延迟，也可能因为接口、脚本错误、性能问题导致核心内容不可见。

商品页、活动页、解决方案页这类依赖自然流量的页面，最好让标题、描述、商品卖点、规格参数、FAQ 等核心内容在 HTML 阶段就可见。这样对搜索引擎更稳定，也对首屏体验更友好。

### 面试收口

可以用这句话收束：

```text
我的选择原则不是固定用 SSR 或 SSG，而是看页面是否需要 SEO、数据是否实时、是否有用户态内容。商城里公共内容和 SEO 内容尽量服务端或静态输出，用户私有数据客户端补充，这样能在性能、SEO 和实现成本之间取得平衡。
```

### 风险边界

- 不要说“所有商品页都必须 SSR”。
- 不要说“SSG 一定比 SSR 好”，它们适用场景不同。
- 不要说“CSR 不利于 SEO 所以不能用”，用户态页面用 CSR 很正常。
- 不要提 Next.js 版本升级经历，当前简历已经去掉升级口径。

## Q11：事件循环怎么理解？

### 推荐回答

事件循环是 JavaScript 处理异步任务的机制。因为 JS 主线程一次只能执行一段同步代码，所以异步任务需要通过任务队列调度。

整体可以分成同步代码、宏任务和微任务。同步代码先执行；执行完一个宏任务后，会清空当前产生的微任务队列；然后浏览器有机会进行渲染；再进入下一个宏任务。常见宏任务包括 `setTimeout`、`setInterval`、I/O、UI 事件；常见微任务包括 `Promise.then`、`queueMicrotask`、`MutationObserver`。

面试里我一般会强调两点：第一，微任务优先级高于下一个宏任务；第二，微任务过多会阻塞渲染，因为浏览器要等微任务队列清空后才进入下一轮。

### 追问逐题回答

#### Promise 和 setTimeout 谁先执行？

同一轮同步代码结束后，`Promise.then` 作为微任务会先执行，`setTimeout` 是宏任务，会进入下一轮事件循环。所以通常是 Promise 先于 setTimeout。

#### async/await 和事件循环有什么关系？

`await` 后面的代码可以理解为被放到微任务里继续执行。`await` 前是同步执行，遇到 await 后让出执行权，等 Promise 状态确定后，再把后续逻辑放入微任务队列。

#### 微任务太多会有什么问题？

微任务队列必须清空后，浏览器才有机会进入渲染或下一个宏任务。如果持续创建大量微任务，可能导致页面渲染被阻塞，表现为卡顿或用户交互延迟。

#### React setState 和事件循环有什么关系？

React 的状态更新会被调度和批处理，具体表现和 React 版本、执行上下文有关。不能简单理解为 setState 立即同步更新 DOM。实际开发里要关注状态更新后的读取时机，以及批处理带来的渲染合并。

### 面试收口

```text
我理解事件循环的关键不是背宏任务微任务列表，而是知道同步代码、微任务、渲染和下一轮宏任务的执行顺序，这对排查异步顺序、渲染卡顿和状态更新问题很有帮助。
```

## Q12：浏览器渲染流程是怎样的？

### 推荐回答

浏览器渲染大致可以分成：解析 HTML 生成 DOM，解析 CSS 生成 CSSOM，合并成 Render Tree，然后进行 Layout 计算位置和尺寸，再 Paint 绘制，最后 Composite 合成图层显示到屏幕。

性能优化里经常提到的重排和重绘就对应这里。修改元素尺寸、位置、布局会触发 Layout，也就是重排；修改颜色、背景这类不影响布局的样式通常触发 Paint；如果使用 transform、opacity 这类合成层友好的属性，可能只触发 Composite，成本相对低。

在项目里做 LCP 和页面性能优化时，我会关注首屏 HTML、CSS、JS、图片加载顺序，避免 JS 长任务阻塞主线程，也避免首屏关键资源加载太慢。

### 追问逐题回答

#### DOMContentLoaded 和 load 区别是什么？

`DOMContentLoaded` 表示 HTML 已解析完成，DOM 可用，不一定等图片、字体等资源加载完。`load` 表示页面依赖的资源基本加载完成。性能分析时两者含义不同，不能混用。

#### 重排和重绘有什么区别？

重排是重新计算元素几何信息，比如尺寸、位置、布局；重绘是重新绘制样式，比如颜色、背景。重排通常成本更高，因为可能影响后续节点布局。

#### 哪些操作容易触发重排？

修改宽高、margin、padding、display、position、字体大小，或者读取 `offsetWidth`、`getBoundingClientRect` 等布局信息并穿插写操作，都可能触发重排。

#### 为什么 transform/opacity 动画性能更好？

它们通常可以在合成阶段处理，不一定触发布局和绘制，能减少主线程压力。但也不是无限制使用，图层过多会增加内存成本。

### 面试收口

```text
浏览器渲染流程对前端性能优化很关键。实际优化时，我会尽量减少首屏阻塞资源、减少重排重绘，并把动画优先放到 transform 和 opacity 这类更适合合成的属性上。
```

## Q13：React diff 怎么理解？

### 推荐回答

React diff 的核心目标是在两棵虚拟 DOM 树之间找到最小必要更新，但它不是通用树 diff，而是基于启发式假设降低复杂度。

主要规则有三点：不同类型的节点会直接替换；同类型节点会比较 props 并递归比较子节点；列表节点依赖 key 来识别稳定身份。key 的作用不是消除 diff，而是帮助 React 判断哪些元素是移动、复用还是新增删除。

实际开发中，diff 相关问题经常体现在列表 key 不稳定、组件拆分不合理、props 引用每次变化导致重复渲染。

### 追问逐题回答

#### 为什么 React diff 不做完整树比较？

完整树 diff 复杂度太高，实际 UI 更新需要高性能。React 基于 UI 的常见变化规律做启发式优化，比如不同类型直接替换、同层比较、依赖 key，这样性能更可控。

#### key 为什么不能用 index？

如果列表顺序会变化、插入或删除，index 会导致元素身份错乱，React 可能错误复用 DOM 或组件状态。稳定业务 ID 更适合作为 key。

#### key 是不是越随机越好？

不是。随机 key 每次渲染都变，React 会认为节点全是新的，导致无法复用，甚至丢失组件状态。key 要稳定、唯一、可预测。

#### diff 和性能优化有什么关系？

diff 本身只是找更新，性能还受组件渲染、props 稳定性、状态粒度影响。合理拆组件、稳定 key、避免无意义状态提升、配合 memo 才能减少重复渲染。

### 面试收口

```text
我理解 React diff 不是追求理论最优树比较，而是用 key、同层比较和节点类型判断，在常见 UI 场景里取得性能和正确性的平衡。
```

## Q14：虚拟 DOM 的价值是什么？

### 推荐回答

虚拟 DOM 本质是用 JavaScript 对象描述 UI 结构。它的价值不是“比真实 DOM 快”这么简单，而是提供一层抽象，让 React 能用声明式方式描述 UI，并通过 diff 和调度机制把状态变化转换成真实 DOM 更新。

虚拟 DOM 的收益包括：跨平台抽象，比如 React DOM 和 React Native 可以共享类似组件模型；声明式开发，开发者更多关注状态到 UI 的映射；批量更新和调度，让框架有机会合并更新、控制优先级。

但虚拟 DOM 也有成本，比如创建对象、diff 计算，并不意味着所有场景都比手写 DOM 快。

### 追问逐题回答

#### 虚拟 DOM 一定比真实 DOM 快吗？

不一定。简单场景下直接操作 DOM 可能更快。虚拟 DOM 的价值在于可维护性、声明式抽象、跨平台和更新调度，而不是绝对性能更快。

#### 虚拟 DOM 和 React Native 有什么关系？

React Native 不渲染浏览器 DOM，但同样使用 React 的组件模型和 reconciliation 思路，把组件树映射到原生视图。虚拟 DOM 或 fiber 这层抽象让 React 能支持不同宿主环境。

#### 虚拟 DOM 为什么能提升开发效率？

开发者只需要根据状态描述 UI，不需要手动维护复杂 DOM 操作。状态变化后由框架计算更新，减少命令式 DOM 操作带来的状态不一致问题。

#### 虚拟 DOM 有哪些成本？

成本包括虚拟节点创建、diff、内存占用和调度开销。大型列表、频繁状态更新场景仍然需要做 memo、虚拟列表、状态拆分等优化。

### 面试收口

```text
虚拟 DOM 的核心价值是抽象和工程化，而不是简单宣称比 DOM 快。它让 React 可以用声明式方式管理 UI，并支持 Web 和 Native 等不同渲染目标。
```

## Q15：Webpack 和 Vite 构建差异是什么？

### 推荐回答

Webpack 和 Vite 最大差异在开发阶段。Webpack 通常会先把项目依赖和业务代码打包后再启动 dev server，项目变大后冷启动和热更新成本会上升。Vite 开发阶段利用浏览器原生 ES Module，源码按需加载，依赖用 esbuild 预构建，所以冷启动通常更快。

生产构建上，Vite 默认基于 Rollup 打包，Webpack 使用自己的打包体系。Webpack 生态成熟、配置能力强，适合复杂历史项目；Vite 开发体验好、配置相对简洁，适合现代 ESM 项目。

我的简历里既有 Next.js 商城，也有 Vite 的 Shopify 应用。实际选型不是谁绝对更好，而是看项目生态、框架约束、插件能力和团队维护成本。

### 追问逐题回答

#### Vite 为什么冷启动快？

因为 Vite 开发阶段不需要先完整打包业务代码，而是利用浏览器 ESM 按需请求源码。依赖部分用 esbuild 预构建，速度比传统 JS 打包器快。

#### Webpack 的优势是什么？

Webpack 生态成熟、loader/plugin 丰富、配置能力强，对复杂资源处理、历史项目兼容、定制构建流程支持更强。

#### Vite 生产环境为什么还要打包？

开发阶段可以用 ESM 按需加载，但生产环境需要考虑兼容性、资源压缩、tree shaking、chunk 分割、缓存和加载性能，所以仍然需要打包优化。

#### esbuild 和 Rollup 在 Vite 里分别做什么？

Vite 开发阶段用 esbuild 做依赖预构建和快速转译；生产构建主要用 Rollup 做打包和优化。

#### 什么时候不适合迁移 Vite？

如果项目强依赖 Webpack 插件、loader、历史构建定制、框架本身不支持，迁移成本可能高于收益。要先评估插件替代、构建产物、CI、部署和兼容性。

### 面试收口

```text
Webpack 和 Vite 的差异核心在开发服务和构建机制。Vite 更强调现代 ESM 带来的开发体验，Webpack 更强调成熟生态和复杂定制能力，选型要看项目约束。
```

## Q16：React Native 新架构怎么理解？

### 推荐回答

React Native 新架构主要包括 Fabric、TurboModules、JSI 这些方向，目标是提升 JS 和原生之间通信效率、渲染性能和类型安全。

旧架构里 JS 和 Native 通信主要通过 Bridge，数据需要序列化和异步传递，频繁通信会有性能成本。新架构通过 JSI 让 JS 可以更直接地调用 C++/原生能力；TurboModules 改进原生模块加载和调用；Fabric 是新的渲染系统，更好地和 React 并发能力配合。

我会把它作为 React Native 性能和原生协作方向的理解，但不会说自己主导过 RN 新架构迁移。如果项目问得很深，我会明确边界。

### 追问逐题回答

#### JSI 解决什么问题？

JSI 允许 JS 引擎和原生 C++ 对象更直接交互，减少传统 Bridge 序列化通信的成本，是 TurboModules 和新架构的重要基础。

#### TurboModules 和旧 Native Modules 有什么区别？

TurboModules 更强调按需加载、类型约束和更高效调用。旧 Native Modules 通常通过 Bridge 异步通信，模块加载和调用成本更高。

#### Fabric 是什么？

Fabric 是 RN 新的渲染系统，用来替代旧 UIManager 渲染链路。它和 React 并发渲染能力更匹配，目标是提升渲染一致性和性能。

#### 新架构是不是一定要马上迁移？

不一定。迁移要看 RN 版本、三方库兼容、原生模块复杂度、团队能力和业务风险。成熟项目不能为了新技术盲目迁移。

#### 你在简历里该怎么讲 RN 新架构？

应该说“了解 RN 新架构方向，理解 Bridge 通信成本和 Native Modules 边界”，不要说主导新架构迁移，除非项目真实做过。

### 面试收口

```text
我对 RN 新架构的理解是，它主要围绕 JSI、TurboModules、Fabric 解决旧 Bridge 通信和渲染链路的性能问题。但我会把它作为理解和协作能力，不夸大成迁移经验。
```

## Q17：JSBridge 原理是什么？

### 推荐回答

JSBridge 是 Web/H5 和 Native 通信的一套机制，本质是让 JS 能调用原生能力，同时让原生能回调 JS。常见场景包括 App 内嵌 H5 调用登录、分享、支付、设备信息、相册、定位等原生能力。

实现方式常见有几种：一种是 Native 向 WebView 注入 JS 对象，JS 调用这个对象的方法；一种是 JS 通过自定义 URL scheme 或 postMessage 发送消息，Native 拦截后执行原生逻辑；原生执行完后再通过 evaluateJavaScript 或事件回调把结果返回给 JS。

设计 JSBridge 时要关注协议格式、权限控制、回调管理、超时处理、版本兼容和安全校验。

### 追问逐题回答

#### JS 调 Native 常见方式有哪些？

常见方式包括注入对象调用、URL scheme 拦截、postMessage/message handler。不同平台和 WebView 容器实现细节不同，但核心都是 JS 发消息，Native 接收并执行。

#### Native 怎么回调 JS？

Native 可以通过 evaluateJavaScript 执行页面上的回调函数，也可以通过统一事件分发机制把结果传回 JS。一般会有 callbackId 对应一次请求。

#### JSBridge 协议怎么设计？

通常包含 method、params、callbackId、version、timestamp 等字段。method 表示调用能力，params 是参数，callbackId 用于异步回调，version 用于兼容。

#### JSBridge 有哪些安全风险？

风险包括任意 H5 调用敏感原生能力、参数注入、来源不可信、回调被伪造。需要做域名白名单、方法白名单、参数校验、权限控制和版本校验。

#### JSBridge 和 React Native Native Modules 有什么区别？

JSBridge 通常是 WebView 内 H5 和 App 原生通信；Native Modules 是 React Native JS 运行时和原生模块通信。两者都解决 JS 与 Native 通信，但宿主环境和调用链路不同。

### 面试收口

```text
JSBridge 的核心是协议设计和安全边界。能调通不难，难的是权限、回调、异常、版本兼容和安全控制都设计清楚。
```
