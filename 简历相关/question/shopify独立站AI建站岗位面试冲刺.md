- 主线不要讲成单纯前端页面开发，要突出：`电商业务理解 + 工程化复用 + AI 提效 + Shopify 生态理解`。

- 回答顺序优先用：`结论 -> 业务理解 -> 技术做法 -> 个人匹配点 -> 边界`。
- 下午临场如果紧张，优先记住一句定位：我能把 React/Next.js、电商前端、Shopify 应用经验和 AI Agent 研发流程结合起来，提升多品牌独立站的交付效率。

### 第 2 小时：突击 JD 高频技术点

#### Shopify Online Store 2.0

必须会说：

> Online Store 2.0 的核心是 Section Everywhere，让页面可以通过 `Liquid + JSON template + Section + Block + Schema` 做模块化配置。Section 负责页面区块，Block 负责区块内部可重复子模块，Schema 把图片、文案、链接、开关、样式等配置暴露给运营。这样可以减少硬编码，提高主题复用和运营效率。

关键词：

- `theme.liquid`：全局布局入口。
- `templates/*.json`：页面模板配置。
- `sections/*.liquid`：可配置页面模块。
- `blocks`：Section 内的可重复配置单元。
- `schema`：给后台主题编辑器暴露配置项。
- `metafield`：扩展商品/集合/页面等业务数据。

#### Liquid 怎么回答

稳妥版：

> Liquid 我理解它是 Shopify 主题层的模板语言，核心是通过 object、tag、filter 渲染商品、集合、购物车、客户等数据。实际开发会结合 Section Schema 做可配置模块，再配合 JavaScript 处理商品变体切换、加购、购物车刷新、营销组件展示等交互。我不是多年 Liquid 主题专家，但我有电商前端和组件化经验，会优先围绕 PDP、Collection、Cart 和可配置 Section 快速补齐。

不要说：

- 不要说“我精通 Liquid 多年”。
- 不要把 Shopify 主题说成 React SPA。
- 不要忽略运营可配置能力，JD 很看重纯代码主题定制。

#### Core Web Vitals

必须背：

- `LCP`：最大内容渲染。优化首屏主图、字体、关键 CSS、图片格式、SSR/缓存、资源优先级。
- `INP`：交互响应。减少主线程阻塞，拆分长任务，延迟非关键 JS，控制第三方脚本。
- `CLS`：布局偏移。图片/视频预留尺寸，避免动态插入广告、评价、推荐组件造成抖动。

Shopify 多图多视频场景优化打法：

- 图片：WebP/AVIF、响应式图片、首屏图 preload、非首屏 lazyload。
- 视频：首屏不直接加载大视频，用 poster 和交互后加载。
- JS：延迟评论插件、推荐插件、营销弹窗、广告脚本。
- CSS：移除未使用样式，控制主题包体。
- 第三方：用 DevTools Coverage/Network 看插件成本，按转化价值决定是否延迟或替换。

#### Shopify API 三件套

推荐回答：

> Storefront API 偏前台购物体验，比如商品、集合、客户侧购物车能力；Admin API 偏后台管理和应用能力，比如商品、订单、客户、配置管理；AJAX API 更常用于主题前台轻量购物车交互，比如加购、更新数量、读取 cart。GraphQL 的优势是按需取字段，减少 over-fetching，对性能和移动端体验更友好。

面试官追问时可以补：

- PDP：商品信息、变体、库存、媒体资源、metafield。
- Cart：AJAX API 更新数量、删除商品、刷新 mini cart。
- Admin：订单拦截、后台配置、商品配置、应用数据管理。

#### GA4 / Meta Pixel

稳妥回答：

> 我之前更多是在商城页面和业务链路侧配合埋点，对 GA4/Meta Pixel 的理解是先定义核心转化事件，再保证事件触发时机、参数、去重和跨页面一致性。常见事件包括 `view_item`、`add_to_cart`、`begin_checkout`、`purchase`。调试时我会用 Tag Assistant、Meta Pixel Helper、Network、Console 和 GA DebugView 验证事件是否准确上报。

不要夸大：

- 不要说自己主导过完整投放归因模型。
- 可以说“能基于文档快速接入和调试，重点保证事件准确性”。

## Q4：独立站矩阵、多站点复用你会怎么做？

### 推荐回答

第二，品牌差异尽量配置化，比如主题色、字体、模块顺序、文案、多语言、导航、支付说明，而不是 fork 多份代码。

## Q5：你怎么看 Shopify 主题模板化？

### 推荐回答

我理解 Shopify 主题模板化的关键不是把页面写出来，而是让运营和品牌侧能低成本复用。

所以要把页面区块做成可配置 Section，比如 Banner、商品列表、卖点模块、FAQ、评价模块；再通过 schema 暴露配置项，让非技术同学可以在后台调整内容。

技术侧要控制好组件边界、样式变量、metafield 使用规范和埋点一致性，避免后期每个站点都变成定制代码。

### 追问：Section 和 schema 的价值是什么？

Section 可以把页面拆成可复用模块，schema 可以把模块的配置项暴露给运营或建站同学。这样开发只需要维护模块能力，运营可以调整图片、文案、链接、排序和开关。

如果设计得好，一个品牌站的首页、活动页、商品页很多内容都能通过配置完成，而不是每次都找前端改代码。

### 追问：Prompt Engineering 你怎么理解？

比如生成一个商品模块，我不会只说“帮我写组件”，而是会补充字段结构、响应式规则、可配置项、埋点要求、异常状态、代码风格和输出格式。这样 AI 生成的结果更接近可落地代码，也方便后续审查和复用。

## Q8：你理解 CRO 吗？

### 推荐回答

我理解 CRO 不是单纯改 UI，而是围绕用户转化链路做优化。

比如商品页首屏加载速度、价格和优惠信息是否清晰、CTA 是否明显、支付流程是否顺畅、移动端体验是否稳定。

技术上我会关注 LCP、CLS、资源体积、图片懒加载、首屏关键内容、埋点完整性。数据上会结合 GA4/GTM 看页面浏览、加购、发起结账、支付成功这些漏斗节点，再判断优化方向。

### 追问：电商站常见转化漏斗有哪些？

常见链路是：访问页面、浏览商品、点击 CTA、加入购物车、进入结账、完成支付。

如果要定位问题，要看每一层的转化率和流失点。比如商品页访问很多但加购少，可能是卖点、价格、信任背书或 CTA 问题；加购多但支付少，可能是运费、税费、支付方式、结账体验或信任问题。

## Q9：GTM/GA4 埋点你怎么理解？

### 推荐回答

GTM 更偏标签管理和触发规则配置，GA4 更偏事件数据采集和分析。电商站里常见事件包括页面浏览、商品曝光、商品点击、加购、开始结账、支付成功等。

我做前端时会关注两个点：第一是事件是否覆盖完整转化链路，第二是事件参数是否规范，比如商品 ID、商品名称、价格、数量、币种、页面来源、按钮位置等。

埋点不是写完代码就结束，还要在预览或调试模式下验证事件是否触发、参数是否正确、是否重复上报。

## Q12：你对 Headless 架构怎么看？

### 推荐回答

我理解 Headless 是把前端展示层和电商后端能力解耦。比如用 Next.js、Hydrogen 或 Nuxt 做前端，把商品、购物车、结账、内容和用户数据通过 API 接进来。

它的好处是前端体验、性能、SEO 和定制能力更强，适合品牌对页面体验要求高、业务定制较多的场景。

但代价是工程复杂度和维护成本更高，比如接口整合、缓存、部署、预渲染、结账链路、数据一致性都要处理。我的判断是：如果只是快速开一个标准品牌站，主题方案可能更高效；如果品牌体验和性能要求很高，Headless 更有空间。

## 最后 15 分钟速记

### 一句话定位

我不是单点 Shopify 主题熟练工，而是能把电商前端、组件复用、AI Agent 和工程化结合起来，提升多站点交付效率。

### 三个必须讲出来的关键词

- `多站点复用`：基础主题层、品牌配置层、差异扩展层。
- `AI 提效`：需求拆解、组件生成、配置生成、埋点校验、代码审查。
- `电商转化`：性能、SEO、CRO、GTM/GA4、支付物流链路。

### 三个不要踩的坑

- 不要说自己精通 Shopify Plus、Liquid 多年经验。
- 不要把 AI 说成“自动写完所有代码”，要强调人负责质量和验收。
- 不要只讲技术栈，要反复回到批量建站、转化率、维护成本。
