# Day 1：简历熟悉与面试表达恢复

目标：把简历内容从“看得懂”变成“能自然讲出来”。今天不投递，不追求背八股，重点是把自我介绍、项目介绍、核心数据和风险边界练顺。

## 今日完成标准

完成 Day 1 后，需要达到以下标准：

- 能在 1 分钟内讲清楚自己是谁、做过什么、优势是什么。
- 能在 3 分钟内讲清楚 1 个核心项目。
- 能说出 4 个项目各自的背景、职责、动作、结果。
- 能记住简历里的关键数据，不出现前后不一致。
- 能明确哪些能力是强项，哪些能力不能说满。

## 3. 项目卡片

每个项目都按 STAR 讲：背景、任务、行动、结果。

### 项目 1：Reolink Web 跨境商城

一句话定位：

```text
这是一个面向海外用户的智能安防官网商城项目，我主要负责商品展示、活动承接、SEO、多端适配和性能体验优化。
```

STAR 版本：

```text
背景：Reolink 官网商城需要承接新品发布、营销活动和内容运营，对页面展示一致性、SEO 和加载体验要求比较高。

任务：我负责商品详情页、活动落地页、解决方案页等核心页面开发，同时推进组件抽象、多端适配和性能优化工作。

行动：我抽象了商品卡片、卖点模块、媒体展示、规格参数、FAQ 等可复用组件，统一页面搭建标准；针对桌面端、平板端、移动端做响应式适配；性能和 SEO 方面，主要通过图片懒加载、路由级代码分割、语义化标签和 Meta 信息结构化配置来优化。

结果：项目覆盖 10+ 核心页面、8+ 可复用组件、20+ 商品/活动页面 SEO 配置，重点活动页在 Lighthouse / Web Vitals 验证中 LCP 控制在 2.5s 内，也沉淀了活动页交付检查清单和常见联调问题处理手册。
```

可能追问：

- LCP 是怎么测的？
- SEO 具体做了哪些？
- 组件抽象的边界怎么判断？
- 活动页上线前检查清单有哪些？
- 新活动页开发效率提升是怎么来的？

回答抓手：

- 不要只说“优化图片”，要讲 `资源体积、加载时机、首屏元素、字体、接口请求`。
- SEO 要讲 `title、description、结构化内容、语义化标签、可索引内容`。
- 组件抽象要讲 `复用频率、业务稳定性、配置复杂度、维护成本`。

### 项目 2：Reolink 移动端项目

一句话定位：

```text
这是一个智能安防设备管理类 React Native 移动端项目，我负责设备接入、实时预览、消息提醒、录像回放等核心链路开发和体验优化。
```

STAR 版本：

```text
背景：智能安防移动端涉及设备状态、消息通知、视频预览、录像回放等链路，交互状态多，兼容性和稳定性要求比较高。

任务：我负责 React Native 双端核心功能迭代，重点围绕核心链路开发、状态拆分、多机型适配、性能治理和测试体系建设展开。

行动：在状态管理上，我基于 React Hooks 将设备状态、消息状态、页面展示状态拆成 3 类，降低页面耦合；在体验上，针对 20+ 主流机型、不同分辨率和系统版本做适配；在性能上，围绕冷启动速度、页面渲染和长列表卡顿，通过懒加载、状态细粒度拆分、组件复用等方式优化；测试上，围绕设备接入、消息提醒、录像回放补充 20+ Jest、RNTL 和 Detox 用例，并推动接入迭代验证流程。

结果：支持了设备接入、实时预览、消息提醒、录像回放 4 条核心链路的持续迭代；通过多端兼容优化减少高频场景适配问题，并在重点列表场景真机验证中改善滑动卡顿，核心功能回归效率得到提升。
```

可能追问：

- React Native 状态怎么拆？
- 长列表卡顿怎么定位？
- Native Modules 用在什么场景？
- Detox 和 Jest 怎么分工？
- 多机型适配怎么做？
- 多端兼容和重点列表场景优化是怎么验证的？
- 20+ 测试用例主要覆盖哪些场景？

回答抓手：

- 状态拆分讲 `设备状态、消息状态、页面状态`。
- 性能优化讲 `渲染次数、列表虚拟化、图片资源、状态粒度、组件复用`。
- 测试分工讲 `Jest 测函数逻辑，RNTL 测组件交互，Detox 测端到端核心链路`。

### 项目 3：GogoalShop 跨境商城

一句话定位：

```text
这是一个跨境电商商城项目，我主要参与性能优化、工程化治理和业务工具建设。
```

STAR 版本：

```text
背景：商城首页资源体积较大，首屏加载链路比较长，影响用户访问体验和转化。

任务：我负责参与首页性能专项优化，同时推进部分工程化治理和多语言工具建设。

行动：性能方面，我先通过 Lighthouse、Performance 和打包分析定位首屏资源问题，再通过代码分割、依赖替换、Tree Shaking、图片与字体优化处理关键资源链路；工程化方面，完成 npm 到 pnpm 的切换，规范依赖安装和构建流程；工具方面，开发 Chrome 翻译数据抓取插件和 Node 自动写入服务，减少多语言配置的重复操作。

结果：关键 JavaScript 资源从 3MB 降到 600KB，首页 LCP 从 4s 降到 2s 以内，用户跳出率从 30% 降到 20%，依赖安装时间从 10s 降到 5s 以内。
```

可能追问：

- JS 体积从 3MB 到 600KB 具体怎么做？
- LCP 元素是什么？
- Tree Shaking 为什么有时不生效？
- pnpm 为什么更快？
- 多语言工具怎么保证数据准确？

回答抓手：

- 性能不要只讲结果，要讲 `定位 -> 拆解 -> 优化 -> 验证`。
- JS 体积治理讲 `依赖替换、按需加载、动态 import、重复依赖、打包分析`。
- pnpm 讲 `硬链接、内容寻址存储、依赖结构更严格`。

```text
这是团队为了降低第三方应用成本、满足业务定制化诉求而做的 Shopify 自研应用，我独立完成了两款应用开发、部署和上架。

STAR 版本：

背景：业务上需要商品定制和黑名单下单拦截能力，直接使用第三方应用会有持续成本，也不方便做定制。

任务：我负责自定义印号 TT-Options 和黑名单下单拦截 TTBlacklist - Order Guard 两款应用的开发、部署和上架。

行动：前端使用 React、Vite、TypeScript 实现 Shopify Admin 侧配置页面；接口上对接 GraphQL；数据层使用 Prisma 做配置和拦截规则持久化；部署上针对脚手架依赖和服务器环境不兼容问题，引入 Docker 镜像化部署，并编写 Shell 脚本处理镜像打包、版本命名和文件压缩。

结果：两款应用完成开发与上架，替代 2 款第三方付费能力，年节省数万元订阅成本；Docker 部署后部署成功率提升至 100%，Shell 脚本将部署操作从 5 步简化为 1 步，发布效率提升 70%，同时沉淀了 Shopify 应用开发、部署、上架全流程文档。
```

可能追问：

- Shopify 应用上架有哪些坑？

- Docker 解决了什么问题？
- Prisma 用来存什么？
