## 一、Reolink Web 跨境商城

## 1. LCP 是怎么测的？

### 推荐回答

LCP 我一般会结合实验室数据和实际页面验证两种方式看。实验室侧主要用 Lighthouse、Chrome DevTools Performance 面板去跑重点页面，比如商品详情页、活动落地页，看最大内容元素出现的时间以及对应元素是什么；实际页面侧会结合 Web Vitals 指标，重点关注首屏主图、Banner、商品卖点图这些最容易成为 LCP 的元素。

当时 Reolink Web 商城重点活动页的 LCP 主要受首屏大图、字体加载、首屏 JS 执行和接口返回影响，所以优化时不是只压图片，而是一起看资源体积、加载顺序和首屏渲染链路。具体做法包括首屏图片压缩和合理尺寸裁剪、懒加载非首屏图片、路由级代码分割、减少首屏无关 JS、Meta 和结构内容提前渲染。优化后重点活动页 LCP 稳定控制在 2.5s 内。

### 简短版

LCP 我主要用 Lighthouse 和 Chrome Performance 测重点页面，同时结合 Web Vitals 看实际表现。定位时先看最大内容元素是谁，一般是首屏 Banner、商品主图或卖点图，再拆资源体积、加载顺序、JS 执行和接口耗时。Reolink 的重点活动页通过图片优化、非首屏懒加载、路由级代码分割和首屏资源治理，把 LCP 控制在 2.5s 内。

### 可展开点

- 工具：Lighthouse、Chrome DevTools Performance、Web Vitals。
- 关注对象：首屏 Banner、商品主图、核心卖点模块。
- 优化方向：图片体积、资源优先级、首屏 JS、接口耗时、字体加载。

### 技术方案细节

- 测量工具：主要用 `Lighthouse` 看整体 Web Vitals，用 `Chrome DevTools Performance` 看主线程、网络瀑布流和 LCP 元素；如果项目已有 Web Vitals 上报，则结合线上采样数据看趋势，不把它说成固定接入的线上监控系统。
- LCP 定位：先在 Performance 面板中确认 LCP element，常见是首屏 `hero banner`、商品主图或核心卖点图。
- 图片优化：使用 `next/image` 或统一图片组件处理尺寸、压缩、懒加载和响应式图片；首屏图避免懒加载，非首屏图使用 lazy loading。
- 资源优先级：首屏主图可用 `priority` / `preload` 思路提升加载优先级，非首屏图片和视频延后加载。
- JS 优化：通过路由级代码分割、组件懒加载、减少首屏无关依赖，降低首屏 JS 执行阻塞。
- 字体优化：使用 `next/font` 或字体预加载策略，减少字体阻塞和布局抖动。

## 2. SEO 具体做了哪些？

### 推荐回答

SEO 这块我主要做的是基础 SEO 和页面结构优化，目标是让商品页、活动页、解决方案页更容易被搜索引擎理解和收录。具体包括几个方面：

第一是 Meta 信息维护，比如 title、description、canonical、OG 信息，保证不同商品和活动页面有清晰的页面主题。

第二是语义化结构，比如合理使用 h1、h2、section、article 这些标签，让页面内容层级更清楚，避免所有内容都只是 div 堆叠。

第三是内容可索引，核心商品卖点、规格参数、FAQ 等内容尽量以服务端或静态内容方式输出，不把关键内容完全依赖客户端异步渲染。

第四是图片和媒体资源优化，包括 alt 文案、尺寸控制、懒加载非首屏图片，避免图片拖慢页面加载。

当时覆盖了 20+ 商品/活动页面的基础 SEO 配置，配合性能优化一起支撑跨境自然流量获取。

### 简短版

SEO 主要做基础配置和页面结构优化，包括 title、description、canonical、OG 信息、语义化标签、h1/h2 层级、图片 alt、FAQ 和规格参数等可索引内容。同时避免核心内容完全依赖客户端渲染，配合图片和加载性能优化，覆盖了 20+ 商品/活动页面。

### 可展开点

- Meta：title、description、canonical、OG。
- 结构：h1/h2、section、article、FAQ。
- 内容：商品卖点、规格参数、活动信息可索引。
- 性能：SEO 和 LCP 不是割裂的，加载慢会影响搜索体验。

### 技术方案细节

- 页面元信息：在 Next.js 页面中维护 `title`、`description`、`canonical`、`open graph`，保证商品页和活动页主题清晰。
- 语义化结构：页面只保留一个主 `h1`，用 `h2/h3` 表达商品卖点、规格参数、FAQ 层级，避免纯 `div` 堆叠。
- 可索引内容：商品标题、卖点、规格参数、FAQ、活动规则尽量在 SSR/SSG 阶段输出，避免完全依赖客户端接口渲染。
- 图片 SEO：商品图和场景图补充有业务含义的 `alt`，同时控制图片尺寸和格式，避免 SEO 与性能互相冲突。
- 链接结构：活动页、商品页、解决方案页之间保持可点击的内部链接，方便搜索引擎发现和理解页面关系。
- 验收方式：上线前检查 HTML 源码中是否包含核心内容、Meta 是否正确、移动端是否可正常访问。

### 避免这样说

```text
SEO 就是加一些关键词。
```

这样会显得理解很浅。要讲页面结构、内容可索引和性能体验。

## 3. 组件抽象的边界怎么判断？

### 推荐回答

我判断组件是否要抽象，一般不会只看“代码重复”，还会看业务稳定性、复用频率、配置复杂度和维护成本。

在 Reolink Web 商城里，像商品卡片、卖点模块、媒体展示、规格参数、FAQ 这些模块，在商品页、活动页、解决方案页里都会反复出现，而且业务形态相对稳定，所以适合抽成可复用业务组件。

但如果某个模块只是一次活动页的强定制内容，或者运营样式变化非常频繁，我不会一开始就抽得很通用。否则组件配置项会越来越多，后期反而难维护。我的做法通常是先沉淀基础展示和交互能力，把样式、文案、图片、跳转链接这些高频变化点做成配置；对于强定制部分保留页面内实现。

所以组件抽象的边界是：稳定且高频复用的抽出来，变化很大的不要过度抽象。

### 简短版

我不会只因为代码重复就抽组件，会看复用频率、业务稳定性、配置复杂度和维护成本。商品卡片、卖点模块、规格参数、FAQ 这种跨商品页、活动页反复出现，且结构稳定的模块适合抽象；一次性活动样式或变化很大的模块就保留页面内实现，避免配置项膨胀。

### 可展开点

- 适合抽象：商品卡片、卖点模块、媒体展示、规格参数、FAQ。
- 不适合过度抽象：一次性活动区块、强视觉定制模块。
- 判断标准：复用频率、业务稳定性、配置复杂度、维护成本。

### 技术方案细节

- 技术形态：基于 `React + TypeScript` 抽业务组件，用 `props` 或配置对象承接文案、图片、按钮、跳转链接等变化点。
- 类型约束：用 TypeScript interface 定义组件入参，比如商品卡片的 `title`、`price`、`image`、`href`、`tags`。
- 样式复用：用 `TailwindCSS` 或 CSS Module 统一间距、断点、字体层级，避免每个页面重新写一套样式。
- 组合方式：活动页更适合“页面编排 + 组件组合”，而不是把所有活动需求做成一个巨型配置组件。
- 维护边界：如果一个组件配置项越来越多，说明抽象过度，需要拆成更小组件或回到页面内定制。

## 4. 活动页上线前检查清单有哪些？

### 推荐回答

活动页上线前我会按“内容、交互、接口、性能、SEO、兼容、埋点和回滚”几个维度检查。

内容层面，检查商品信息、价格、活动时间、优惠文案、图片素材和跳转链接是否正确。

交互层面，检查按钮、弹窗、轮播、表单、购买跳转这些关键动作是否可用。

接口层面，检查商品库存、价格、活动状态、异常兜底是否正常，避免接口失败时页面空白。

性能层面，主要看首屏资源、图片体积、LCP、非首屏图片懒加载和是否引入了不必要的大依赖。

SEO 层面，检查 title、description、canonical、h1、图片 alt、核心内容是否可索引。

兼容层面，至少看桌面端、平板端、移动端三类视口，以及常见浏览器表现。

最后是埋点和回滚，确认曝光、点击、购买入口这些关键埋点是否正常，如果上线后发现问题，要能快速回滚或降级。

### 简短版

活动页上线前我主要按 8 类检查：内容、交互、接口、性能、SEO、兼容、埋点、回滚。比如商品信息和活动时间是否正确，购买跳转是否可用，接口失败有没有兜底，图片和首屏资源是否过大，title/description/h1/alt 是否完整，三类视口是否适配，关键点击和曝光埋点是否正常。

### 可展开点

- 内容：价格、活动时间、优惠文案、图片、链接。
- 交互：按钮、弹窗、表单、购买跳转。
- 性能：LCP、图片体积、懒加载、大依赖。
- SEO：title、description、canonical、h1、alt。
- 风险控制：埋点、回滚、降级。

### 技术方案细节

- 内容检查：用运营配置表或 CMS 配置对照商品标题、价格、活动时间、优惠文案、跳转链接。
- 交互检查：覆盖 CTA 按钮、购买跳转、弹窗、轮播、表单提交、异常提示。
- 接口检查：用 Network 面板或接口联调工具检查商品、价格、库存、活动状态接口是否正常，并验证接口失败时的兜底 UI。
- 性能检查：用 Lighthouse 和 Performance 检查 LCP、图片体积、首屏 JS、是否误引入大依赖。
- SEO 检查：确认 `title/description/canonical/h1/alt`，以及核心商品内容是否在 HTML 中可见。
- 兼容检查：至少覆盖桌面、平板、移动端 3 类视口，并检查 Safari/Chrome 常见差异。
- 发布检查：确认埋点、灰度、回滚方案，如果活动上线后出现价格或跳转问题，能快速下线或回退。

## 5. 新活动页开发效率提升是怎么来的？

### 推荐回答

新活动页开发效率提升主要来自三件事：组件复用、交付清单和问题沉淀。

第一是组件复用。商品卡片、卖点模块、媒体展示、规格参数、FAQ 这些高频模块抽出来后，新活动页不需要每次从零写页面结构，只需要组合组件和配置内容。

第二是交付清单。活动页通常涉及设计、运营、后端和前端协作，以前容易漏掉 SEO、埋点、移动端适配、活动时间、跳转链接这些细节。沉淀检查清单后，开发和验收可以按同一套标准走，返工会少很多。

第三是常见联调问题沉淀。比如商品价格、库存、活动状态、跳转链接、接口异常兜底这些问题，如果每次都重新排查会浪费时间。整理成处理手册后，后续遇到类似问题可以快速定位。

所以效率提升不是单纯写代码更快，而是减少重复开发、减少遗漏和减少联调返工。

### 简短版

主要来自组件复用、交付清单和联调问题沉淀。高频模块抽成 8+ 业务组件后，新活动页可以通过组合和配置完成；检查清单统一了 SEO、埋点、适配、跳转、活动时间等验收标准；常见联调问题手册减少了接口和配置类问题的重复排查，所以整体交付更稳定。

### 可展开点

- 组件复用减少从零开发。
- 检查清单减少遗漏。
- 联调手册减少重复排查。
- 效率提升不只来自编码速度，也来自返工减少。

### 技术方案细节

- 组件层：商品卡片、卖点模块、规格参数、FAQ、媒体展示组件复用，减少页面重复结构开发。
- 类型层：用 TypeScript 固定组件数据结构，减少运营配置或接口字段变更导致的低级错误。
- 样式层：沉淀统一响应式断点和 Tailwind class 组合，避免每个活动页重新处理移动端样式。
- 流程层：上线检查清单覆盖 SEO、埋点、跳转、接口兜底、响应式、LCP，减少验收遗漏。
- 联调层：把价格、库存、活动状态、跳转链接、接口异常这些高频问题沉淀成排查路径。

## 二、Reolink 移动端项目

## 6. React Native 状态怎么拆？

### 推荐回答

React Native 项目里我不会把所有状态都放在一个大对象里，而是按业务职责拆。Reolink 移动端主要拆成三类：设备状态、消息状态、页面展示状态。

设备状态包括设备在线离线、连接中、预览中、回放中这些和设备生命周期相关的数据；消息状态包括告警消息、未读状态、推送入口状态；页面展示状态包括 loading、弹窗、筛选条件、当前 tab、列表分页等 UI 状态。这样拆的好处是一个状态变化不会引起整个页面无意义重渲染，也方便后续写测试。

技术上主要基于 `React Hooks` 做局部状态和业务状态拆分，比如简单 UI 状态用 `useState`，复杂状态流转用 `useReducer`，跨组件复用逻辑抽成自定义 Hook，避免页面组件里混太多业务细节。

### 技术方案细节

- UI 状态：`useState` 管理 loading、modal、tab、filter。
- 复杂状态：`useReducer` 管理设备连接、预览、回放这类状态机式流转。
- 复用逻辑：用 `useDeviceStatus`、`useMessageList` 这类自定义 Hook 抽离业务逻辑。
- 渲染优化：配合 `useMemo`、`useCallback`、`React.memo` 控制子组件重复渲染。
- 边界控制：页面负责组合，Hook 负责逻辑，基础组件负责展示。

## 7. 长列表卡顿怎么定位？

### 推荐回答

我会先区分是数据量问题、渲染问题，还是图片资源问题。React Native 长列表一般优先检查 `FlatList` 或 `SectionList` 的配置，比如是否使用了 `keyExtractor`、是否设置 `initialNumToRender`、`windowSize`、`maxToRenderPerBatch`，以及 item 组件是否每次都重复渲染。

定位时我会先用 RN 自带性能面板、React DevTools Profiler、Flipper 做初步判断，必要时再结合 Android Studio Profiler 或 Xcode Instruments 看 JS 线程和 UI 线程是否被阻塞。优化上主要是列表虚拟化、item 组件 memo 化、图片懒加载和缓存、减少 inline 函数和对象、避免列表项里做复杂计算。

### 技术方案细节

- 列表组件：优先使用 `FlatList` / `SectionList`，避免直接 `ScrollView` 渲染大量数据。
- 虚拟化参数：`initialNumToRender`、`windowSize`、`maxToRenderPerBatch`、`removeClippedSubviews`。
- Item 优化：`React.memo` 包装列表项，`useCallback` 固定事件函数，`useMemo` 缓存计算结果。
- 图片优化：缩略图优先、懒加载、缓存、避免原图直接进入列表。
- 排查工具：React DevTools Profiler、Flipper、RN Performance Monitor、Android Studio Profiler、Xcode Instruments。

## 8. Native Modules 用在什么场景？

### 推荐回答

Native Modules 主要用于 React Native JS 层无法直接完成，或者用 JS 做性能和能力都不合适的场景。比如访问原生设备能力、接入原生 SDK、处理部分系统级能力、或者需要和已有 Android/iOS 原生模块打通。

在智能安防 App 这类项目里，Native Modules 常见边界会包括设备能力、推送、音视频相关 SDK、系统权限、文件或缓存能力等。我的理解是，不应该什么都写 Native Module，只有当 JS 层能力不足、性能不够，或者必须复用原生 SDK 时才下沉到原生。

### 技术方案细节

- JS 与原生通信：通过 `NativeModules` 暴露原生方法，JS 侧封装统一调用层。
- 事件回传：原生侧通过 event emitter 通知 JS，比如连接状态、播放状态、权限结果。
- 类型约束：TypeScript 为 JS 调用层定义入参和返回值，减少调用错误。
- 失败兜底：原生调用失败要有错误码、超时、重试或降级 UI。
- 使用边界：原生 SDK、系统权限、推送、音视频、文件能力适合下沉；普通业务逻辑不下沉。

## 9. Detox 和 Jest 怎么分工？

### 推荐回答

我会按测试金字塔来分工。`Jest` 更适合测纯函数、状态流转、工具方法；`React Native Testing Library` 更适合测组件渲染和用户交互；`Detox` 更适合测端到端核心链路，比如设备接入、消息提醒、录像回放这种跨页面流程。

不会所有场景都用 Detox，因为 E2E 用例成本高、执行慢、稳定性也更依赖环境。我的做法是用 Jest 和 RNTL 覆盖大部分逻辑和组件，Detox 只覆盖最关键、最容易回归出问题的链路。

### 技术方案细节

- `Jest`：测试 reducer、Hook 逻辑、数据转换、异常分支。
- `React Native Testing Library`：测试组件是否展示正确、按钮点击后状态是否变化。
- `Detox`：测试从进入页面到完成设备接入、消息查看、录像回放的端到端流程。
- Mock 策略：接口、权限、设备状态用 mock 数据隔离不稳定依赖。
- CI 接入：核心用例放进迭代验证流程，避免每次靠人工回归。

## 10. 多机型适配怎么做？

### 推荐回答

多机型适配我会从屏幕尺寸、安全区、系统版本、字体缩放、平台差异几个方面处理。React Native 里不能只按一个设计稿尺寸写死，需要考虑不同屏幕宽高、刘海屏、底部安全区、Android/iOS 默认行为差异。

技术上会用 `SafeAreaView` 或安全区相关库处理顶部和底部区域，用响应式尺寸工具或按比例计算处理间距和图片尺寸；对于平台差异，用 `Platform.select` 做少量差异化处理。上线前至少覆盖 20+ 主流机型和不同系统版本，重点看设备管理、列表浏览、预览操作这些高频场景。

### 技术方案细节

- 安全区：`react-native-safe-area-context` 处理刘海屏和底部区域。
- 响应式：基于屏幕宽度、比例工具或设计系统处理尺寸。
- 平台差异：`Platform.OS` / `Platform.select` 处理 Android/iOS 差异。
- 列表和预览：重点验证横竖屏、长列表、视频预览区域、底部操作区。
- 验收：覆盖小屏、大屏、高分辨率、不同 Android/iOS 版本。

## 11. 崩溃率 0.2% 以下和 55fps 是怎么衡量的？

### 推荐回答

崩溃率这类指标需要以项目实际接入的 crash 统计或版本质量记录为准，通常按版本、机型、系统版本聚合，看核心场景发布后的 crash 趋势。55fps 主要针对列表滑动和高频页面交互，我会用 RN 性能面板、真机调试、Flipper，必要时结合 Android Studio Profiler 或 Xcode Instruments 观察帧率和卡顿情况。

这类指标是团队工程质量结果，我负责的部分是高频链路的兼容修复、渲染优化、测试补充和验证流程接入。

### 技术方案细节

- 崩溃率：线上 crash 平台按版本、设备、系统、页面聚合统计。
- 帧率：RN Performance Monitor、Flipper、Android Studio Profiler、Xcode Instruments。
- 定位维度：JS 线程阻塞、UI 线程掉帧、图片加载、列表渲染、频繁 setState。
- 优化手段：列表虚拟化、状态细粒度拆分、组件 memo、图片缓存、延迟非关键任务。

## 12. 20+ 测试用例主要覆盖哪些场景？

### 推荐回答

20+ 用例主要围绕设备接入、消息提醒、录像回放三类核心链路。设备接入覆盖连接成功、连接失败、权限缺失、loading、重试；消息提醒覆盖消息列表、未读状态、点击跳转、空状态；录像回放覆盖时间选择、列表加载、播放入口、异常提示。

用例类型上，Jest 测状态流转和工具函数，RNTL 测组件交互，Detox 测核心端到端链路。重点不是追求数量，而是覆盖最容易回归、最影响用户体验的路径。

### 技术方案细节

- 设备接入：成功、失败、超时、重试、权限异常。
- 消息提醒：未读、已读、空列表、点击跳转、异常提示。
- 录像回放：时间筛选、列表加载、播放入口、无录像兜底。
- 状态流转：用 Jest 测 reducer / hook。
- 端到端：用 Detox 测关键路径，避免覆盖过多低价值 UI。

## 三、GogoalShop 跨境商城

## 13. JS 体积从 3MB 到 600KB 具体怎么做？

### 推荐回答

我会先用打包分析工具看体积来源，比如哪些依赖最大、是否有重复依赖、哪些页面把非首屏代码打进了首页。优化时主要做了代码分割、依赖替换、Tree Shaking 和首屏资源治理。

具体来说，页面级模块用动态导入延后加载，非首屏组件通过 `next/dynamic` 拆出去；依赖上检查是否引入了完整大包，比如工具库、日期库、图表库，如果只是用少量能力就换成更轻的包或按需引入；再检查包是否支持 ESM 和 Tree Shaking，避免 CommonJS 导致无效代码被打进去。

### 技术方案细节

- 分析工具：`@next/bundle-analyzer`、构建产物分析、`pnpm why` 查重复依赖。
- 代码分割：`next/dynamic`、路由级拆包、非首屏模块延迟加载。
- 依赖治理：大库替换轻量库，按需引入，避免整包导入。
- Tree Shaking：优先使用 ESM 包，检查 `sideEffects`，避免 CommonJS 影响摇树。
- 验证：优化前后对比 bundle size、Lighthouse、Network 下载体积。

## 14. LCP 元素是什么？

### 推荐回答

LCP 元素不是固定的，需要在具体页面里用 Performance 或 Lighthouse 看。商城首页和活动页里，常见 LCP 元素一般是首屏 Banner、商品主图、活动主视觉图，或者首屏最大标题块。

我当时会先确认 LCP element 是图片还是文本。如果是图片，就重点看图片尺寸、格式、CDN、加载优先级、是否被懒加载误伤；如果是文本，就看字体加载、CSS 阻塞和服务端输出时机。

### 技术方案细节

- 图片型 LCP：优化图片格式、尺寸、CDN、`priority/preload`、避免首屏 lazy。
- 文本型 LCP：优化字体加载、CSS 阻塞、SSR/SSG 输出。
- 定位工具：Lighthouse 明细、Performance 面板中的 LCP marker。
- 验证方式：优化前后比较 LCP 时间和 Network 瀑布流。

## 15. Tree Shaking 为什么有时不生效？

### 推荐回答

Tree Shaking 不生效通常有几类原因：依赖不是 ESM，而是 CommonJS；代码有副作用，打包器不敢删；导入方式不对，比如从库入口整包导入；或者 Babel/构建配置把 ESM 转成了 CommonJS。

所以排查时不能只说“我用了 Tree Shaking”，要看依赖包格式、导入方式和构建产物。比如工具函数尽量按需导入，优先选支持 ESM 的包，必要时替换依赖。

### 技术方案细节

- 包格式：ESM 更利于 Tree Shaking，CommonJS 通常效果差。
- 副作用：检查 `package.json` 的 `sideEffects` 配置。
- 导入方式：避免 `import _ from 'lodash'` 这类整包导入，改成按需导入或轻量替代。
- 构建配置：避免提前把 ESM 转成 CommonJS。
- 验证方式：看 bundle analyzer 中目标依赖是否仍被整包打入。

## 16. pnpm 为什么更快？

### 推荐回答

pnpm 快主要是因为它使用内容寻址存储和硬链接机制。相同版本依赖不会在每个项目里重复完整安装，而是存到全局 store，再通过硬链接引用。这样磁盘占用和安装时间都会更好。

另外 pnpm 的 node_modules 结构更严格，可以减少幽灵依赖问题。对团队项目来说，pnpm 的 lockfile 也能让依赖版本更稳定，减少“我本地能跑、别人不行”的情况。

### 技术方案细节

- 内容寻址存储：相同包只保存一份。
- 硬链接：项目中依赖链接到全局 store。
- 严格依赖：避免未声明依赖被误用。
- 锁文件：`pnpm-lock.yaml` 固定版本，提升 CI 和本地一致性。
- 验证：对比 npm 和 pnpm 的 install 时间、node_modules 体积。

## 17. 多语言工具怎么保证数据准确？

### 推荐回答

多语言工具最容易出问题的是 key 对不上、覆盖已有翻译、写入格式错误。所以我会把流程拆成抓取、校验、写入、回滚几个环节。

Chrome 插件负责抓取翻译数据，Node 服务负责结构化写入。写入前会校验 key 是否存在、语言字段是否完整、是否覆盖已有人工翻译；写入时尽量保持 JSON 结构稳定；异常时保留日志和备份，方便回滚。

### 技术方案细节

- 数据结构：统一 `locale -> namespace -> key -> value` 或项目已有结构。
- 写入校验：检查 key 缺失、重复、空值、非法字符。
- 格式化：写入后统一 JSON formatting，减少无意义 diff。
- 安全策略：写入前备份，失败时保留错误日志。
- 验证：抽样检查页面展示和 fallback 文案。

## 四、Shopify 自研应用

## 18. Shopify 应用上架有哪些坑？

### 推荐回答

Shopify 应用上架除了功能开发，还要注意权限、回调地址、隐私合规、Webhook、错误处理和审核材料。常见问题是权限申请过大、OAuth 回调不稳定、卸载后数据处理不符合要求、隐私协议和应用说明不完整。

我的做法是先把应用权限收敛到业务必须范围，再保证 OAuth、安装、卸载、回调地址在生产环境稳定；同时准备隐私协议、使用说明和测试账号，减少审核沟通成本。

### 技术方案细节

- 鉴权：OAuth 安装流程、access token 存储、回调地址配置。
- 权限：只申请业务需要的 Admin API scope。
- Webhook：安装、卸载、订单相关事件需要稳定处理。
- 合规：隐私协议、数据删除、应用说明、测试账号。
- 部署：生产域名、HTTPS、环境变量和回调 URL 要一致。

## 19. GraphQL 查询怎么设计？

### 推荐回答

GraphQL 查询我会遵循“按页面需要取字段”的原则，不一次性取过多数据。比如配置页只需要商品 ID、标题、选项配置，就不取完整商品详情。列表类数据要做分页，复杂字段用 fragment 复用，接口错误要有明确兜底。

这样做的好处是减少接口体积，也让前端数据结构更稳定。

### 技术方案细节

- 精简字段：只查询页面展示和业务逻辑必须字段。
- 变量传参：用 variables 传商品 ID、分页 cursor、筛选条件。
- 分页：使用 cursor-based pagination，避免一次拉取大量数据。
- Fragment：复用商品、订单、配置字段片段。
- 错误处理：处理 GraphQL errors、userErrors、网络异常。

## 20. 黑名单拦截失败怎么兜底？

### 推荐回答

黑名单拦截不能只依赖单点判断。失败兜底我会从规则校验、日志记录、订单标记和人工复核几个方向做。

如果实时拦截失败，至少要在后续订单流程里标记风险订单，比如给订单打 tag、记录命中规则、通知运营或客服复核。这样即使没有在第一时间阻断，也能避免问题订单完全漏掉。

### 技术方案细节

- 规则校验：邮箱、手机号、地址、用户标识等规则统一管理。
- 失败记录：记录请求参数、命中规则、接口返回、时间。
- 订单标记：通过 Shopify Admin API 给订单加 tag 或风险备注。
- 人工复核：高风险订单进入运营复核流程。
- 降级策略：规则服务异常时不让页面空白，要提示或走保守策略。

## 21. Docker 解决了什么问题？

### 推荐回答

Docker 主要解决环境一致性和部署可复现问题。Shopify 应用脚手架和服务器环境可能存在 Node 版本、系统依赖、包管理工具差异，直接部署容易出现本地能跑、服务器不能跑。

引入 Docker 后，把 Node 版本、依赖安装、构建命令、启动方式都固化到镜像里，服务器只需要运行镜像，减少宿主机差异影响。

### 技术方案细节

- 镜像构建：在 `Dockerfile` 中固定 Node 版本、安装依赖、执行 build。
- 环境变量：通过 `.env`、服务器环境变量或 CI/CD 配置注入 Shopify key、secret、database url，避免写死在代码里。
- 运行方式：容器统一暴露端口和启动命令。
- 发布脚本：Shell 脚本串联 build、tag、压缩、上传或部署。
- 收益：减少因宿主机 Node 版本、系统依赖差异导致的部署失败；回滚时可以回到旧镜像版本。

## 22. Prisma 用来存什么？

### 推荐回答

Prisma 主要用于存业务配置和应用状态，比如店铺安装信息、access token、商品定制配置、黑名单规则、订单拦截记录等。它的价值是把数据库模型类型化，前端和服务端处理配置时更稳定。

### 技术方案细节

- 店铺信息：shop domain、token、安装状态。
- 配置数据：商品定制选项、黑名单规则、开关状态。
- 业务记录：命中规则、拦截日志、异常日志。
- 类型安全：Prisma schema 生成类型，减少字段写错。
- 迁移管理：用 migration 管理表结构变化。

## 23. 年节省数万元是怎么估算的？

### 推荐回答

这个数据通常按第三方应用订阅费用估算。比如原来依赖两款付费应用，每款按月订阅费用计算，一年就是月费乘以 12，再加上可能的高级套餐或店铺数量。自研应用上线后替代了这部分能力，所以可以估算年节省数万元。

### 技术方案细节

- 估算公式：第三方应用月费 * 12 * 店铺数量。
- 对比范围：只对比被自研替代的能力，不把无关成本算进去。
- 边界说明：这是订阅成本节省，不等同于 GMV 或利润增长。

## 24. 部署成功率 100% 和发布效率 70% 怎么统计？

### 推荐回答

部署成功率主要看引入 Docker 之后，应用在目标服务器上的构建和启动是否还因为环境问题失败。之前可能会因为 Node 版本、依赖、系统环境导致失败；镜像化后这些问题被隔离。我会按实际发布记录来讲：在我参与的发布周期里，环境类部署失败被消除，所以可以说部署成功率提升到 100%。

发布效率 70% 主要来自操作步骤减少。原来需要手动安装依赖、构建、命名版本、压缩、上传等多个步骤，脚本化后变成一条命令串起来，从 5 步简化为 1 步。这个 70% 更适合解释为“按人工操作步骤和单次发布耗时粗略估算的提效”，不是业务收入类指标。

### 技术方案细节

- 成功率统计：按发布记录统计成功/失败次数，重点排除宿主机环境导致的失败。
- 效率统计：对比脚本化前后的人工步骤数和耗时。
- 自动化内容：镜像打包、版本命名、文件压缩、产物校验。
- 风险控制：脚本失败要输出日志，不吞错误。
