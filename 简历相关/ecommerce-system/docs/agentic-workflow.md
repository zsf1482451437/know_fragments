# AI Agent 研发流程说明

本文档用于说明本项目如何使用 OpenSpec、TDD 和 AI Agent 协作完成需求拆解、实现、验证和复盘。

## OpenSpec 文档映射

- `proposal.md`：说明项目为什么要做，以及它需要覆盖哪些 JD 能力信号。
- `design.md`：记录技术选型、替代方案、风险、取舍和迁移策略。
- `specs/*/spec.md`：把商品、购物车、订单、前端工程、Go 后端和 Agent 流程拆成可验证行为。
- `tasks.md`：驱动实现顺序、任务进度和交付验收。

## TDD 覆盖范围

优先对“容易静默回归”的业务规则采用测试先行：

- 价格计算：小计、优惠、运费和应付金额。
- 库存校验：库存不足时，结算必须失败。
- 订单创建：下单后的商品价格快照必须保持不变。
- 订单流转：非法状态转换必须被拒绝。
- SQLite 事务：订单创建、订单项写入、时间线写入和库存扣减必须同时成功或同时回滚。

## AI Agent 审查清单

- 正确性：规格文档里的需求是否都被实现和测试覆盖。
- 可维护性：前端特性边界、后端分层边界是否清晰，业务规则是否没有散落在 UI 代码里。
- 性能体验：是否具备路由懒加载、过期请求保护、图片懒加载和稳定列表 key。
- 安全与健壮性：接口是否校验请求体，错误信息是否避免暴露内部细节。
- 测试覆盖：价格、库存、订单状态、SQLite 事务等核心规则是否有自动化测试。
- 链路可解释性：每个新增需求都必须补充“用户交互 → API/状态 → 服务层 → 数据库操作”的 Mermaid 链路图。

## 交互到数据库链路图规范

后续每个新增需求都必须同步更新 `docs/user-interaction-db-flow.md`，至少补充：

- 用户从哪个页面或组件触发交互。
- 前端更新了哪些本地状态，或调用了哪个 `apiClient` 方法。
- 请求进入哪个 HTTP API、handler 和 service。
- 经过哪些 domain 规则，例如价格计算、库存校验、状态机流转。
- repository 层执行了哪些数据库读写。
- 涉及写操作时，必须说明事务、回滚、幂等或错误处理策略。

## 技术取舍

- 选择 REST 而不是 GraphQL：更容易展示 HTTP 语义，首个里程碑实现成本也更低。
- 选择 Go 标准库而不是 Web 框架：减少依赖噪音，更突出 `net/http`、中间件、context 和优雅退出等 Go 基础能力。
- 选择“SQLite 优先，预留 PostgreSQL 迁移路径”：本地演示无需额外数据库服务，同时具备 SQL、事务、索引和真实持久化能力。
- 保留 MemoryStore：用于快速单元测试和实现对比，但不作为默认运行时持久化方案。
- 选择本地购物车而不是登录态购物车：首阶段聚焦商品、购物车、结算和订单主链路。

## 面试复盘表达

可以这样概括：

> 这个项目不是让 AI 直接写代码，而是先用 OpenSpec 把目标、设计、行为规格和任务拆清楚，再对价格、库存、订单状态和 SQLite 事务这些高风险规则做 TDD。实现过程中，AI Agent 主要负责按任务推进、补测试、跑验证和更新文档。这样既能提升实现效率，也能保证关键业务行为有可回归的自动化验证。
