# 计算机网络原理梳理：基于阿里云 Ubuntu + VLESS + REALITY 实践

## 目标

- 这份文档不是落地 SOP，而是把这次 `阿里云 Ubuntu + Xray + VLESS + REALITY + v2rayNG / Clash` 的实践过程，拆回到计算机网络原理层面。
- 重点不是记命令，而是理解：
  - 为什么服务端要监听端口
  - 为什么客户端必须知道 `IP + PORT`
  - 为什么 `UUID`、`PUBLIC_KEY`、`SHORT_ID` 缺一不可
  - 为什么 `8443` 能通但 `443` 可能失败
  - 为什么有的客户端能连、有的客户端直接报 `unsupported proxy type: vless`
  - 为什么服务器有 `IPv4` 不代表一定有 `IPv6`

---

## 一张总图

你这次的实际链路，可以先抽象成下面这张图：

```text
手机 v2rayNG / 电脑 Clash
        |
        | 1. 通过公网 IP + PORT 发起 TCP 连接
        v
阿里云公网入口（安全组 / 云网络）
        |
        | 2. 放行 8443/tcp
        v
Ubuntu 宿主机
        |
        | 3. Docker host network 让 Xray 直接监听宿主机端口
        v
Xray 入站（VLESS + REALITY）
        |
        | 4. 校验 UUID / SHORT_ID / SNI / PUBLIC_KEY 对应关系
        v
Xray 出站（freedom）
        |
        | 5. 代理流量转发到真实目标网站
        v
目标网站
```

把它翻译成网络语言，就是 5 层问题：

- 网络可达性：客户端能不能到服务器 `IP`
- 传输层连通性：客户端能不能连到服务器 `PORT`
- 协议兼容性：客户端是否支持 `vless + reality`
- 加密与身份校验：`UUID / PUBLIC_KEY / SHORT_ID / SNI` 是否匹配
- 应用层访问：代理建立后，最终网站能不能打开

---

## 第一层：IP 地址是什么

### 1.1 IP 的作用

`IP` 可以理解为网络中的“地址”。

没有 `IP`，客户端不知道该去找哪台服务器。

在这次实践里，客户端配置里最核心的一项就是：

- `server: YOUR_SERVER_IP`

也就是：

- 手机或电脑，先知道服务器是谁
- 再去尝试连接它

### 1.2 IPv4 和 IPv6 的区别

你这次同时碰到了 `IPv4` 和 `IPv6`。

最重要的理解不是背定义，而是知道：

- `IPv4` 和 `IPv6` 是两套不同的地址体系
- 一台机器可以：
  - 只有 `IPv4`
  - 只有 `IPv6`
  - 同时有两者
  - 表面支持双栈，但实际某一栈没有公网出口

### 1.3 为什么你机器 `IPv4` 能用，不代表 `IPv6` 一定能用

你这次实测里出现过：

- `curl -4` 能返回公网 IP
- `curl -6` 失败
- `ip -6 addr` 没有全局 IPv6
- `ip -6 route` 没有默认路由

这说明：

- 服务器当前具备 `IPv4` 出口
- 但不具备真正可用的公网 `IPv6`

网络原理上，这是因为：

- 是否“支持 IPv6”，不是看控制台写没写双栈
- 而是看系统里是否真的拿到了：
  - 全局 IPv6 地址
  - 默认 IPv6 路由

只要这两项没有到位，`IPv6` 就只是“名义支持”，不是“实际可用”。

## 第二层：端口是什么

### 2.1 为什么有 IP 还不够

`IP` 只能定位到一台机器。

但一台机器上可能同时跑很多服务，比如：

- SSH
- Nginx
- Docker
- Xray

所以还需要 `PORT` 来定位“这台机器上的哪个服务”。

这就是为什么客户端配置里既要有：

- `server`
- `port`

### 2.2 这次为什么是 `8443`

你原本尝试的是 `443`，但容器启动时报：

- `bind: permission denied`

后来改成 `8443` 后恢复正常。

从原理上说：

- `443` 是低位端口，通常和 HTTPS 强绑定
- 某些环境下，即使进程本身没问题，也可能因为权限、策略、宿主机限制、云环境安全能力而绑定失败
- `8443` 是高位端口，语义上常被当作“备用 HTTPS 端口”，更容易快速落地

所以这次不是协议错了，而是：

- 网络服务正常
- 但端口选型在当前环境下不够顺

### 2.3 为什么服务端监听 `8443`，但 `dest` 还是 `:443`

这次文档里一个容易混淆的点是：

- 本机监听端口：`8443`
- `realitySettings.dest`：`www.cloudflare.com:443`

这两个不是一回事。

它们分别对应：

- `8443`：你的服务器真正对外提供服务的入口
- `www.cloudflare.com:443`：REALITY 伪装时参考的目标站点端口

也就是说：

- 客户端连的是你的服务器 `47.x.x.x:8443`
- REALITY 握手时伪装参考的是 `www.cloudflare.com:443`

一个是“你自己开门的门牌号”，一个是“你伪装成别人家的门牌号”。

## 第三层：TCP 是什么

### 3.1 为什么客户端写的是 `type=tcp`

在这次方案里，底层传输用的是 `TCP`。

你可以先把 TCP 理解成：

- 一种“先建立连接，再可靠传输数据”的方式

相比“发了就发了、不保证到达”的方式，TCP 的特点是：

- 面向连接
- 有顺序
- 可靠传输
- 能重传

这也是为什么很多代理协议仍然建立在 TCP 上。

### 3.2 为什么手机能导入配置，不代表 TCP 一定建立成功

配置导入成功，只说明：

- 应用层配置格式没问题

但要真正连通，还要继续经过：

- 能否连到目标 `IP`
- 能否连到目标 `PORT`
- TCP 三次握手是否成功

你这次外部测试 `47.251.26.205:8443` 成功，说明至少：

- 到服务器的网络路径是通的
- TCP 端口也在监听

这就把问题缩小到了更上层：协议与参数。

---

## 第四层：客户端和服务端协议为什么必须一致

### 4.1 为什么有的客户端直接报 `unsupported proxy type: vless`

这是一个典型的“应用层协议不兼容”问题。

含义不是：

- 服务器坏了
- IP 不通了
- 端口没开

而是：

- 客户端程序本身不认识 `vless`

这就像：

- 服务器说普通话
- 客户端只会粤语

两边都正常，但对不上话。

### 4.2 为什么后来换成 `v2rayNG` 就能继续推进

因为 `v2rayNG` 支持：

- `vless`
- `reality`

所以它至少具备理解这套配置的能力。

这里的核心原理是：

- 网络连通性和协议兼容性是两回事
- 先有“客户端支持协议”，才谈得上“能否真正连通”

---

## 第五层：VLESS 是什么

### 5.1 VLESS 在这套链路里的位置

你可以把 `VLESS` 理解为：

- Xray 体系里的一种代理接入协议

它主要负责：

- 定义客户端和服务端如何组织代理连接
- 如何携带客户端身份信息

在这次配置里，最典型的字段是：

- `protocol: vless`
- `id: UUID`

### 5.2 UUID 为什么重要

`UUID` 在这里可以理解为：

- 客户端身份标识

服务端通过它判断：

- 这是不是被允许的客户端

如果客户端 UUID 和服务端配置里的 UUID 不一致，就算：

- IP 对
- 端口对
- 协议也支持

连接仍然无法正常建立。

## 第六层：REALITY 为什么比“普通 TLS”更复杂

### 6.1 REALITY 不是单纯开个 TLS 就结束

这次你用的是：

- `VLESS + REALITY`

它的关键特点是：

- 不要求你自己准备域名证书
- 但要求客户端和服务端在一组更严格的参数上完全一致

简单理解：

- 普通 TLS 更像“拿着自己证件进门”
- REALITY 更像“模仿一个合法场景进门，但双方必须提前对暗号”

### 6.2 这次 REALITY 里最关键的参数有哪些

你这次真正影响连通性的核心参数有 4 类：

- `UUID`
- `PUBLIC_KEY / PRIVATE_KEY`
- `SHORT_ID`
- `SERVER_NAME / SNI`

其中：

- 服务端保存 `PRIVATE_KEY`
- 客户端保存 `PUBLIC_KEY`
- 两边都要知道 `SHORT_ID`
- 两边都要在 `SERVER_NAME / SNI` 上一致

### 6.3 为什么你这次会出现“端口通了但还是连不上”

因为后来定位到：

- 手机里填的 `PUBLIC_KEY`
- 和服务端当前 `PRIVATE_KEY`

根本不是一对。

这就是非常典型的：

- 第 2 层传输是通的
- 第 4 / 5 层应用协议和加密参数不匹配

所以最终会表现为：

- 连接超时
- 握手失败
- 延迟测试失败

但端口探测却是成功的。

### 6.4 为什么轮换密钥后，旧链接要失效

因为 REALITY 的安全性部分建立在这组参数上：

- `UUID`
- `PRIVATE_KEY / PUBLIC_KEY`
- `SHORT_ID`

任何一项换掉，本质上都等于：

- 旧客户端凭据作废
- 新客户端必须同步更新

这就是为什么文档里要专门补一个“一键轮换”小节。

## 第七层：SNI 和伪装目标是什么

### 7.1 什么是 `SNI`

在这次实践里，客户端里有：

- `sni=www.cloudflare.com`

服务端里有：

- `serverNames: ["www.cloudflare.com"]`

你可以先把它理解成：

- 客户端在握手时，对外表现出“我要访问这个域名”

所以 `SNI` 的作用不是 DNS 解析本身，而是：

- 握手时携带“我看起来像要访问谁”

### 7.2 为什么 `SNI` 和 `SERVER_NAME` 必须一致

因为 REALITY 会拿这组信息参与校验。

如果客户端写的是：

- `www.cloudflare.com`

但服务端只接受：

- `www.microsoft.com`

那么两边期望不一致，握手就会失败。

---

## 第八层：为什么 Docker 不等于“自动能上网”

### 8.1 容器只是进程运行方式，不是网络魔法

这次服务端跑在 Docker 里，但网络问题依然要一层层排查。

Docker 只解决：

- 部署方式
- 依赖封装
- 运行隔离

它不替你自动解决：

- 阿里云安全组
- 宿主机端口绑定
- IPv6 默认路由
- 客户端协议兼容

### 8.2 为什么这里用了 `network_mode: host`

因为你这次是单服务节点，目标是尽快落地。

`host` 模式的含义是：

- 容器直接共享宿主机网络栈

好处是：

- 简单
- 少一层端口映射
- 更容易验证“宿主机有没有在监听这个端口”

代价是：

- 容器和宿主机网络隔离更弱

但对个人节点来说，这种取舍通常是合理的。

---

## 第九层：阿里云安全组为什么这么关键

### 9.1 为什么服务端进程正常，客户端还是可能连不上

因为“应用已经启动”不代表“公网一定能访问到”。

链路上至少还有一道关卡：

- 云厂商安全组

如果安全组没放行 `8443/tcp`，那么会出现：

- 容器是 `running`
- 宿主机也在监听
- 但公网连入还是失败

### 9.2 安全组在网络链路里的位置

它可以理解为：

- 云服务器外侧的一层入口防火墙

流量路径是：

```text
客户端
  -> 云网络入口
  -> 安全组检查
  -> 服务器网卡
  -> 宿主机端口
  -> Xray 进程
```

所以：

- 安全组没放开，后面的服务再正常也没用

## 第十层：DNS 在这套方案里是什么角色

### 10.1 这次为什么“客户端配置能连上”和“网站能打开”是两回事

因为连上节点以后，访问网站还需要域名解析。

这就是 DNS 的职责：

- 把域名翻译成 IP 地址

比如你打开：

- `https://www.google.com`

设备不是直接认识 `www.google.com`，而是要先问 DNS：

- 这个域名对应哪个 IP

### 10.2 为什么你在 Clash 配置里看到 `dns:` 一整段

那段配置的作用是：

- 控制客户端如何解析域名
- 减少污染
- 让规则匹配和代理行为更稳定

但在 `v2rayNG` 路径里，这不是最关键的起点。

因为你这次先遇到的核心问题不是 DNS，而是：

- 客户端不支持 `vless`
- `PUBLIC_KEY` 填错

所以排障顺序应该是：

1. 客户端是否支持协议
2. 服务器端口是否可达
3. 参数是否匹配
4. 最后再看 DNS 和目标网站访问

---

## 第十一层：为什么排障一定要分层

这次实践最有价值的，不是最后“连通了”，而是你已经走过了一次完整的分层排障。

可以总结成下面这张表：

| 现象 | 可能层级 | 典型原因 |
|---|---|---|
| `unsupported proxy type: vless` | 客户端协议层 | 客户端不支持 `vless` |
| 端口探测失败 | 网络 / 传输层 | 安全组没放行、端口没监听 |
| 端口通但节点不通 | 加密 / 参数层 | `PUBLIC_KEY`、`SHORT_ID`、`UUID` 不匹配 |
| `curl -4` 通、`curl -6` 不通 | IP 层 | 没有真正可用的公网 IPv6 |
| 配置导入成功但网站打不开 | DNS / 出站 / 规则层 | 解析、路由、目标站点访问异常 |

所以一个好的排障顺序永远是：

1. 先看客户端是否支持协议
2. 再看服务器端口是否可达
3. 再看参数是否一致
4. 再看目标网站访问与 DNS

不要一上来就把所有问题都归结为：

- “服务器坏了”
- “代理坏了”
- “客户端不行”

真正有用的是先定位问题属于哪一层。

## 第十二层：把这次实践映射回计算机网络知识框架

如果把你这次实操，映射到常见网络知识体系，可以这样理解：

### 12.1 网络层

关注：

- `IPv4`
- `IPv6`
- 路由
- 公网可达性

这次对应的问题：

- 服务器有 `IPv4`
- 服务器当前没有真正可用的 `IPv6`

### 12.2 传输层

关注：

- `TCP`
- 端口
- 监听
- 建连

这次对应的问题：

- `8443` 监听成功
- `443` 在当前环境下绑定失败

### 12.3 应用层

关注：

- `VLESS`
- `REALITY`
- `DNS`
- 客户端兼容性

这次对应的问题：

- 有些客户端不支持 `vless`
- `PUBLIC_KEY` 填错会导致握手失败
- `DNS` 是后续访问网站阶段才重点参与

### 12.4 工程实现层

关注：

- Docker
- 安全组
- 宿主机网络
- 配置管理

这次对应的问题：

- Docker 只是运行方式
- 安全组是公网入口
- `network_mode: host` 简化了网络路径
- 轮换密钥是运维安全的一部分

## 最后记住这几个核心句子

- `IP` 用来找到机器，`PORT` 用来找到机器上的服务。
- 客户端支持协议，只代表“会说这种话”；不代表已经连通。
- 端口能连通，只代表 TCP 路径通；不代表 `REALITY` 参数一定对。
- `UUID`、`PUBLIC_KEY`、`SHORT_ID`、`SNI` 不一致，都会导致握手失败。
- `IPv4` 可用不代表 `IPv6` 也可用，必须看系统里有没有全局地址和默认路由。
- Docker 解决的是部署问题，不会自动解决网络问题。
- 安全组没放行，服务端再正常，公网也进不来。
- 排障时一定要分层，不要把所有故障混成一个问题。
