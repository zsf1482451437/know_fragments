# iPhone Calendar Flutter Design Spec

## 1. 背景与目标

本项目的目标不是做一个“像日历的 Flutter Demo”，而是基于 Flutter Cupertino 生态，构建一套 **iPhone 优先**、**尽量全量复刻 iPhone 日历主入口** 的日历应用。

当前项目里已经存在一个苹果风格月视图原型，但它仍然停留在“单页演示”层面，缺少以下关键能力：

- 没有整套 App 的主入口与页面层级
- 没有统一的数据模型与状态组织
- 没有为后续高保真页面打好可扩展结构
- 没有明确阶段性交付边界，容易继续堆代码

因此，本次设计的核心目标是：

1. 先搭出完整 App 框架，而不是只优化一个首页
2. 以 `月视图 + 底部日程卡片` 作为第一体验锚点
3. 让后续 `日视图 / 列表 / 搜索 / 收件箱 / 日历管理 / 事件编辑流` 能在同一套结构下自然扩展
4. 优先还原 iPhone 日历的视觉层级、入口关系、切换逻辑与交互气质

非目标：

- 第一阶段不接入系统日历 API
- 第一阶段不追求真实账户同步、邀请邮件解析、通知中心深度集成
- 第一阶段不做 Android 风格适配优先级，平台策略明确为 iPhone 优先

## 2. 用户已确认约束

- 目标：完美复刻苹果手机上的日历 App
- 技术栈：Flutter / Cupertino 生态优先
- 产品策略：先搭完整 App 框架，再逐页补齐高保真
- 核心首页：`月视图 + 底部日程卡片`
- 入口范围：尽量全量复刻 iPhone 日历现有主入口
- 平台优先级：`iPhone 优先`
- 质量要求：不能继续生成“低质量原型堆叠代码”，需要结构清晰、可持续演进

## 3. 方案比较与决策

### 方案 A：继续在单文件里快速扩展

特点：

- 直接继续扩写现有 `lib/main.dart`
- 短期见效快
- 不需要立刻拆结构

问题：

- 文件会迅速失控
- 共享状态难维护
- 后续很难同时做高保真页面与交互
- 与“整套 App 复刻”目标不匹配

### 方案 B：按页面拆文件

特点：

- 以 `month_page.dart`、`day_page.dart` 等方式拆分
- 比单文件更好维护
- 适合中等复杂度应用

问题：

- 共享模型、视觉 token、状态流仍可能散落
- 容易形成“页面分散、系统感不统一”的问题

### 方案 C：按领域模型 + 视图壳层拆分

特点：

- 先建立 iPhone 风格应用壳层
- 再围绕共享日历领域模型构建各页面模块
- 视图、状态、模型、视觉规范职责清晰

优势：

- 最适合“全量复刻主入口”的长期目标
- 支持月页先做透，同时不阻碍其他页面接入
- 便于后续做高保真动画、页面切换与数据接入

最终决策：**采用方案 C**

## 4. 产品信息架构

本项目第一阶段需要具备以下 6 个主体验模块：

1. `月视图`
2. `日视图`
3. `列表视图`
4. `搜索`
5. `收件箱`
6. `日历管理`

此外还需要一个贯穿所有模块的 `事件编辑流`。

### 4.1 月视图

定位：

- 默认首页
- 整套 App 的体验锚点
- 最优先高保真还原的页面

核心组成：

- 顶部导航
- 月份标题与切月操作
- 顶部周条
- 月历网格
- 底部当日日程卡片（Agenda Sheet）
- 全局新建事件入口

### 4.2 日视图

定位：

- 查看单日详细日程
- 展示全天区、小时轴、当前时间线、事件块布局

### 4.3 列表视图

定位：

- 以时间顺序连续浏览事件
- 强调长列表扫读效率

### 4.4 搜索

定位：

- 搜索事件、地点、日历源与文本内容
- 点击结果后跳回对应日期与事件上下文

### 4.5 收件箱

定位：

- 承接邀请、待响应事项与事件更新
- 第一阶段以高保真结构与状态流为主，不接真实账户体系

### 4.6 日历管理

定位：

- 管理日历显隐、颜色、分组与默认显示范围
- 对应月页左上 `Calendars` 入口

### 4.7 事件编辑流

定位：

- 新建和编辑事件
- 是全局能力，不属于单一页面

第一阶段至少包含字段：

- 标题
- 全天
- 开始时间
- 结束时间
- 地点
- 备注
- 所属日历

## 5. 页面关系与主导航原则

### 5.1 页面关系

- App 启动后默认进入月视图
- 在月视图选中某一天，会同步更新底部 Agenda Sheet
- 点击某条事件进入事件详情或编辑页
- 点击左上 `Calendars` 进入日历管理页
- 搜索结果可跳转回指定日期和事件
- 新建事件可从月页、日页、列表页进入

### 5.2 导航原则

本项目不应简单做成“Flutter 通用底部 Tab App”。

导航设计要遵循以下原则：

- 优先还原 iPhone 日历的信息层级，而不是追求技术实现上的方便
- 主入口切换要自然、克制、系统感强
- 返回路径要清晰，避免多层嵌套导航造成迷失
- 页面间跳转时要保持日期上下文一致

## 6. 技术架构设计

### 6.1 总体结构

建议采用以下分层：

- `AppShell`：负责应用壳层、主入口容器、全局跳转与全局弹层
- `Shared Domain`：负责共享数据模型与时间语义
- `State Layer`：负责当前日期、当前视图、事件集合、可见日历等状态
- `Feature Modules`：负责各个页面模块与可复用视图组件
- `UI Tokens / Styles`：负责 Cupertino 风格下的视觉常量与局部样式规范

### 6.2 核心模块

#### AppShell

职责：

- 管理整个应用的视图入口
- 容纳全局页面切换
- 统一挂载新建事件流与全局弹层

#### CalendarStore

职责：

- 统一维护全局状态
- 为页面提供单一可信数据源

第一阶段至少管理以下状态：

- `selectedDate`
- `activeView`
- `events`
- `calendarSources`
- `visibleCalendars`
- `searchQuery`
- `searchResults`
- `inboxItems`
- `eventDraft`

#### MonthModule

推荐继续拆成：

- `MonthHeader`
- `WeekStrip`
- `MonthGrid`
- `AgendaSheet`

#### DayModule

推荐继续拆成：

- `DayHeader`
- `AllDaySection`
- `TimelineHours`
- `TimelineEventLayer`

#### ListModule

职责：

- 以日期分组展示事件流
- 提供跳转到日期与事件的能力

#### SearchModule

职责：

- 维护搜索输入、搜索结果与跳转逻辑

#### InboxModule

职责：

- 维护邀请与待处理项的 UI 展示与响应状态

#### CalendarsModule

职责：

- 展示日历源
- 管理显隐、颜色与分组

#### EventEditorModule

职责：

- 新建/编辑事件
- 编辑草稿与保存提交

## 7. 共享领域模型

第一阶段至少抽离以下模型：

### CalendarEvent

表示日程事件，建议包含：

- `id`
- `title`
- `startAt`
- `endAt`
- `allDay`
- `calendarId`
- `location`
- `notes`
- `isInvitation`
- `attendeeStatus`

### CalendarSource

表示一个日历源，建议包含：

- `id`
- `name`
- `color`
- `visible`
- `group`

### InboxItem

表示收件箱项，建议包含：

- `id`
- `eventId`
- `title`
- `summary`
- `status`
- `receivedAt`

### SearchResult

表示搜索结果，建议包含：

- `id`
- `eventId`
- `matchedText`
- `date`
- `subtitle`

### EventDraft

表示编辑中的事件草稿，建议包含：

- `title`
- `allDay`
- `startAt`
- `endAt`
- `location`
- `notes`
- `calendarId`

## 8. 状态流设计

### 8.1 日期状态

`selectedDate` 是整套应用的核心状态。

它会被以下入口共同驱动：

- 月视图网格点击
- 顶部周条切换
- 日视图跳转
- 列表项跳转
- 搜索结果跳转

所有日期驱动页面都必须围绕这个状态联动更新。

### 8.2 视图状态

`activeView` 决定当前正在展示的主模块：

- `month`
- `day`
- `list`
- `search`
- `inbox`
- `calendars`

### 8.3 日历过滤状态

`visibleCalendars` 控制哪些日历源参与展示。

受影响模块：

- 月视图
- 日视图
- 列表视图
- 搜索结果

### 8.4 编辑状态

`eventDraft` 用于新建/编辑流程。

编辑页保存后：

- 先回写事件集合
- 再刷新当前日期相关视图
- 如有必要更新列表与搜索索引

### 8.5 搜索状态

搜索模块维护：

- `searchQuery`
- `searchResults`

点击搜索结果时需要：

1. 定位目标日期
2. 更新 `selectedDate`
3. 跳转到合适视图
4. 高亮或聚焦目标事件

### 8.6 收件箱状态

收件箱项本身独立存在，但其处理结果可能反向更新事件数据。

例如：

- 接受邀请后生成对应事件
- 拒绝邀请后更新状态
- 暂定后更新事件参与人状态

## 9. 视觉与交互原则

### 9.1 视觉原则

- 以 `Cupertino` 风格为主，不混入明显 Material 风格痕迹
- 强调留白、信息层级、轻量投影与系统感圆角
- 控制颜色使用密度，重点使用系统红、系统蓝、灰阶
- 动效以短促、克制、可预期为主

### 9.2 月页交互原则

- 月份切换要自然
- 选中日期要显式高亮
- 顶部周条与月网格要保持联动
- 底部 Agenda Sheet 要有明显的层级感，但不能喧宾夺主

### 9.3 事件编辑交互原则

- 更接近 iOS 设置页和编辑页的混合体验
- 字段区块清晰，避免 Flutter 表单风味过重
- 第一阶段先保证结构与系统感，真实验证逻辑可以逐步补强

### 9.4 日历管理交互原则

- 勾选显隐必须直观
- 颜色点与名称分层清晰
- 从月页进入和返回的路径简单明确

## 10. 错误处理与空态策略

虽然第一阶段主要是 UI 和本地状态，但仍需定义基础错误与空态行为：

- 无日程日期：Agenda Sheet 显示“没有日程”空态
- 无搜索结果：搜索页显示空结果提示
- 无收件箱项：收件箱显示已清空状态
- 所有日历被隐藏：相关页面显示空列表并提供返回日历管理页的提示
- 编辑取消：安全丢弃 `eventDraft`
- 编辑保存失败：先以内存场景处理，出现异常时显示轻量提示，不做复杂恢复逻辑

## 11. 测试策略

本项目必须避免“只有界面，没有可验证行为”的低质量产物。

### 11.1 第一阶段测试重点

至少覆盖：

- 应用壳层能够启动
- 主入口能够切换
- 月页日期选择能够联动 Agenda Sheet
- 新建事件后当前日期能够展示新事件
- 日历显隐过滤会影响月页与列表

### 11.2 测试层次

- `Widget Test`：覆盖页面骨架与关键交互
- `Store / Model Test`：覆盖状态更新逻辑
- `Focused UI behavior test`：覆盖月页日期联动、编辑保存、视图切换

### 11.3 不做的测试

第一阶段不做低价值像素快照测试，不做大面积纯样式断言。

重点是：

- 行为正确
- 状态联动正确
- 回归可控

## 13. 实施边界与重构要求

当前 `lib/main.dart` 已承载了一个原型页面，但后续不能继续把整套应用堆在这一个文件里。

实施阶段需要遵守以下要求：

- 尽快拆出应用壳层与月页模块
- 把数据模型从页面代码里抽离
- 把测试围绕关键行为补起来
- 尽量避免一次性引入过多第三方依赖，优先用 Flutter 自身能力完成第一版

## 14. 验收标准

当第一阶段完成时，应满足以下标准：

- 用户能感知到这不是普通 Flutter 日历 Demo，而是明确朝 iPhone Calendar 复刻方向前进的产品骨架
- 月视图具备明显的苹果原生气质
- 整体已经具备完整 App 框架，而不是只有首页
- 页面与状态结构可持续扩展
- 关键交互已有自动化验证

## 15. 最终结论

本项目采用 **iPhone 优先的 Flutter Cupertino 复刻方案**。

实施策略是：

- 先搭完整 App 框架
- 以月视图与底部日程卡片作为首个高保真锚点
- 基于 `AppShell + CalendarStore + Feature Modules + Shared Domain` 组织代码
- 分阶段补齐 `日视图 / 列表 / 搜索 / 收件箱 / 日历管理 / 事件编辑流`

这份设计的价值在于，它不仅定义了“要做什么页面”，更定义了“如何避免再次产出低质量、不可演进的原型代码”。
