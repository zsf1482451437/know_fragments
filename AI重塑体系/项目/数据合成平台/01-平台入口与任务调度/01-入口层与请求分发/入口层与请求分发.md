# 入口层与请求分发

## 模块定位

这一层对应项目中的 `controller/`，核心职责是把外部 HTTP 请求转成平台内部可执行的任务。

它不负责真正的代码生成，也不负责沙箱、模型、测试这些重活，而是承担三个动作：

1. 暴露接口。
2. 把请求映射到具体 pipeline。
3. 把结果查询接口暴露出去。

---

## 关键文件

- `controller/proxy.ts`
- `controller/synthesis-config.ts`

其中主链路几乎都从 `controller/proxy.ts` 进入。

---

## 详细过程

### 1. 接收执行请求

客户端向：

- `POST /api/proxy/execute`

发送请求，请求体中最重要的字段是：

- `method`：指定要走哪条 pipeline
- `question`：任务输入，通常是一个 JSON 字符串
- `uuid`：任务唯一标识，用于后续轮询结果
- `biz_id / biz_sub_id / dag_runtime_id`：链路上下文信息

这一层不直接理解 `question` 的业务内容，只关心它要交给哪条 pipeline。

### 2. 注册可用 pipeline

在每次执行前，`ProxyController.execute()` 会把当前系统里注入进来的 pipeline 全部注册到 `ProxyService`。

这种写法的含义是：

- Controller 维护“有哪些可执行产线”
- `ProxyService` 维护“如何按名字找到这些产线”

这里实际完成了一次运行时路由表构建。

### 3. 选择目标 pipeline

`method` 可能是：

- `CommonFrameworkV1`
- `TraeHtmlRubrics`
- `SeedSpExpert`

如果 `method` 中带点号，平台会拆成：

- `mainMethod`
- `subMethod`

主方法用于选择 pipeline，子方法用于在 pipeline 内做进一步分支。

### 4. 异步触发执行

Controller 只是调用 `proxyService.execute(body)`，随后直接返回空对象。

这说明接口层默认采用的是“提交任务 + 异步轮询”的交互方式，而不是同步阻塞等待整个代码生成完成。

### 5. 提供结果查询接口

客户端后续通过：

- `POST /api/proxy/get_synthesis_result`

拿 `uuid` 查询结果。

这一步并不重新执行任务，而是去 Redis 里读缓存好的执行状态和最终结果。

---

## 入口层的职责边界

入口层只做“接”和“转”，不做：

- `question` 结构解析细节
- 模型调用
- 沙箱申请
- Git clone
- 工具执行
- 构建与测试

这些能力都下沉到 `service/proxy/` 和更深的层里。

---

## 这一层的价值

这一层虽然逻辑不复杂，但作用很关键：

- 它把外部调用协议稳定下来
- 它屏蔽了内部 pipeline 的复杂性
- 它把平台从“脚本集合”变成了“可服务化访问的系统”

如果把整个数据合成平台看成一条流水线，这一层就是流水线的进料口和状态查询口。
