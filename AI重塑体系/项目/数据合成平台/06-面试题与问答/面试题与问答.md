# 面试题与问答

## 文档定位

这份文档用于收录“数据合成平台”项目在面试场景下最容易被问到的问题，并给出可直接复述、可继续展开的参考答案。

适用场景：

- 项目介绍准备
- 架构设计追问
- 平台流程追问
- 技术取舍追问
- 稳定性与质量保障追问

---

## 一、项目总览类

### 1. 这个项目是做什么的？

答：

这是一个面向代码与产物生成场景的数据合成平台。它接收一份模板 JSON 作为任务输入，里面会描述目标场景、技术栈、模型配置、沙箱配置、Git 配置、知识集合和生命周期节点。平台会根据这些配置选择对应 pipeline，在沙箱中驱动模型和工具协同工作，最终生成代码、运行校验、执行测试、提交 Git，并把结果持久化给外部系统读取。

一句话概括：它不是单纯调用大模型出一段代码，而是把“模型生成 + 工具执行 + 环境准备 + 校验测试 + 结果沉淀”组织成一条可重复执行的自动化流水线。

### 2. 这个项目的核心价值是什么？

答：

核心价值有三层：

- 第一层是把模型能力工程化。模型不再只是回答问题，而是被放进真实代码仓库、真实沙箱和真实构建链路里执行任务。
- 第二层是把任务模板化。平台输入不是自然语言随便描述，而是结构化模板 JSON，这让任务更稳定、可配置、可复用。
- 第三层是把结果验证化。平台不会只相信模型说“完成了”，而是通过 build、analyze、test、monitor、signal 等机制判断任务是否真的可交付。

### 3. 这个项目和普通 AI Coding Agent 的区别是什么？

答：

普通 AI Coding Agent 更像一个通用助手，强调交互式编程；这个平台更像一个可配置的批处理执行框架，强调规模化、标准化和结果可评估。

主要区别有：

- 输入不同：这里的输入是模板 JSON，而不是纯聊天。
- 执行方式不同：这里会走固定 pipeline 和生命周期节点。
- 环境不同：这里默认运行在沙箱和 Git 仓库里。
- 结果判定不同：这里有 checker、测试、监控和评分体系。
- 产物沉淀不同：这里会写回 Redis，并可提交到目标分支。

## 二、架构与模块类

### 4. 你会怎么概括这个项目的整体架构？

答：

我会把它概括成五层：

- 入口与调度层：负责接收请求、分发任务、读写执行状态。
- 通用执行框架层：负责把任务组织成状态初始化、主循环、验证、收尾等生命周期。
- 基础设施层：负责提供模型调用、沙箱、RDS、工具适配等底层能力。
- 质量保障层：负责监控、评分、构建校验和测试闭环。
- 案例与配置层：负责把抽象框架映射到真实模板 JSON 和具体任务。

如果从调用路径看，就是：

请求进入 -> pipeline 选择 -> 框架初始化 -> 环境准备 -> 模型/工具循环 -> 校验测试 -> Git 提交 -> Redis 持久化结果

### 5. `controller/proxy.ts` 和 `service/proxy/index.ts` 分别负责什么？

答：

`controller/proxy.ts` 负责 HTTP 入口，它处理接口暴露、参数接收、pipeline 注册和结果查询入口。

`service/proxy/index.ts` 负责真正的任务调度，它会解析 `question`、定位 `mainMethod.subMethod`、调用目标 pipeline 的 `process()`，并在执行前后把状态和结果写入 Redis。

可以理解成：

- `controller` 解决“请求怎么进来”
- `ProxyService` 解决“任务怎么真正跑起来”

### 6. 为什么项目里需要 `CommonFrameworkV1` 这种通用执行框架？

答：

因为平台要支持的不是单一任务，而是一批“同构但配置不同”的任务。如果每种任务都写一条独立逻辑，代码会迅速碎片化，扩展成本很高。

`CommonFrameworkV1` 的价值在于把共性流程抽象出来：

- 状态初始化
- 知识注册
- 节点注册
- 沙箱与 Git 初始化
- 模型 / 工具主循环
- checker / test / beforeFinish 等后处理

这样不同任务只需要通过模板 JSON 去配置 `knowledge`、`steps`、`sandbox`、`git`、`modelCall`，就能复用同一套执行骨架。

## 三、执行流程类

### 7. 给你一份模板 JSON，这个平台会怎么执行？

答：

可以分成九步：

1. 入口层接收请求，并根据 `method` 选择 pipeline。
2. `ProxyService` 解析模板 JSON，建立执行上下文。
3. `CommonFrameworkV1` 初始化 `state`，注册 `knowledge` 和 `nodes`。
4. 按模板申请沙箱，初始化 Git 仓库和工作目录。
5. 组合技术栈、知识、目录结构和协议说明，构造模型上下文。
6. 进入模型 / 工具循环，模型输出工具调用，平台在沙箱里执行并回填结果。
7. 模型调用 `finish` 后，继续走启动、构建、checker、测试生成和测试执行。
8. 执行截图、上传、Git commit / push 等收尾动作。
9. 把最终结果写入 Redis，供外部轮询读取。

### 8. 平台为什么要设计“节点生命周期”？

答：

因为任务不是单次模型调用，而是分阶段执行的。节点生命周期解决的是“在什么时机做什么动作”的问题。

例如：

- `generalizePrompt` 负责前置改造需求
- `beforeEnterTrace` 负责进入主循环前补充约束
- `afterModelCall` 负责在模型调用后做监控或修复
- `commonChecker` 负责统一质量检查
- `beforeFinish` 负责最后的交付动作

这种设计带来三个好处：

- 可编排：不同任务可以通过 `steps` 自定义流程
- 可扩展：新增节点不需要改主框架
- 可观测：每个阶段都可以挂监控、校验和统计

### 9. 模型和工具是怎么协同工作的？

答：

框架会先把系统提示、用户需求、知识片段、工具协议和仓库上下文拼成对话，然后调用模型。模型输出的不是普通文本，而是带协议的工具调用意图，比如 NFC、XML 或 JSON 结构。平台解析这些结构后，会映射到真实工具能力，例如读文件、写文件、执行命令、搜索文件等。

工具执行完成后，平台把结果再追加回对话，让模型基于最新环境继续决策。这个过程会循环进行，直到模型调用 `finish`。

本质上这是一个“模型负责决策，平台负责执行，环境负责提供反馈”的闭环。

## 四、沙箱、Git 与环境类

### 10. 为什么一定要在沙箱里执行，而不是只让模型生成代码文本？

答：

因为很多任务只有在真实环境里才能验证真假。比如：

- 项目能不能安装依赖
- 代码能不能 build
- 测试能不能跑通
- 服务能不能启动
- 仓库结构是不是符合预期

如果没有沙箱，模型只能“猜测自己是对的”；有了沙箱，平台才能把生成过程变成真实执行过程。

### 11. Git 在这个平台里承担什么角色？

答：

Git 在这里不是简单存代码，而是交付闭环的一部分。

它至少承担三件事：

- 提供真实仓库上下文，让模型理解当前项目结构
- 承载最终产物，使结果可以沉淀到分支中
- 让任务结果具备可追溯性，方便复盘、验收和回滚

另外平台会改写目标分支名，把任务和 trace 关联起来，避免直接污染原始分支。

## 五、质量保障类

### 12. 平台怎么判断一个任务到底是不是“完成了”？

答：

平台不会只看模型有没有调用 `finish`。`finish` 只代表模型认为自己做完了，但平台还会继续做后验验证。

常见验证包括：

- 构建检查，比如 `flutter build`
- 静态分析，比如 `dart analyze`
- 测试生成和测试执行，比如 `flutter test`
- 监控与 trace 汇总
- signal 分数计算和 `pass_rate` 汇总

只有这些结果整体可接受，平台才会把任务视为高质量完成。

### 13. `enableMonitor` 和 `checkSignal` 分别解决什么问题？

答：

`enableMonitor` 解决“过程是否可观测”的问题，它记录模型调用、工具调用、重试次数、trace summary 等运行信息。

`checkSignal` 解决“结果质量是否可量化”的问题，它把构建、分析、测试等检查结果转成统一分数，最后再汇总出 `pass_rate`。

一个偏过程，一个偏结果，两者一起才能既看到“怎么做的”，也看到“做得怎么样”。

### 14. 为什么测试要拆成 `genUnitTest` 和 `runUnitTest` 两个阶段？

答：

因为“生成测试”和“正式评估”本质上不是一回事。

`genUnitTest` 更像探索和生成阶段，它会根据仓库现状生成测试脚本，运行失败时还可能把错误反馈给模型继续修复。

`runUnitTest` 更像正式评估阶段，它负责把最终测试脚本执行一遍，并把测试结果接入统一评分体系。

拆开的好处是：

- 探索性修复不会污染正式评估语义
- 测试生成逻辑和评分逻辑解耦
- 平台更容易复用测试执行阶段

## 六、设计取舍类

### 15. 为什么这个平台选择“模板 JSON + pipeline”的方式，而不是完全自由对话？

答：

因为平台的目标是规模化生产，而不是单次高自由度聊天。

模板 JSON 的优势在于：

- 任务输入标准化
- 执行过程可配置
- 结果更可复现
- 更方便做批量跑数和质量统计

自由对话适合探索，模板配置适合生产。这个平台显然更偏生产型系统。

### 16. 这个项目最关键的技术难点是什么？

答：

我认为有三个关键难点：

- 第一是把模型输出稳定地映射成工具协议，并在多轮调用中保持上下文一致性。
- 第二是把不确定的模型生成过程嵌入到确定性的工程流水线里，比如沙箱、构建、测试、Git 提交。
- 第三是建立一套可观测、可评分、可复盘的质量体系，让平台结果可以真正比较和迭代。

换句话说，难点不只是“怎么生成代码”，而是“怎么把代码生成这件事做成工程系统”。

### 17. 如果让你继续优化这个平台，你会优先做什么？

答：

我会优先做四件事：

- 增强节点级失败归因，把失败更精确定位到具体阶段和具体工具调用。
- 完善质量信号体系，让不同语言和框架下的评分更可比。
- 提升模板配置治理能力，比如模板版本化、兼容策略和静态校验。
- 加强案例沉淀，把成功与失败任务都转成可检索知识，反哺后续执行。

这四件事分别对应可运维性、可评估性、可扩展性和可学习性。

## 七、面试回答建议

### 18. 面试时应该怎么讲这个项目？

答：

建议按下面顺序讲：

1. 先讲项目定位，它是面向代码生成任务的数据合成平台。
2. 再讲主链路，说明模板 JSON 如何变成一次真实执行。
3. 再讲核心抽象，也就是 pipeline、CommonFrameworkV1、knowledge、steps、sandbox、checker。
4. 再讲质量保障，强调平台不是只生成，而是会验证、测试和评分。
5. 最后讲你自己参与的部分，比如文档梳理、模块拆解、流程抽象、稳定性优化或某个具体节点改造。

这样讲的好处是从业务目标到技术实现再到个人贡献，层次会比较清楚。

### 19. 如果面试官让你一句话总结这个项目，怎么说？

答：

我会说：这是一个把大模型代码生成能力放进真实工程环境中执行、验证、评分并沉淀结果的自动化数据合成平台。
