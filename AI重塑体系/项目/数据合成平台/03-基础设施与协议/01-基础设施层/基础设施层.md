# 基础设施层

## 模块定位

这一层对应 `service/infra/`，是整个平台的能力底座。

平台上层负责决定“做什么”，基础设施层负责提供“在哪做、怎么做、通过什么依赖做”。

---

## 组成模块

主要包括：

- `service/infra/llm/`
- `service/infra/sandbox/`
- `service/infra/rds/`
- `service/infra/workflow/`
- `service/infra/index.ts`

其中和主执行链最紧密的是 LLM 与 sandbox。

---

## 详细过程

### 1. `InfraService` 聚合底座能力

`service/infra/index.ts` 会把各类基础设施组合成：

- `llm`
- `sandbox`
- `rds`

上层服务只需要注入一个 `InfraService`，就能统一访问这些能力。

这降低了上层对具体实现细节的耦合。

### 2. LLM 子系统

`service/infra/llm/` 负责对接不同模型来源。

它不是把不同模型直接暴露给业务层，而是先统一封装一层调用接口。这样在 `CommonFrameworkV1` 里，模型来源可以通过配置切换：

- `claude`
- `gpt`
- `marscode`
- `gemini`
- `seed`
- `seed-psm`
- `glm`

执行过程通常是：

1. 上层传入 prompt 和模型配置
2. LLM service 根据 `source` 选择具体实现
3. 向对应模型服务发请求
4. 返回统一结构的 `content/thinking/logId` 等字段

### 3. Sandbox 子系统

沙箱是平台最重要的运行环境依赖。

它解决的是：

- 代码修改在哪里发生
- 命令在哪里执行
- 服务在哪里启动
- 构建与测试在哪里跑

`CommonFrameworkV1` 会通过 sandbox SDK 申请：

- Mars sandbox
- 或 VKE sandbox

执行过程一般是：

1. 根据 `sourceToken` 和 `source` 选择沙箱来源
2. 根据 `image/spec/ttl` 申请会话
3. 获得 session 对象
4. 后续所有命令和工具都在这个 session 中执行
5. 最终释放 session

### 4. RDS 子系统

`service/infra/rds/` 提供数据库访问能力。

在当前主执行链中，它不是最显眼的部分，但它承担的是平台级数据落库、配置查询、状态维护等底层能力。

### 5. workflow 子系统

`service/infra/workflow/` 提供更底层的工作流节点能力。

它和 `CommonFrameworkV1` 的节点体系不是同一个抽象层级：

- `CommonFrameworkV1/nodes` 更贴近具体合成任务生命周期
- `infra/workflow` 更像通用工作流基础设施

---

## 基础设施层在一次任务中的位置

一次任务里，这一层并不会主动驱动流程，而是被上层调用：

1. `ProxyService` 设置 LLM 上下文
2. `CommonFrameworkV1` 选择模型服务
3. 引擎申请 sandbox session
4. 后续所有工具调用与命令执行依赖 sandbox
5. 某些状态与配置能力可能通过 RDS 参与

所以它更像“供电系统”和“运行环境系统”，而不是业务编排者。

---

## 这一层的价值

如果没有基础设施层，上层逻辑会直接耦合：

- 各模型 SDK
- 各沙箱 SDK
- 各数据库实现

那整个系统会非常难维护。

通过 `service/infra/` 抽象后，上层可以用统一接口调用底层能力，平台也更容易扩展新的模型、新的沙箱来源和新的存储方式。
