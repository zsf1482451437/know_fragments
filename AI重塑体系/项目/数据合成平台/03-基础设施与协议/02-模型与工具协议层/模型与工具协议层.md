# 模型与工具协议层

## 模块定位

这一层对应：

- `service/mcp-tool/`
- `service/strategy/agent/trae/`
- `service/proxy/pipelines/common-framework-v1/core/model.ts`

它负责把“模型输出”变成“平台可执行动作”。

这是 Agent 模式里最关键的桥梁层。

---

## 这一层解决的问题

平台不是让模型直接输出最终代码文件，而是让模型输出工具调用。

因此必须解决两个问题：

1. 模型的工具调用格式如何被识别和解析。
2. 解析出来的工具调用如何映射到真实执行能力。

---

## 详细过程

### 1. 模型返回原始内容

`LLMModel.agentInvoke()` 首先会根据 `question.modelCall.source` 选择具体模型实现，得到原始输出：

- `thinking`
- `content`

这些内容此时还只是文本，并没有直接进入执行阶段。

### 2. 根据 toolFormat 解析工具调用

平台支持多种协议：

- `nfc`
- `xml`
- `json`

不同协议对应不同解析器：

- XML 走 `parseTraeToolcall`
- JSON 走 `parseJSONToolCalls`
- NFC 走 `parseXml65ToToolCalls`

也就是说，模型可以用不同格式表达“我要调用什么工具”，平台负责把它翻译成统一的 `toolCalls` 列表。

### 3. 做格式校验与重试

如果出现这些情况，平台不会直接继续：

- 没解析出工具
- 工具个数超限
- finish 和其他工具混用
- thinking 语言不符合要求
- JSON 不合法
- NFC 标签不合法

此时系统会把错误提示写回 `state.emphasis`，然后重新调用模型。

这相当于在模型和执行器之间增加了一层协议纠偏机制。

### 4. 得到结构化工具调用

最终模型输出会被统一成：

- `toolCalls: [{ name, params }]`
- `originalResult`
- `thinking`
- `content`

从这一步开始，平台不再关心原始文本长什么样，而只关心结构化的工具调用序列。

### 5. 工具名映射与参数适配

在执行工具前，平台会通过工具适配层处理：

- 工具名映射
- 参数字段名转换
- 结果格式包装

例如同一个逻辑工具，在不同协议下可能叫不同名字，参数结构也可能不同，适配层会把它们收敛成 sandbox 可识别的真实工具调用格式。

### 6. 在沙箱里执行工具

引擎最终调用：

- `marsSandboxSession.executeTraeTool({ toolName, args, version })`

这说明真实工具执行并不发生在当前 Node 进程，而是在远端沙箱环境里。

### 7. 把工具结果回写给模型

工具执行完成后，平台会把结果按协议格式化：

- NFC 用 `traeNfcToolResultFormat`
- 非 NFC 用 `traeToolResultFormat`

然后把这段结果作为新的 user message 追加到对话里，再次喂给模型。

这就形成了标准的 Agent 回路：

- 模型规划
- 平台执行
- 平台反馈
- 模型继续规划

---

## 为什么这层很关键

如果没有这层，模型只能输出一段文本，平台无法稳定执行。

有了这层之后：

- 模型的输出被协议化
- 协议被结构化解析
- 结构化调用被适配成真实工具动作
- 执行结果再回流给模型

这让整个系统从“文本生成器”升级成了“可执行 Agent 平台”。
