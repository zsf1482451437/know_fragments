## 项目运作过程

### 主链路

1. 外部系统向 `/api/proxy/execute` 发送请求，请求中带上 `method`、`question`、`uuid` 等任务信息。
2. 入口层根据 `method` 选择目标 pipeline，并把任务交给 `ProxyService`。
3. `ProxyService` 解析 `question`，建立任务上下文，把任务状态先写入 Redis。
4. 如果目标是 `CommonFrameworkV1`，任务会进入通用执行框架，开始初始化 `state`、注册 `knowledge` 和 `nodes`。
5. 框架根据模板配置申请沙箱、初始化 Git 仓库、准备工作目录，并构造模型对话上下文。
6. 模型进入 Agent 模式，持续输出工具调用；平台在沙箱中执行工具，再把结果回填给模型。
7. 当模型调用 `finish` 后，框架不会立刻结束，而是继续执行启动、构建、checker、测试生成、测试执行等后置验证流程。
8. 验证完成后，平台执行截图、上传、Git 提交、结果整理等收尾动作。
9. 最终结果被写回 Redis，外部系统通过 `/api/proxy/get_synthesis_result` 轮询获取执行状态和结果。

### 模块如何接力

- `平台入口与任务调度`：
  负责把外部请求接进来，并把任务正确分发给具体 pipeline。

- `通用执行框架`：
  负责把一条任务组织成真正可执行的生命周期，包括输入改造、主循环、验证和交付。

- `基础设施与协议`：
  负责提供模型调用、协议解析、沙箱环境和 Git 初始化等底层支撑，让上层流程能落到真实环境中执行。

- `质量保障与评估`：
  负责给整个执行过程加上监控、评分和测试闭环，确保结果不是“模型说完成了”，而是“平台验证过完成了”。

- `案例与样例`：
  负责把上面的抽象模块重新映射回真实模板任务，帮助理解一份 JSON 是如何在平台里完整流转的。

- `面试题与问答`：
  负责把项目背景、架构、流程和设计取舍整理成可直接复述的问答素材。

### 从输入到输出的抽象公式

```text
输入模板 JSON
-> 入口接入
-> 任务调度
-> 执行框架初始化
-> 沙箱 / Git / Prompt 准备
-> 模型 / 工具循环
-> 构建 / Checker / 测试
-> 收尾交付
-> Redis 持久化结果
```

### 时序图文本版

```text
外部系统
  -> ProxyController
      发送 /api/proxy/execute，请求中包含 method、question、uuid

ProxyController
  -> ProxyService
      注册 pipeline，并把任务交给 execute()

ProxyService
  -> Redis
      写入 doing 状态
  -> CommonFrameworkV1
      调用 process()，进入通用执行框架

CommonFrameworkV1 / CoreEngine
  -> Engine State
      初始化 state、trace_id、extra、messages
  -> Knowledge Registry
      注册 knowledge
  -> Node Registry
      注册 nodes
  -> Sandbox / Git
      申请沙箱、clone 仓库、执行初始化命令
  -> Prompt Builder
      泛化 prompt，拼接工具协议、技术栈、知识和目录结构

CoreEngine
  -> LLMModel
      发起模型调用
LLMModel
  -> 协议解析层
      解析 nfc / xml / json 工具调用
协议解析层
  -> Sandbox Tool Executor
      在沙箱中执行工具
Sandbox Tool Executor
  -> CoreEngine
      返回工具结果
CoreEngine
  -> LLMModel
      继续下一轮调用

循环终止条件
  -> 模型调用 finish

CoreEngine
  -> StartService Nodes
      启动服务或执行可运行动作
  -> Checker Nodes
      执行构建、静态分析、渲染检查
  -> Test Nodes
      生成测试、执行测试、回灌错误并统计结果
  -> BeforeFinish Nodes
      汇总监控、截图、上传产物、git commit / push

CommonFrameworkV1
  -> ProxyService
      返回最终结果

ProxyService
  -> Redis
      写入 success / failed 状态和最终 answer

外部系统
  -> /api/proxy/get_synthesis_result
      轮询任务结果
```

### 为什么这部分要放在根文档

知识图谱解决的是“这套文档怎么组织”。

项目运作过程解决的是“这套系统怎么运转”。

两者放在一起后，根文档同时具备两种能力：

- 可以作为文档导航入口
- 也可以作为平台总览入口
